## SEC-M5-01 — Spill blob upload allows cross-node forensic-evidence tampering: mint binds no eventId↔node ownership, blob write deterministically overwrites another node's spill object — **HIGH** (3/3)

> **STATUS: FIXED** (2026-06-09) — both remediation parts applied. **(1) Bind mint to node:** `MintSpillUpload` reads the mTLS-authenticated Thing (`enroll.ThingFromContext`) and namespaces the storage key as `<nodeId>/<day>/<eventId>-<direction>.bin`, bound into the HMAC-signed token (`Claims.NodeID`). `localfs.Put` now writes to the exact signed `claims.Key` (new `PutOptions.Key`) instead of re-deriving the shared key from `eventId+direction+today`, so two nodes minting for the same `eventId` get **different** keys — node A can never address or overwrite node B's object (also closes a latent midnight day-drift). Service-token callers (trusted, no Thing) keep the flat key and spill via direct in-process `Put`. **(2) Verify on read:** CP `resolveSpillBody` recomputes the SHA-256 of fetched bytes and refuses to serve a body whose hash ≠ the `ref.SHA256` recorded on the `traffic_event` (tamper detection; legacy empty-sha refs are skipped). Regression: `TestStore_Put_HonorsExplicitKey` (localfs), `TestMint_NamespacesKeyByNode` (cross-node key isolation + signed `NodeID`/`Key`), `TestResolveSpillBody_IntegrityCheck` (CP). Pre-GA: existing flat-key blobs stay readable (their `ref.Key` is stored); new agent blobs are namespaced. Smoke N/A (spill upload/read path, not the ai-gateway `traffic_event` cost/cache surface the smoke validates). Doc: `spillstore-architecture.md` §4. **This clears all non-gated HIGH findings.**

- **Persona:** A3 rogue/compromised node (any valid enrolled device-token / mTLS thing identity)
- **Invariant:** A traffic_event's captured request/response body (the spilled blob at <day>/<eventId>-<direction>.bin) must reflect what THAT node actually transmitted, and one node must not be able to overwrite another node's spill object.
- **Arch cross-ref:** F-0060

**Preconditions.** (1) SpillBackend == "localfs" so PUT /api/internal/spill/blob/:token is registered (routes.go:229-230). (2) Attacker holds any valid device/service identity accepted by enroll.DeviceOrServiceAuth on the /api/internal/things group (one rogue or compromised node suffices). (3) Attacker knows or guesses a victim's traffic_event UUID (eventId) and direction; a same-day overwrite lands on today's date directory automatically since the victim's legitimate spill is also written today. eventIds are UUIDs the attacker may learn from shared analytics, its own prior events, or enumeration of co-resident events.

**Attack steps.** 1. Attacker POSTs /api/internal/things/spill-uploads with {eventId: <victim traffic_event UUID>, direction: "response", sizeBytes: <len of forged body>, sha256: <sha256 of forged body>}. MintSpillUpload (spill_uploads.go:111-183) NEVER reads the authenticated thing identity from context — it validates eventId only as UUID-shape (validateMintRequest 201-224) and signs an HMAC token whose Key is derived purely from the caller-supplied eventId+direction via presigner.KeyFor (line 132 -> localfs.go:167-170). No check that eventId belongs to the caller. 2. Hub returns a localfs upload URL embedding the token. 3. Attacker PUTs the forged body to /api/internal/spill/blob/<token>. PutSpillBlob (spill_uploads.go:256-348) verifies the HMAC (valid — Hub just signed it), checks ContentLength==claims.SizeBytes and body-SHA256==claims.SHA256 (both satisfied because attacker chose a self-consistent body+hash), then Spill.Put with PutOptions{EventID,Direction}. localfs.Put (localfs.go:184-263) re-derives the destination key from eventId+direction with today's date (keyFor 158-161) and atomically os.Rename's over the existing file (line 251) — overwriting the victim node's already-spilled blob. 4. Per-token Redis dedup (DedupKey=sha256(token), secrets.go:174-176) does NOT block this: attacker's token hashes differently from victim's, so both pass SetNX writing the same deterministic key — last write wins. 5. When an admin opens the victim's traffic detail, CP resolveSpillBody (control-plane traffic.go:306-331) fetches bytes at ref.Key and serves them verbatim with NO re-verification against ref.SHA256. The forged body is presented as the genuine captured payload. Yield: tamper with / fabricate the recorded request or response body of another node's audited traffic event — corrupting compliance evidence and forensic audit trail, enabling repudiation or false attribution.

- **Affected:** packages/nexus-hub/internal/traffic/ingest/spill/spill_uploads.go:111-183 (MintSpillUpload — no eventId↔caller binding); packages/shared/storage/spillstore/localfs/localfs.go:158-170,192,251 (deterministic key + atomic overwrite); packages/shared/storage/spillupload/secrets.go:174-176 (dedup keyed per-token not per-object-key); packages/control-plane/internal/traffic/handler/traffic/traffic.go:306-331 (read path serves blob with no SHA re-verification)
- **Re-check probe.** Static: `rg -n 'func .*MintSpillUpload' -A40 packages/nexus-hub/internal/traffic/ingest/spill/spill_uploads.go` and assert the handler reads a caller identity (c.Get thingId / enroll context) and binds it to req.EventID's owning node before signing — currently it does not. Dynamic PoC: with nodes A and B, have B produce a spilled traffic_event E; from node A's device identity mint a spill token for {eventId:E, direction, sha256:<A-chosen body>} and PUT it; then read E's traffic detail via CP and confirm A's body is returned. Regression: Hub test asserting MintSpillUpload returns 403 when the authenticated thing does not own eventId; CP test asserting resolveSpillBody rejects a blob whose recomputed sha256 != recorded ref.SHA256.
- **Remediation.** Apply both: (1) Bind mint to caller — in MintSpillUpload read the authenticated thing identity (already established by enroll.DeviceOrServiceAuth on /api/internal/things) and either namespace the blob key by node identity (<nodeId>/<day>/<eventId>-<direction>.bin) or reject an eventId not owned/originated by the caller, so node A can never address node B's object. (2) Verify integrity on read — resolveSpillBody (and other spill readers) must recompute sha256 of fetched bytes and compare against ref.SHA256 persisted on the traffic_event, returning a tamper flag on mismatch instead of serving blindly. Residual to arch fix F-0060: F-0060 bound device-token to a Thing for SHADOW writes; the spill mint path has no analogous eventId↔node binding, so the cross-Thing-write class is unfixed here.

## SEC-M5-02 — Rogue node can flip the FLEET-WIDE kill-switch in either direction via a self-issued break-glass report (node-supplied desired-config provenance on a safety-critical key) — **HIGH** (3/3) 
> **STATUS: FIXED** (2026-06-09, FIX-2) — node-initiated break-glass can no longer write the fleet-wide `thing_config_template`: killswitch (and every key) now adopts as a PER-THING override (self-protection only); fleet-wide killswitch is operator-only via `POST /api/admin/compliance/killswitch`. Removed `breakGlassFleetScopedKeys` + the dead `adoptBreakGlassFleet` fleet-write path; updated kill-switch-architecture.md §131. Regression: `TestHandleBreakGlassReport_Killswitch_WritesPerThingOverride`. **Note: this narrows the documented "node break-glass is fleet-wide" semantics — see commit + doc.**


- **Persona:** A3 rogue/compromised node (any valid device-token node of a data-plane type, e.g. compliance-proxy or ai-gateway)
- **Invariant:** A fleet-wide safety control (kill-switch Category-A inline config) may only be changed by an authenticated Hub/admin actor; a single node must never be able to engage, disengage, or otherwise rewrite the kill-switch state that applies to every other node of its type.

**Preconditions.** Attacker holds (or has compromised) one valid device token for any data-plane Thing type T (issued at enrollment; A3 is explicitly in scope). Hub WS /ws or HTTP /api/internal/things/shadow/break-glass reachable (it is — the device is connected). No break-glass eligibility/trust gate exists on the node side.

**Attack steps.** 1. The node opens its normal authenticated WS session (server.go authenticate(): device token validated, thingID/thingType bound to the authenticated identity). 2. It emits one dedicated break-glass frame: {"type":"shadow_report_break_glass","reported":{"killswitch":{"engaged":false}},"reportedVer":<big>,"actorTokenId":"deadbeef","keyVersions":{"killswitch":<big>}} where <big> is any int64 strictly greater than the current template version (an attacker can just use a very large value). server.go:286-298 unmarshals this and calls HandleShadowReport with Reason="break_glass". 3. shadow.go:92-96 routes to handleBreakGlassReport. ValidateBreakGlassReport (break_glass.go:79-100) passes: killswitch IS in breakGlassWritableKeys and {"engaged":false} decodes cleanly into interception.Killswitch — validation checks ONLY the allowlist + schema, never the version magnitude, the direction of the change, or whether THIS node is authorized to set a fleet default. 4. killswitch is in breakGlassFleetScopedKeys (break_glass.go:56-58), so handleBreakGlassReport calls adoptBreakGlassFleet(ctx, thing.Type, "killswitch", state, reportedVer, ...) (break_glass.go:173-177) — note thing.Type, the FLEET, not thing.ID. 5. adoptBreakGlassFleet (break_glass.go:191-238) calls UpsertConfigTemplateAt with the node-supplied state and node-supplied version; the SQL (config_template.go:151-168) writes state+version verbatim, gated only by `EXCLUDED.version > thing_config_template.version`, which the attacker's large version always satisfies. 6. thing_config_template[T, killswitch] is now attacker-controlled. BulkConfigPull (internal_things.go:240) and SingleConfigPull (internal_things.go:330) serve thing_config_template as the authoritative desired source, so on the next config pull / reconnect / drift reconciliation EVERY node of type T adopts the attacker's killswitch value. Direction A — engaged=true: forces the entire fleet of that type into kill-switch passthrough (no TLS-bump / compliance interception disabled) = fleet-wide compliance bypass and safety-control DoS. Direction B — engaged=false: cancels an operator's deliberately-engaged emergency cutoff for the whole fleet = fail-open of the safety control the operator just engaged. Either way a single node dictates the fleet safety posture without any admin involvement; the change can be re-asserted on every reconnect.

- **Affected:** packages/nexus-hub/internal/fleet/manager/break_glass.go:173-177 (fleet-scope routing on thing.Type), break_glass.go:191-238 (adoptBreakGlassFleet writes node-supplied state+version fleet-wide), break_glass.go:79-100 (ValidateBreakGlassReport gates only allowlist+schema, not provenance/direction/magnitude), packages/nexus-hub/internal/fleet/shadow/config_template.go:151-168 (UpsertConfigTemplateAt accepts node-chosen version verbatim), packages/nexus-hub/internal/ws/server.go:280-301 (WS break-glass dispatch trusts payload.Reported/ReportedVer for a fleet key), packages/nexus-hub/internal/fleet/handler/hubapi/internal_things.go:179-209 (HTTP /shadow/break-glass twin)
- **Re-check probe.** grep -n 'breakGlassFleetScopedKeys' packages/nexus-hub/internal/fleet/manager/break_glass.go — finding stands as long as configkey.Killswitch is in that map AND adoptBreakGlassFleet is reached from a node-authenticated path. Code probe: in ValidateBreakGlassReport / adoptBreakGlassFleet there is NO check that (a) the change is operator-authenticated, (b) restricts the killswitch direction, or (c) limits which node may set a fleet default. Behavioural test: stand up Hub + two compliance-proxy device tokens; from node-A send a shadow_report_break_glass with keyVersions.killswitch=1e18 and reported.killswitch.engaged toggled; assert thing_config_template[compliance-proxy,killswitch] changed and a SingleConfigPull as node-B now returns the attacker value. Regression guard pattern: assert handleBreakGlassReport rejects a killswitch break-glass that is not corroborated by an admin/Hub-authenticated actor (grep for any new actor-authority check around break_glass.go:173).
- **Remediation.** Treat the fleet-wide kill-switch as Hub/admin-authority-only on the desired side: a node break-glass must NEVER write thing_config_template[T, killswitch] for the whole fleet on the node's say-so. Options, in preference order: (1) Remove configkey.Killswitch from breakGlassFleetScopedKeys and from breakGlassWritableKeys entirely — a node's local emergency passthrough is already enforced node-side; it does not need to mutate the fleet template. The fleet kill-switch stays admin-only via UpdateConfig. (2) If a node-originated fleet engage must exist for an outage, constrain it to the safe direction only (accept engaged=true, reject engaged=false so a node can never cancel an operator cutoff), require the node's break-glass to be corroborated by an out-of-band admin-authenticated confirmation before the fleet template is rewritten, and cap/normalise the adopted version to current+1 (never trust the node's reportedVer magnitude). (3) Scope ALL node break-glass adoptions to the reporting Thing's own desired override (per-Thing, like exemptions) and let only an admin promote a fleet default — i.e. delete the fleet-scoped node path. Record the kill-switch-architecture.md decision in the same change.

## SEC-M5-03 — Single compromised node can flip the fleet-wide killswitch (engage OR disengage) via break-glass shadow report with no operator approval — **LOW** (2/3) 
> **STATUS: FIXED** (2026-06-09, FIX-2) — node-initiated break-glass can no longer write the fleet-wide `thing_config_template`: killswitch (and every key) now adopts as a PER-THING override (self-protection only); fleet-wide killswitch is operator-only via `POST /api/admin/compliance/killswitch`. Removed `breakGlassFleetScopedKeys` + the dead `adoptBreakGlassFleet` fleet-write path; updated kill-switch-architecture.md §131. Regression: `TestHandleBreakGlassReport_Killswitch_WritesPerThingOverride`. **Note: this narrows the documented "node break-glass is fleet-wide" semantics — see commit + doc.**


- **Persona:** A3 rogue/compromised node
- **Invariant:** A fleet-wide safety control (killswitch) must only be changed by an operator/authoritative-admin path; no single data-plane node may unilaterally set the killswitch state for every other node of its type — and never DISENGAGE a killswitch an operator engaged.

**Preconditions.** Attacker holds a valid device token (or the shared internal service token) for one node of a given type (e.g. compliance-proxy / ai-gateway / agent) — i.e. one enrolled node is compromised, OR the attacker can open one authenticated /ws connection. No operator credentials, no second factor, no fleet-admin role required.

**Attack steps.** 1. The break-glass key 'killswitch' is FLEET-SCOPED: breakGlassFleetScopedKeys = {Killswitch:true} (break_glass.go:56-58). handleBreakGlassReport routes it to adoptBreakGlassFleet, which writes thing_config_template[thingType,'killswitch'] — the authoritative fleet-wide template every node of that type reads (break_glass.go:173-177, 191-238).
2. The Killswitch schema is just {Engaged bool} (interception/killswitch.go:12-14). ValidateBreakGlassReport only checks (a) key in {killswitch,exemptions} allowlist and (b) the state decodes cleanly into the struct (break_glass.go:79-131). It does NOT check the engaged value, does NOT require an operator, does NOT ratchet (no 'engaged may only go true' rule).
3. The only write guard is version monotonicity: UpsertConfigTemplateAt uses 'WHERE EXCLUDED.version > thing_config_template.version' (config_template.go:166). The version is reportedVer/KeyVersions[key] taken straight from the attacker's payload (break_glass.go:166, ws/message.go:69,72), so the attacker passes the guard by sending any large integer.
4. Reach: WS frame type 'shadow_report_break_glass' (server.go:280-301) — thingID is the authenticated identity but is irrelevant here because the write targets the TYPE template, not this node's row; the WS path does not even require ActorTokenID. HTTP fallback POST /api/internal/things/shadow/break-glass (internal_things.go:179-213) is equivalent.
5. Yield: attacker sends killswitch{engaged:true},keyVersions{killswitch:<big>} → every node of that type converges to killswitch-engaged (fleet-wide DoS of the data plane). OR killswitch{engaged:false},keyVersions{killswitch:<big>} → DISENGAGES a killswitch an operator deliberately engaged during an incident, silently disabling the fleet-wide safety cut-off. requireThingMatch (F-0060) does NOT help: it binds per-thing operations to the caller id, but the killswitch template is fleet-scoped by design, so it escapes that control entirely.

- **Affected:** packages/nexus-hub/internal/fleet/manager/break_glass.go:56-58 (fleet-scoped set), :173-238 (adoptBreakGlassFleet), :79-131 (validator with no engaged/ratchet check); packages/nexus-hub/internal/fleet/shadow/config_template.go:166 (version-only guard); packages/shared/schemas/configtypes/interception/killswitch.go:12-14; reachable from packages/nexus-hub/internal/ws/server.go:280-301 and packages/nexus-hub/internal/fleet/handler/hubapi/internal_things.go:179-213
- **Re-check probe.** rg -n 'breakGlassFleetScopedKeys' packages/nexus-hub/internal/fleet/manager/break_glass.go  # confirm Killswitch still fleet-scoped; then assert adoptBreakGlassFleet has NO operator-authority / engaged-ratchet guard: rg -n 'func .*adoptBreakGlassFleet' -A40 packages/nexus-hub/internal/fleet/manager/break_glass.go | rg -i 'engaged|operator|ratchet|admin' || echo 'NO ENGAGED/OPERATOR GUARD — finding stands'. Live PoC: open one authed /ws as a node, send {"type":"shadow_report_break_glass","reported":{"killswitch":{"engaged":false}},"reportedVer":1,"keyVersions":{"killswitch":9999999}} and observe thing_config_template[type,'killswitch'].state become {engaged:false}.
- **Remediation.** Treat fleet-scoped break-glass keys as operator-authority-only: (a) reject killswitch adoption that arrives without an operator-authenticated actor (require a verified break-glass token whose authority is checked server-side, not a self-asserted ActorTokenID), OR (b) restrict the data-plane break-glass write to the ENGAGE direction only (ratchet: a break-glass report may set engaged=true but may never set engaged=false; disengage must come from the admin/CP path), and (c) rate-limit / alert on any fleet-template break-glass write. The disengage-via-node path in particular has no legitimate use and should be removed.

## SEC-M5-04 — A single compromised node can engage the fleet-wide kill-switch via break-glass, disabling TLS-bump / compliance inspection for every node of its type — no rate-limit, approval, or auto-revert — **LOW** (2/3) 
> **STATUS: FIXED** (2026-06-09, FIX-2) — node-initiated break-glass can no longer write the fleet-wide `thing_config_template`: killswitch (and every key) now adopts as a PER-THING override (self-protection only); fleet-wide killswitch is operator-only via `POST /api/admin/compliance/killswitch`. Removed `breakGlassFleetScopedKeys` + the dead `adoptBreakGlassFleet` fleet-write path; updated kill-switch-architecture.md §131. Regression: `TestHandleBreakGlassReport_Killswitch_WritesPerThingOverride`. **Note: this narrows the documented "node break-glass is fleet-wide" semantics — see commit + doc.**


- **Persona:** A3 rogue/compromised node
- **Invariant:** A compliance-enforcement control (TLS-bump MITM inspection) must not be disablable fleet-wide by the unilateral, unattested action of a single data-plane node.
- **Arch cross-ref:** F-0138

**Preconditions.** Attacker controls one node of type compliance-proxy (or agent) with its own valid device token and a WS connection (A3). The break-glass allowlist permits configkey.Killswitch and the killswitch key is fleet-scoped by design.

**Attack steps.** 1. Node sends a shadow_report_break_glass frame (or POST /shadow/break-glass) for its OWN thingID with KeyVersions={killswitch: <very-large>} and Reported={killswitch:{engaged:true}}. requireThingMatch passes (it is its own id) and ValidateBreakGlassReport passes (killswitch is allowlisted, schema valid). 2. handleBreakGlassReport routes killswitch through adoptBreakGlassFleet (break_glass.go:173-178), because breakGlassFleetScopedKeys={killswitch}. 3. adoptBreakGlassFleet calls UpsertConfigTemplateAt (config_template.go:151-176) whose only guard is WHERE EXCLUDED.version > current — the attacker simply supplies a higher version, so the write succeeds and rewrites thing_config_template[type,killswitch] = {engaged:true} for the WHOLE type. 4. The fleet template is pushed to every node of that type as desired config; each compliance-proxy computes BumpEnabled = !KillSwitch.IsEngaged() (compliance-proxy/internal/runtime/handler/handler.go:39), so TLS-bump / MITM inspection is turned OFF fleet-wide. 5. Unlike the per-Thing exemption path (8h TTL via breakGlassPerThingTTL), the fleet killswitch deliberately has NO auto-revert (break_glass.go:29-32), and there is no rate-limit, no admin approval, and no anomaly throttle — so one compromised node silently suppresses compliance inspection across the fleet until an operator notices.

- **Affected:** packages/nexus-hub/internal/fleet/manager/break_glass.go:56-58 (breakGlassFleetScopedKeys), :173-238 (adoptBreakGlassFleet); packages/nexus-hub/internal/fleet/shadow/config_template.go:158-168 (version-only guard); packages/compliance-proxy/internal/runtime/handler/handler.go:39 (BumpEnabled = !IsEngaged())
- **Re-check probe.** rg -n 'breakGlassFleetScopedKeys' packages/nexus-hub/internal/fleet/manager/break_glass.go and confirm killswitch routes to the fleet template with no per-node throttle/approval; rg -n 'BumpEnabled' packages/compliance-proxy/internal/runtime/handler/handler.go to confirm engaged⇒inspection-off. Test idea: drive handleBreakGlassReport with killswitch engaged=true from a single thingID and assert whether any rate-limit/alert/approval gate exists before the fleet template write (currently none).
- **Remediation.** This fleet-scope is accepted by F-0138 for availability, but the security blast radius (1 node → fleet inspection-off, no attribution-bound limit) is the residual. Add a defense-in-depth control: (a) emit a high-severity SIEM alert whenever a break-glass killswitch ENGAGE hits the fleet template (so a single-node fleet-wide disable is never silent), and/or (b) require N-node corroboration or an admin-confirm window before the fleet template adopts a node-originated killswitch engage, and/or (c) cap break-glass killswitch engages per node per window to throttle a compromised node.

## SEC-M5-05 — Rejected/over-privileged break-glass attempts produce no audit/SIEM event — **LOW** (2/3)
> **STATUS: FIXED** (2026-06-09) — on a rejected break-glass (`handleBreakGlassReport` error: non-allowlisted key or schema-invalid state) `HandleShadowReport` now writes a durable `break_glass_denied` config_change_event (authenticated thingID, attempted keys, validation error) via `emitBreakGlassDenied`, so a node probing/fuzzing the break-glass authority surface is visible to the audit/SIEM stream instead of leaving only a rotatable Hub log line. Best-effort (never changes the report outcome). Regression: `TestManager_HandleShadowReport_BreakGlass_DeniedWritesAudit`.

- **Persona:** A3 rogue/compromised node
- **Invariant:** An attempt by a node to perform a disallowed privileged config write (break-glass with a non-allowlisted key or malformed state) must generate a durable security event, not just a transient server log line.
- **Arch cross-ref:** F-0139

**Preconditions.** Attacker controls one enrolled node with a valid device token. Hub /ws reachable.

**Attack steps.** 1) handleBreakGlassReport re-validates via ValidateBreakGlassReport (break_glass.go:152) and returns an error for any key outside {killswitch, exemptions} or any schema-invalid state. 2) But HandleShadowReport swallows that error: shadow.go:92-95 logs it at Error and returns nil — there is no config_change_event, no AdminAuditLog, no SIEM-classifiable row written on rejection (the only audit writes happen on the SUCCESS paths: break_glass.go:220 fleet, break_glass.go:333 per-thing). 3) ClassifyAdminEvent (classify.go:52-69) only maps successful thing_override_set / canonical pairs; there is no event type for a denied node-side privilege escalation, so the SIEM whitelist and admin audit picker never see the attempt. 4) On the WS path there is additionally no edge pre-validation and no ActorTokenID-required check (unlike the HTTP handler, internal_things.go:196-207), so a rogue node can probe the writable-key allowlist and schema gate repeatedly (key enumeration / fuzzing of the privileged reconciliation path) leaving only easily-rotated Hub Error logs — an attacker mapping the break-glass authority surface is invisible to defenders watching the audit/SIEM stream.

- **Affected:** packages/nexus-hub/internal/fleet/manager/shadow.go:92-96; packages/nexus-hub/internal/fleet/manager/break_glass.go:152-154; packages/nexus-hub/internal/observability/siem/classify.go:52-69; packages/nexus-hub/internal/ws/server.go:280-301
- **Re-check probe.** ripgrep: rg -n 'break-glass reconciliation' packages/nexus-hub/internal/fleet/manager/shadow.go — if the only sink for bgErr is m.logger.Error with no audit/event insert nearby, the gap persists. Test idea: send a break-glass WS frame with key='credentialKeys' (not in breakGlassWritableKeys); assert (a) HandleShadowReport returns nil (current behavior) and (b) a security/audit row with action like 'break_glass_denied' exists — it currently does not.
- **Remediation.** On break-glass rejection, write a durable denied-attempt security event (config_change_event or AdminAuditLog with action 'break_glass_denied', the authenticated thingID, the offending key, and the validation error) and give it a cataloged ClassifyAdminEvent identity so SIEM whitelists surface it. Also mirror the HTTP edge checks on the WS path (require a Hub-derived actor; reject before dispatch) so denied attempts are uniform across transports.
