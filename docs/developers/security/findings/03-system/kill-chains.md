Fixing C3-02/M5-02 — demote killswitch to per-thing for node break-glass, require a Hub-verified operator proof for any fleet-scoped key — removes the fleet-blackout stage. ### A1-KC2 — Network-delivered attestation forgery turns the :3128 MITM proxy into a permanent, fleet-wide, unrevocable compliance-inspection bypass once one agent's attestation key is read  
**Persona** A1 · **Severity** HIGH · **Feasibility** requires-strong-preconditions

- **Steps:** 1. Precondition acquisition (A6, one-time): the agent's Ed25519 attestation private key is written as a plaintext PKCS8 PEM at 0600 (`attestation-key.pem`, enroll.go:392/399) and loaded with a bare os.ReadFile (signer.go:170) — NOT the platform keystore the Hub CA comment falsely claims (ca.go:123-125). Any filesystem/backup/snapshot read of one monitored host yields the key (SEC-M4-02).
2. A1 reconstructs the canonical signature pre-image (tlsbump/attestation.go:61: version/ts/nonce/sha256("")/agent_id). Because v1 binds the body as sha256(""), the signature is NOT content-bound — one key forges attestation for arbitrary request bodies (SEC-M4-02 step 3).
3. From any network position, A1 issues CONNECTs to :3128 (all-interfaces listener, verified) carrying a self-signed `X-Nexus-Attestation` header. The compliance-proxy verifier AttestationVerifier.Verify (attestation.go:132-214) checks only the ±5min ts window + Ed25519 sig + nonce-LRU — it performs ZERO cert-expiry and ZERO revocation check (SEC-M4-01 / SEC-C2-02). The stolen key matches the stamped pubkey, so Verify returns Valid.
4. Valid routes the flow to attestationPassthrough (forward_handler.go:129-137 → helpers:372): forwarded upstream with NO compliance pipeline, NO payload capture, NO CP-side audit emission (SEC-M4-01).
5. Durability: the 90-day cert NotAfter is never read at verify time (GetAttestationPubKey returns publicKey only, internal_things.go:86-89), AttestationKeyCache.Invalidate is dead for attestation in production, and no decommission/rotation path clears sysinfo.attestation. So the stolen key is a PERMANENT bypass oracle — rotating the victim's device token or re-imaging the box does not revoke it (SEC-M4-01, SEC-C2-02).
- **Findings used:** SEC-M4-02 (plaintext on-disk attestation key + non-body-bound v1 signature), SEC-M4-01 (verify path ignores expiry + no revocation lever), SEC-C2-02 (attestation trust decoupled from device-token lifecycle — revoke gives false assurance).
- **Net effect:** A single one-time key read converts into an indefinite, network-reachable, unrevocable ability to send uninspected and CP-unaudited AI traffic past the compliance MITM proxy — defeating the product's core control (compliance inspection) for arbitrary content, fleet-wide, with no admin lever to stop it short of disabling attestation cluster-wide.
- **Weakest link (cheapest break):** SEC-M4-01 — the verifier's missing expiry+revocation enforcement is the single fix that bounds the chain: if Verify rejected on now>certExpiresAt and honored a revoked flag cleared on decommission, the stolen key would self-expire in ≤90 days and be killable on demand, collapsing 'permanent oracle' to 'bounded, revocable window'. (The chain also depends on the A6 key-read precondition, which is why it is requires-strong-preconditions for a pure-network A1.)

### W-A5-3 — Cross-VK semantic-cache poisoning → fabricated LLM responses served across the VK isolation boundary  
**Persona** A5 · **Severity** MEDIUM · **Feasibility** demonstrated

- **Steps:** 1) (SEC-C4-01) Semantic L2 cache is enabled fleet-wide. A5 calls POST /api/admin/semantic-cache/prewarm with entries:[{prompt:'<text semantically near a victim's expected prompt>',response:'<attacker-chosen answer — malicious instruction or false data>',vkScope:'<VICTIM_VK_ID>',model:'gpt-4o',ttlSeconds:604800}]. VERIFIED: the CP handler forwards the caller-supplied vkScope VERBATIM (semantic_prewarm.go:57-59 defines vkScope as a free JSON field; the finding confirms no per-caller authorization of it) — it never validates that vkScope is a VK the caller owns or that it even exists. 2) AI-GW SemanticPrewarmHandler writes a Valkey HASH with VKScope = the attacker's value and ResponseBody = the attacker's response (semantic_prewarm_endpoint.go:247,252). 3) The victim VK sends a genuine request whose prompt is within the cosine threshold of the planted prompt. tryL2Lookup computes vkScope = the victim VK id and the FT.SEARCH read filters ONLY on @vk_scope:{victim} (lookup.go:82). The planted entry carries exactly that tag → returned and served as a cache HIT. If the fleet runs varyBy='none', the empty scope matches every lookup and no victim id is even needed. 4) (Combine with W-A5-1) If A5 first self-escalates via the keystone chain, it also enumerates live VK ids (credential/VK read) to target high-value lanes precisely, and can disable a victim's real VK (SEC-M6-05) to force traffic onto a fresh key whose first responses are all poisoned.
- **Findings used:** SEC-C4-01 (prewarm accepts arbitrary vkScope → cross-VK cache poisoning); optionally chained behind SEC-M6-03/02 (escalation to enumerate/target VKs) and SEC-M6-05 (VK revoke to force re-resolution).
- **Net effect:** A low-privilege caching operator injects attacker-fabricated LLM responses into any victim VK's answer lane — a response-integrity / prompt-injection breach crossing the VK isolation boundary, served as a legitimate cache hit with no error or log distinguishing it from a real response.
- **Weakest link (cheapest break):** SEC-C4-01 — the unvalidated caller-supplied vkScope in PrewarmCache. Fix: drop vkScope from the prewarm API entirely (always write a dedicated corpus scope), OR validate that every entry.vkScope names a VK the caller is authorized to write for; reject otherwise. Either makes the planted entry land in a scope the victim VK never reads, breaking step 3 regardless of escalation.

### A1-KC4 — Live virtual key harvested from a fronting-proxy access log via the `?key=` URL carrier, then replayed for full cross-VK impersonation  
**Persona** A1 · **Severity** MEDIUM · **Feasibility** plausible

- **Steps:** 1. A victim caller (or an attacker steering an OpenAI-route request) reaches the data plane carrying the VK as `?key=nvk_<64hex>`. extractVKToken reads the `?key=` query carrier and auth succeeds on Gemini routes (vkauth.go:237). On an OpenAI route this carrier is unlocked by setting header `x-nexus-aigw-body-format: gemini`, which flips BodyFormat→gemini (ingress.go:201-214 → proxy.go:291) — SEC-M3-02 preconditions, verified.
2. The live VK now sits in the request-line URL. The gateway's own logs use r.URL.Path only (no leak), but every fronting hop — nginx/ALB/CDN $request_uri, corporate egress proxy, browser history, Referer — captures the full query in cleartext (SEC-M3-02 steps 3-4). This is the standard prod topology since :3050 binds loopback behind nginx (verified: ai-gateway.config.yaml host 127.0.0.1, nginx fronts /v1).
3. A1 with read access to any access-log store recovers the live `nvk_<64hex>` in cleartext — no decryption. At rest the gateway stores only HMAC(key)+prefix, so the URL/log copy is the ONLY plaintext copy of a still-valid credential (SEC-M3-02 step 4).
4. A1 replays `nvk_<64hex>` against nginx→:3050: full cross-VK impersonation — the victim's model allowlist, quota, billing attribution, and source-app identity — until rotation (SEC-M3-02 step 5).
- **Findings used:** SEC-M3-02 (VK accepted as ?key= URL carrier; OpenAI-route override unlocks it). Adjacent amplifier: SEC-M3-01 (revoke-user-access pushes no virtual_keys cache invalidate, so even a noticed-and-revoked principal's keys keep authenticating for up to VKTTL≤30s, slightly widening A1's replay window after rotation is triggered).
- **Net effect:** Recovery and replay of a live victim virtual key by a passive log reader, yielding full cross-VK impersonation (the victim's spend, allowlist, and attribution) with no decryption and no in-band signal to the victim.
- **Weakest link (cheapest break):** SEC-M3-02 — making the `?key=` carrier opt-in (default off) and refusing the x-nexus-aigw-body-format override that lets an OpenAI route inherit the URL-key carrier removes the only path by which the credential enters a loggable request component. Header carriers (Authorization/x-goog-api-key) already cover real SDKs, so the carrier can be disabled by default with no functional loss. The chain depends on A1 having read access to a fronting log and a victim using the URL carrier — hence plausible, not demonstrated.

### W-A6-4 — Access-log read of a plaintext VK → live cross-VK impersonation (no decryption)  
**Persona** A6 · **Severity** MEDIUM · **Feasibility** plausible

- **Steps:** 1) The data plane accepts a VK as the ?key=<vk> URL query parameter on Gemini routes (vkauth.go:237), and an OpenAI-route caller can flip BodyFormat to gemini via the x-nexus-aigw-body-format header (ingress.go:201-214 → proxy.go:291) to unlock the same carrier. The gateway's OWN logs use only r.URL.Path, but every fronting/adjacent component (nginx/ALB/CDN $request_uri, corporate forward-proxy, browser history, Referer) logs the full URI WITH the query. Since :3050 binds loopback behind a fronting proxy in standard prod topology, at least one such logging hop exists. [SEC-M3-02]
2) A6 reads any such access-log store and recovers the full live nvk_<64hex> in cleartext. At rest the gateway stores only HMAC(key)+12-char prefix, so this URL/log copy is the ONLY plaintext copy of a still-valid credential — and the easiest to read, requiring no key recovery.
3) A6 replays nvk_<64hex> against :3050: full cross-VK impersonation — the victim's model allowlist, quota, billing attribution, source-app identity — until rotation.
4) Amplifier (optional): once authenticated as the victim VK, A6 can reach a provider/model OUTSIDE the victim's allowedModels via an inline FallbackChain recovery target (allowlist not enforced on FallbackChain entries), consuming a different provider's upstream credential under the victim's identity. [SEC-C1-01]
- **Findings used:** SEC-M3-02 (VK accepted as ?key= URL param → leaks to fronting-proxy/LB/CDN access logs, browser history, Referer; OpenAI-route override unlocks it), optional amplifier SEC-C1-01 (allowedModels bypass via inline FallbackChain). Verified glue: vkauth.go:237 extractVKToken reads ?key=; ingress.go:201-214 + proxy.go:291 format override.
- **Net effect:** A6 with read-only access to a routine prod logging tier (not even the application's own logs) obtains a live, replayable virtual key in cleartext and impersonates the victim VK end-to-end — spending the victim's quota, billing the victim, and (with C1-01) reaching models the VK was never allowed to use — with zero cryptographic work and no foothold on the application host itself.
- **Weakest link (cheapest break):** The ?key= URL carrier being default-on (and silently reachable from OpenAI routes via the body-format override), routing a long-lived bearer credential through the one request component intermediaries persist in cleartext. Making ?key= opt-in (default off; header carriers already cover real Gemini SDKs) and restricting the x-nexus-aigw-body-format override removes the only plaintext-in-logs copy of the VK, breaking the chain regardless of who reads the logs.

### VK-in-URL log harvest → cross-VK credential replay (host/log-read pivot seeded by A2)  
**Persona** A2 · **Severity** MEDIUM · **Feasibility** requires-strong-preconditions

- **Steps:** 1) A2 issues GET/POST /v1beta/models/gemini-2.5-pro:generateContent?key=nvk_<64hex> against :3050; OR forces an OpenAI-route request into Gemini ingress with x-nexus-aigw-body-format: gemini, which flips BodyFormat to gemini and unlocks the ?key= carrier (ingress.go:201-214 → proxy.go:291). extractVKToken reads r.URL.Query().Get('key') for FormatGemini (verified vkauth.go:233-239) — auth succeeds with the VK now in the request-line URL. 2) The gateway's own logs use r.URL.Path only, so it does not leak — but every standard fronting/adjacent hop that logs the request URI (nginx/ALB/CDN $request_uri, corporate egress proxy, browser history, Referer) captures the full live VK in cleartext. The window is widened by SEC-M3-01: a 'revoke user access' does NOT push a virtual_keys cache invalidate, so even after the victim's owner is revoked the leaked VK keeps authenticating for up to the 30s cache TTL on every gateway.
- **Findings used:** SEC-M3-02 (VK accepted as ?key= URL param → plaintext leak to fronting-proxy/LB/CDN access logs + Referer + history; OpenAI-route unlock via body-format override), SEC-M3-01 (revoke-user-access pushes no virtual_keys invalidate → revoked/leaked key keeps authenticating for the cache TTL — widens the replay window).
- **Net effect:** A long-lived bearer VK is forced into cleartext in standard log/intermediary surfaces; whoever reads those logs replays it for full cross-VK impersonation (quota theft, billing fraud, model-allowlist abuse, attribution spoofing) with a revocation grace window that does not promptly stop in-flight abuse.
- **Weakest link (cheapest break):** SEC-M3-02 — the ?key= URL carrier (vkauth.go:237) combined with the body-format override that unlocks it on non-Gemini routes. Requires a logging intermediary that retains query strings — the standard but not guaranteed prod topology.
