# C1 — Credential lifecycle

> Phase 2 chain audit. Confirmed findings only (≥2/3 adversarial Opus verifiers). Base = post-arch-audit + SEC-M2-01 fix. Each finding attacks an end-to-end trust-boundary crossing.


## SEC-C1-01 — Per-VK allowedModels allowlist bypassed by a routing rule's inline FallbackChain (recovery targets dispatched + their provider credential used without the allowlist check) — **HIGH** (3/3)

> **STATUS: FIXED** (2026-06-09) — the inline `FallbackChain` loop in `resolver.go` now routes its targets through `narrowingEngine.Filter` (the same `allowedModels` + narrowing gate the primary path and the `fallbackRules` path already use) before appending to `plan.RecoveryTargets`. A `FallbackChain` entry whose `(providerID, modelID)` is outside the VK's `allowedModels` is dropped, so a primary failure can no longer advance to — or consume the upstream credential of — a model the VK is not permitted to use. Regression: `TestResolver_InlineFallbackChain_RespectsVKAllowedModels` (out-of-allowlist entry excluded) + `TestResolver_InlineFallbackChain_KeepsAllowlistedEntry` (positive control, no over-blocking). Scoped to resolver unit tests: the change adds a routing-layer allowlist filter and touches none of the traffic_event / cost / cache / normalize / codec / adapter surface the ai-gateway smoke uniquely validates. Doc: `routing-architecture.md`.

- **Persona:** A2 valid-VK
- **Invariant:** Every upstream target a VK's request is dispatched to — primary AND every failover/recovery target — must satisfy that VK's allowedModels allowlist; the credential used must belong to a provider/model the VK is permitted to use.
- **Arch cross-ref:** F-0222-adjacent (LEDGER.md:1511). The arch finding noted plan.RecoveryTargets bypassed the EMBEDDINGS capability pre-filter; that gap was fixed (resolver.go:191-193 now run applyCapabilityFilter on both plan.Targets and plan.RecoveryTargets). The residual: the same RecoveryTargets path still bypasses the SECURITY allowlist (narrowingEngine.Filter / AllowedModels) for inline FallbackChain entries — capability correctness was patched, the per-VK allowlist enforcement was not.

**Preconditions.** (1) A routing rule with PipelineStage==1 matches the VK's request and carries a non-empty inline FallbackChain (a normal, supported routing feature — distinct from separate 'fallback' StrategyType rules). (2) The matching VK has a non-empty allowedModels (a restricted VK — the common multi-VK case). (3) At least one FallbackChain entry references a (providerID, modelID) NOT in that VK's allowedModels — e.g. a different provider whose upstream credential the VK is not authorized to consume. (4) The primary target fails (any retriable upstream error / circuit-open) so the executor advances to the recovery target. No attacker-supplied body field is needed; the steering is structural.

**Attack steps.** Stage 1 (auth): A2 authenticates with a restricted VK whose allowedModels=[X] (model X on provider P_X). stickyKey is correctly set to vkMeta.ID (proxy.go:432) and the global Credential/Provider catalog is shared across all VKs (schema.prisma:388 Credential has no VK scoping). Stage 2 (route): request for model X matches a stage-1 rule that has an inline FallbackChain entry {providerID:P_Y, modelId:Y}. In resolver.go Resolve, the PRIMARY targets go through r.narrowingEngine.Filter (resolver.go:134) which enforces ModelMatchesAllowedRefs against allowedModels, and the separate fallbackRules path is also filtered (resolver.go:167). BUT the inline FallbackChain loop (resolver.go:147-153) calls r.lookupTarget(P_Y,Y) and appends the result straight to plan.RecoveryTargets with NO narrowingEngine.Filter call — the AllowedModels check is skipped for these entries. Stage 3 (flatten): ResolveTargets concatenates plan.Targets + plan.RecoveryTargets into allTargets (resolver.go:385-387) and returns them all as dispatchable. Stage 4 (dispatch): executor.go:254 `for tIdx, target := range targets` iterates every target; on primary (X) failure it advances to the recovery target Y. Stage 5 (credential): provtarget resolveCredential (target/resolver.go:182-212) resolves the credential purely from target.ProviderID==P_Y (enforcing only mr.ProviderID==providerID at resolver.go:145, which Y satisfies for P_Y), decrypts P_Y's upstream key (credmanager.GetForProvider / GetDecrypted), and serves the request on model Y / provider P_Y. The executor and provider dispatch never re-check allowedModels (grep of internal/execution + internal/providers for AllowedModels returns nothing). Net effect: a VK restricted to model X is served by, billed for, and uses the upstream API key of provider P_Y / model Y — silently crossing the per-VK model-scope boundary the allowlist exists to enforce.

- **Affected:** packages/ai-gateway/internal/routing/resolver.go:142-154 (inline FallbackChain loop appends to plan.RecoveryTargets without narrowingEngine.Filter; contrast primary at :134 and fallbackRules at :167 which both Filter). Bypass dispatched via packages/ai-gateway/internal/routing/resolver.go:385-387 (flatten) and packages/ai-gateway/internal/execution/executor/executor.go:254 (target iteration). Credential resolved from the unchecked target at packages/ai-gateway/internal/providers/target/resolver.go:182-212. Allowlist enforcement that is missing here lives in packages/ai-gateway/internal/routing/matcher/narrowing.go:123-137 (Filter applies ModelMatchesAllowedRefs to every target it receives).
- **Re-check probe.** In packages/ai-gateway/internal/routing/resolver.go, locate the inline FallbackChain block (the `if len(primaryRule.FallbackChain) > 0 {` loop that calls r.lookupTarget and appends to plan.RecoveryTargets). Assert the candidate finding is OPEN if, between that loop's lookupTarget result and the append to plan.RecoveryTargets, there is NO call to r.narrowingEngine.Filter (or any ModelMatchesAllowedRefs / AllowedModels check). Grep guard: `grep -n 'narrowingEngine.Filter\|ModelMatchesAllowedRefs' packages/ai-gateway/internal/routing/resolver.go` — finding stays OPEN while the only Filter calls are the primary (:134) and fallbackRules (:167) sites and none wrap the FallbackChain loop. Also confirm `grep -rn 'AllowedModels' packages/ai-gateway/internal/execution packages/ai-gateway/internal/providers` returns nothing (no downstream re-enforcement). Closed when a Filter/allowlist check is applied to FallbackChain-derived recovery targets.
- **Remediation.** Route the inline FallbackChain recovery targets through the same gate as the primary and fallbackRules paths: after building the FallbackChain targets, call `r.narrowingEngine.Filter(fbTargets, narrowing, rctx)` before appending to plan.RecoveryTargets, so ModelMatchesAllowedRefs(allowedModels) excludes any entry outside the VK's allowlist. Equivalently, add a single defense-in-depth allowedModels re-check in the executor/credential-resolution boundary so no dispatched target can ever escape the per-VK allowlist regardless of which routing path produced it.

## SEC-C1-02 — Provider-credential ciphertext is not AAD-bound to its credential id/provider — cross-credential ciphertext swap is silently decrypted and used as the wrong (higher-privilege) upstream key — **MEDIUM** (3/3)

> **STATUS: FIXED** (2026-06-09, FIX-5/A) — every provider-credential seal/open now binds Additional Authenticated Data = `keyderive.ProviderCredentialAAD(credentialID, providerID)` (shared `packages/shared/core/keyderive`). The CP `Vault`/`MultiVault` (`aes_gcm.go`) and the ai-gw `Decryptor`/`MultiDecryptor` (`decrypt.go`) take an `aad []byte` on Encrypt/Decrypt; the credential manager, rotation worker, provider-test-conn, and both create paths thread it from the row's `(id, providerID)`. A ciphertext copied from credential A's row into credential B's row now fails GCM authentication on open (`ErrDecryptFailed`) instead of yielding A's upstream key under B's identity. To make the id available at seal time, the create paths generate the credential id (and, for the create-provider-with-inline-credential path, the provider id) **app-side** before encrypting (`CreateCredentialParams.ID` / `providerstore.CreateParams.ID`), replacing the DB `gen_random_uuid()` default; the rotation worker re-seals under the same AAD so the binding survives rotation. Regression: `TestEncryptDecrypt_AADBinding_DefeatsCrossCredentialSwap` (seal under cred-A AAD → opens under cred-A, REJECTED under cred-B's AAD and under nil AAD) + AAD-exactness/distinctness in `keyderive_test`; every existing credential round-trip test updated to seal+open with matching AAD. All 5 services build; full CP + ai-gw + hub sweeps green. Doc: `credentials-architecture.md` §2.1; one-time re-entry in `prod-deploy` §5.6. Smoke: ai-gateway smoke (live model call per provider) to be run on the FIX-5/A prod deploy after provider keys are re-entered.

- **Persona:** A6 host/DB-read escalated to DB-write (also: DBA, backup/restore operator, or any SQL-injection sink that can write Credential rows)
- **Invariant:** Each stored credential ciphertext MUST be cryptographically bound to its own credential id (and provider) so that ciphertext belonging to credential-A cannot be decrypted and used under credential-B's row; AES-GCM must carry the row identity as Additional Authenticated Data.

**Preconditions.** Attacker can write the encryptedKey/encryptionIv/encryptionTag columns of the Credential table directly (DB write access, a malicious migration/seed, a restore-from-backup with row reassignment, or a SQL-injection write sink in another module). The application admin HTTP API does NOT expose raw ciphertext fields (CreateCredential/UpdateCredential accept only plaintext apiKey and encrypt server-side), so this is NOT reachable through the normal CP API — it is a defense-in-depth / integrity-binding gap that turns any DB-write primitive into a confused-deputy upstream-key swap.

**Attack steps.** 1) Provider P_low has credential C_low (a low-privilege/cheap upstream key) and provider P_high has credential C_high (a high-privilege/expensive key). Both ciphertexts are sealed with the SAME master key (raw 32-byte CREDENTIAL_ENCRYPTION_KEY, no per-row/per-provider KDF) and with nil AAD: control-plane/internal/platform/crypto/aes_gcm.go:124 `aead.Seal(nil, iv, []byte(plaintext), nil)`.
2) Attacker copies C_high's three columns {encryptedKey, encryptionIv, encryptionTag, encryption_key_id} INTO C_low's row (UPDATE "Credential" SET ... WHERE id = C_low). No application audit log fires (the write bypasses the handler) and no integrity field ties the blob to C_low's id.
3) A request that routes to provider P_low calls the gateway credential manager. GetForProvider/GetDecrypted (ai-gateway/internal/credentials/manager/manager.go:99 and :135) read C_low's now-swapped columns and call decrypt → creddecrypt.Decryptor.Decrypt (decrypt/decrypt.go:85 `d.gcm.Open(nil, iv, sealed, nil)`).
4) Because AAD is nil on both seal and open, GCM authentication SUCCEEDS on the swapped blob — it yields C_high's plaintext upstream key. The gateway sends C_high's high-privilege key upstream while attributing the call (cost, usage counters, lastUsedAt, traffic_event credentialId) to C_low. Cross-credential confused-deputy: a low-privilege/low-cost credential identity now drives a high-privilege upstream key, with attribution and billing pinned to the wrong credential.
5) Durability: the next key-rotation pass (control-plane/internal/ai/providers/handler/key_rotation.go:167-184) decrypts (AAD-blind) and re-encrypts (AAD-blind) the swapped blob in place under the new key — the mis-binding survives rotation and there is no integrity check anywhere that would surface it.

- **Affected:** packages/control-plane/internal/platform/crypto/aes_gcm.go:124 (Encrypt: nil AAD) and :179 (Decrypt: nil AAD); packages/ai-gateway/internal/credentials/decrypt/decrypt.go:85 (Open with nil AAD); packages/ai-gateway/internal/credentials/manager/manager.go:99,135 (decrypt blindly trusts row columns, no id-binding check); packages/control-plane/internal/ai/providers/handler/key_rotation.go:167-184 (re-encrypt is AAD-blind, makes the swap durable)
- **Re-check probe.** grep -nE 'gcm\.Open\(|aead\.Open\(|aead\.Seal\(' packages/control-plane/internal/platform/crypto/aes_gcm.go packages/ai-gateway/internal/credentials/decrypt/decrypt.go | grep -E ', *nil\)\s*$'  # any match = the 4th arg (AAD) is nil → ciphertext is not bound to credential id/provider. Expected after fix: zero matches (AAD arg carries the credential id).
- **Remediation.** Bind each ciphertext to its credential identity via AAD. In Encrypt/Decrypt (both the CP crypto.Vault and the gateway creddecrypt.Decryptor) pass a stable, row-pinned AAD — e.g. aad = []byte("cred:"+credentialID+":provider:"+providerID) — into Seal/Open instead of nil. Plumb credentialID (and providerID) through CreateCredential/UpdateCredentialEncryption/rotateOne on the write side and through manager.decrypt on the read side. After deploy, a one-time re-encrypt pass (the existing rotation worker) re-seals every row with the new AAD; thereafter any swapped blob fails GCM authentication (Open returns ErrDecryptFailed) instead of silently yielding the wrong upstream key. This does not overlap F-0019/F-0084/SEC-M2-01 — it is a distinct record-integrity-binding gap on the main provider-credential path.
