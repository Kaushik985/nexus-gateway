# M9 — Secrets management

> Phase 1 module audit (wave 2). Confirmed findings only (≥2/3 adversarial Opus verifiers). Base = post-arch-audit + SEC-M2-01 fix.


## SEC-M9-01 — ADMIN_KEY_HMAC_SECRET drift: Control Plane silently hashes admin keys + virtual keys with a committed public constant when the production flag is unset, while the AI Gateway unconditionally requires a real secret — **HIGH** (2/3)


> **STATUS: FIXED** (2026-06-09) — deleted the committed `hmacDevFallback`/`HMAC_DEV_FALLBACK` constant (Go + seed lib.ts); `HMACSecret()` reads env verbatim; `ValidateHMACSecret()` now fails closed UNCONDITIONALLY (symmetric with the AI Gateway, no production-flag gate); dev-start.sh injects a per-developer random secret. Breaks the A6 CRITICAL kill-chain (offline VK/admin-hash forgery from a public constant).
- **Persona:** A6 host/log read (and A5 low-priv insider / external once the fallback is in effect — every stored key hash becomes computable from a public constant)
- **Invariant:** The single shared HMAC key behind admin-API-key and virtual-key hashing (ADMIN_KEY_HMAC_SECRET) must be a real, secret, identical value on every minting and verifying side; no side may substitute a hardcoded known constant.

**Preconditions.** Control Plane is started with ADMIN_KEY_HMAC_SECRET unset (env empty AND no yaml auth.hmacSecret bridge) and Crypto.Production left at its default false (CONTROL_PLANE_CRYPTO_PRODUCTION not set to true/1). This is the zero-value default; a real operator who migrates only INTERNAL_SERVICE_TOKEN / DATABASE_URL but forgets the production flag + HMAC secret hits it, and CP boots with no error. dev-start.sh:119 ships exactly this constant.

**Attack steps.** 1) CP boot path: cmd/control-plane/wiring/bootstrap.go:44 calls auth.ValidateHMACSecret(cfg.Crypto.Production). In apikey.go ValidateHMACSecret returns nil when ADMIN_KEY_HMAC_SECRET is empty AND production==false — so boot SUCCEEDS. 2) From then on HMACSecret() (apikey.go:29-34) returns the hardcoded constant hmacDevFallback = "nexus-gateway-default-hmac-secret" (apikey.go:18). 3) HashAPIKey (apikey.go:53-54) = HMAC-SHA256("nexus-gateway-default-hmac-secret", rawKey). This same function hashes: admin/user API keys (platform/middleware/adminauth.go:182 on the VERIFY side; users/handler/api_keys.go:83/190/271 on the MINT side) and virtual keys (ai/virtualkeys/handler/handler.go:210). 4) Teeth A — admin-key forgery within CP: the HMAC key is a public committed string, so an attacker can pick any rawKey, compute HashAPIKey(rawKey) offline, and any DB-write or hash-injection path yields a working admin key; conversely any leaked key_hash column / DB backup (A6) is now offline-forgeable and crackable because the MAC key is known. 5) Teeth B — cross-service drift: the AI Gateway's config.validate() (internal/config/config.go:417-418) hard-fails boot if ADMIN_KEY_HMAC_SECRET is empty, UNCONDITIONALLY (no production gate) — vkauth.NewAuthenticator (vkauth.go:117-126) is documented to do no dev-fallback. So in the same deployment CP stores VK hashes under the known constant while the gateway looks up VK hashes under the real secret → silent total VK-auth failure (DoS of the auth path) OR, if the deployment also feeds the gateway the dev constant, every VK in the fleet is hashed under a public key. The fail-safe is opt-IN (Crypto.Production defaults false at config.go:220/379-380), and the two services disagree on whether empty is fatal.

- **Affected:** packages/control-plane/internal/identity/authn/apikey.go:18 (hardcoded hmacDevFallback), :29-34 (HMACSecret fallback), :42-50 (ValidateHMACSecret only gates on production), :53-54 (HashAPIKey uses it); packages/control-plane/cmd/control-plane/wiring/bootstrap.go:44 (production-gated validation = the only guard); packages/control-plane/internal/platform/middleware/adminauth.go:182 (admin-key verify uses HashAPIKey); packages/control-plane/internal/ai/virtualkeys/handler/handler.go:210 (VK mint uses HashAPIKey); packages/control-plane/cmd/control-plane/config/config.go:220 & :379-380 (Crypto.Production defaults false); contrast packages/ai-gateway/internal/config/config.go:417-418 (gateway requires HMAC unconditionally, no production gate); scripts/dev-start.sh:119 ships the same constant.
- **Re-check probe.** rg -n 'hmacDevFallback|nexus-gateway-default-hmac-secret' packages/ — the hardcoded constant must not exist. Also assert symmetry: rg -n 'HMACSecret == \"\"|hmacSecret is required|ADMIN_KEY_HMAC_SECRET is required' on both CP and ai-gateway validate() must show an UNCONDITIONAL non-empty requirement on BOTH sides (no `if production` gate). Behavioural test: boot CP with ADMIN_KEY_HMAC_SECRET unset and CONTROL_PLANE_CRYPTO_PRODUCTION unset — boot must FAIL, not succeed with the fallback.
- **Remediation.** Delete hmacDevFallback and make HMACSecret() return the configured value with no substitution; make CP's config.validate() require ADMIN_KEY_HMAC_SECRET non-empty UNCONDITIONALLY (same as the AI Gateway), removing the production-flag gate entirely so dev and prod both fail-closed on an empty secret. Pre-GA there are no installed users, so drop the dev-fallback path outright rather than keeping a parallel dev behaviour. Keep dev-start.sh injecting a real per-developer random value (as it already does for CREDENTIAL_ENCRYPTION_KEY) instead of a fixed committed string.

## SEC-M9-02 — Control Plane silently falls back to the committed dev HMAC secret unless a separate, easily-omitted production flag is set (asymmetric with AI Gateway fail-closed) — **LOW** (2/3)

- **Persona:** A6 host-or-log-read (DB/keyHash exfil) with secondary A5 low-priv-insider; misconfig-enabled
- **Invariant:** The single secret behind admin-key and virtual-key hashing (ADMIN_KEY_HMAC_SECRET) must never resolve to a value committed in the public repository in any production deployment.

**Preconditions.** Control Plane is deployed without ADMIN_KEY_HMAC_SECRET set AND without crypto.production=true in its yaml (the bare CryptoConfig struct defaults Production=false) AND without CONTROL_PLANE_CRYPTO_PRODUCTION=true|1 in env. The blessed AMI path (nexus-ami/scripts/first-boot-secrets.sh generates a random secret; nexus-ami/artifacts/configs/control-plane.config.yaml sets production: true) is safe, so this triggers on a hand-rolled / minimal yaml or an operator who migrated off NODE_ENV but never set the new flag.

**Attack steps.** 1) packages/control-plane/internal/identity/authn/apikey.go:18 hardcodes hmacDevFallback = "nexus-gateway-default-hmac-secret"; HMACSecret() (line 29-34) returns this committed constant whenever ADMIN_KEY_HMAC_SECRET is empty. 2) The only boot guard, ValidateHMACSecret(production) (line 42-46), returns nil (boot succeeds) when production is false — and production is cfg.Crypto.Production, set true ONLY when yaml crypto.production:true or env CONTROL_PLANE_CRYPTO_PRODUCTION is exactly "true"/"1" (config.go:378-381). config.validate() (config.go:412-435) does NOT include HMACSecret in its required-set, so nothing else fails the boot. 3) Contrast the AI Gateway, which fails closed UNCONDITIONALLY: packages/ai-gateway/internal/config/config.go:417 returns an error if Auth.HMACSecret == "" regardless of any production flag. This asymmetry means CP is the weak link. 4) With CP running on the public-repo secret, every admin API key (nxk_) and VK is hashed via HMAC-SHA256(known_secret, rawKey) with no per-key salt and no work factor (apikey.go:50-53, used by adminauth.go:181 FindByKeyHash lookup). An A6 attacker who exfiltrates the keyHash column (DB backup, log, replica) can confirm any guessed/leaked raw key offline and pre-compute a dictionary for any low-entropy or externally-sourced key, because the keyed digest is now a public, unsalted, single-iteration function. Note: high-entropy crypto/rand keys are not directly forgeable via the hash-lookup, which is why this is hardening/info-correlation severity rather than a direct auth bypass — but it removes the keyed-secret defense-in-depth layer entirely and is a genuine silent fail-open on a real misconfig path.

- **Affected:** packages/control-plane/internal/identity/authn/apikey.go:18 (hmacDevFallback constant), :29-34 (HMACSecret fallback), :42-46 (ValidateHMACSecret gated on production bool); packages/control-plane/cmd/control-plane/config/config.go:378-381 (Production set only on exact true/1) and :412-435 (validate omits HMACSecret); asymmetry reference packages/ai-gateway/internal/config/config.go:417
- **Re-check probe.** grep -n 'hmacDevFallback' packages/control-plane/internal/identity/authn/apikey.go (constant still present) AND grep -nE 'if production' packages/control-plane/internal/identity/authn/apikey.go (guard still gated on a flag rather than unconditional). Positive re-detection: a CP boot with ADMIN_KEY_HMAC_SECRET unset and crypto.production:false should be asserted to FAIL; today it succeeds. Cross-check: git grep -n 'cfg.Auth.HMACSecret == ""' packages/control-plane/cmd/control-plane/config/config.go should return a hit (it currently does NOT — that absence is the gap vs ai-gateway).
- **Remediation.** Make CP symmetric with the AI Gateway: require ADMIN_KEY_HMAC_SECRET unconditionally in config.validate() (return error when Auth.HMACSecret == ""), and delete the hmacDevFallback constant + the dev branch in HMACSecret() so there is no committed secret to inherit at all. Dev convenience is already covered by scripts/dev-start.sh writing a real value into .env; the in-code fallback adds only risk. If a dev fallback must stay, gate the fallback itself on production==false rather than gating the boot guard on production==true, so the default (flag absent) is fail-closed.
