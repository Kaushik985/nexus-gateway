# W2–W4 — Whole-system findings (blast-radius / fail-open / supply-chain)

> Phase 3 system sweeps. Confirmed (≥2/3 adversarial Opus verifiers). Some W2 SPOFs share a root cause with module findings (cross-ref noted) but carry a distinct blast-radius remediation (split / rotate / per-purpose-derive the shared secret).


## SEC-W2-01 — ADMIN_KEY_HMAC_SECRET is a single un-rotatable value that conflates TWO distinct authentication trust domains — admin/user API-key admission AND virtual-key data-plane admission — so one leak forges both control-plane admin access and every data-plane VK, with no keyring to rotate out the compromised value — **HIGH** (3/3)

> **STATUS: FIXED (domain separation)** (2026-06-09, FIX-5/B) — closed part (a) of the invariant: admin-API-key admission and virtual-key admission no longer share one HMAC key. Both derive a class-separated sub-key from the master `ADMIN_KEY_HMAC_SECRET` via the shared `keyderive.DeriveSubkey` (HKDF-SHA256): the Control Plane hashes admin/user keys under `ClassAPIKeyAdmin` (`auth.HashAPIKey`) and virtual keys under `ClassAPIKeyVirtualKey` (new `auth.HashVirtualKey`, used by `generateVirtualKey`); the AI Gateway VK admission derives the SAME `ClassAPIKeyVirtualKey` sub-key once at `NewAuthenticator` and hashes with it. A forgery oracle / key_hash exfiltration scoped to one domain can no longer mint a credential in the other. `DeriveSubkey` is the total (never-erroring) variant — the empty-secret protection stays in the boot gates (`ValidateHMACSecret` / `config.validate`). `ADMIN_KEY_HMAC_SECRET` is unchanged (no new env var; sub-keys derived in-process), but existing `key_hash`es were computed undivided and an HMAC hash can't be re-derived without the raw key, so admin keys + VKs are **re-issued** on the FIX-5/B deploy (dev re-seed; prod-deploy §5.6 documents it). Regression: `keyderive_test` (DeriveSubkey total + admin≠vk separation + matches strict for non-empty) + `TestHashAPIKey_DomainSeparatedFromVirtualKey` (CP) + `TestHashKey_HMAC` (ai-gw, derived sub-key); all VK-hashing test helpers (envelope/proxy usage) updated to derive. All 5 build; full CP + ai-gw sweeps green. Smoke: ai-gateway smoke (VK admission end-to-end) on the FIX-5/B prod deploy after keys are re-issued.
>
> **REMAINING (part b — rotatability):** the versioned HMAC keyring (so a known-compromised secret rotates out without bulk re-issue) is NOT yet implemented — tracked as a follow-up (todo) to design alongside the KMS envelope-custody direction (HMAC is one-way, so its rotation still needs a lazy-rehash-on-auth keyring regardless of KMS). Until then, rotation = fleet re-issue (acceptable pre-GA). This finding stays partially open until part (b) lands.

- **Persona:** A6 host/log/DB-read (exfiltrates the env value or a key_hash column); secondarily A5 low-priv insider who can read the CP/ai-gateway process environment or systemd EnvironmentFile
- **Invariant:** A single secret must not be the sole gate for two independent trust domains (admin-console authentication and data-plane API authentication); and any secret backing authentication must be rotatable without re-issuing every credential, so a known-compromise has a bounded recovery path.

**Preconditions.** Single ADMIN_KEY_HMAC_SECRET provisioned (the only supported form — it is one flat env string, NOT a key map). Attacker obtains the secret value (env read, EnvironmentFile read, crash dump, config-management leak) OR exfiltrates the Account.keyHash / VirtualKey.keyHash columns and wants offline forgery. No second factor exists on either the admin-key or VK admission path.

**Attack / impact.** 1) The SAME secret keys two unrelated MACs: admin/user API keys are admitted via HashAPIKey = HMAC-SHA256(HMACSecret(), rawKey) at packages/control-plane/internal/identity/authn/apikey.go:53-56 (verified in adminauth.go:182 FindByKeyHash), and virtual keys are admitted via hmac.New(sha256.New, a.hmacSecret) at packages/ai-gateway/internal/auth/vkauth/vkauth.go:284 where a.hmacSecret is sourced from the identical ADMIN_KEY_HMAC_SECRET (vkauth.go:117 comment + config wiring). 2) Possession of the one value lets the attacker compute HashAPIKey(rawKey) for any chosen rawKey offline; combined with any DB-write or hash-injection primitive this mints a working admin key (control-plane admission) — and independently mints/forges any nvk_ VK hash (data-plane admission, victim model allowlist + quota + billing). 3) Even WITHOUT a write primitive, an A6 who exfiltrates the keyHash columns now holds an offline forgery/cracking oracle for BOTH domains simultaneously: the digest is unsalted, single-iteration HMAC, so any low-entropy or externally-sourced key is dictionary-attackable. 4) Crucially there is NO containment: unlike CREDENTIAL_ENCRYPTION_KEY (which has a CREDENTIAL_KEY_MAP keyring + rotation worker), ADMIN_KEY_HMAC_SECRET is a single string with no versioned keyring (HMACSecret() returns one os.Getenv value, apikey.go:29-34; NewAuthenticator takes one string, vkauth.go:120). Recovery from a known leak requires re-hashing and re-issuing EVERY admin key AND EVERY virtual key in the fleet under a new secret simultaneously across CP and ai-gateway, because both sides must flip in lockstep — there is no rolling rotation. The blast radius of one value is the union of the entire control-plane admin auth surface and the entire data-plane VK auth surface.

- **Affected:** packages/control-plane/internal/identity/authn/apikey.go:29-34 (HMACSecret single env value, no keyring) + :53-56 (admin/user key MAC); packages/control-plane/internal/platform/middleware/adminauth.go:182 (admin-key verify uses this hash); packages/ai-gateway/internal/auth/vkauth/vkauth.go:117-126 (same secret) + :284 (VK MAC uses it); contrast packages/control-plane/internal/platform/crypto/aes_gcm.go:187-259 (CREDENTIAL_ENCRYPTION_KEY DOES have a MultiVault keyring) — the HMAC secret deliberately has none.
- **Re-check probe.** Confirm single-domain conflation: rg -n 'HMACSecret\(\)|hmacSecret' packages/control-plane/internal/identity/authn/apikey.go packages/ai-gateway/internal/auth/vkauth/vkauth.go — both admin-key and VK paths must resolve the SAME ADMIN_KEY_HMAC_SECRET. Confirm no keyring: rg -n 'KEY_MAP|MultiDecryptor|keyring|keyID|CurrentKeyID' packages/control-plane/internal/identity/authn/ packages/ai-gateway/internal/auth/vkauth/ returns ZERO (no versioned rotation for the HMAC secret), whereas the same grep over packages/*/internal/**/crypto* and credentials/decrypt returns the credential keyring — proving the asymmetry. Closed when (a) admin-key and VK hashing use DISTINCT secrets/derived sub-keys (e.g. HKDF(masterHMAC, 'admin') vs HKDF(masterHMAC, 'vk')) so one domain's leak does not forge the other, AND (b) a versioned key map exists so a known-compromised secret can be rotated out without bulk re-issuing every key.
- **Remediation.** (1) Split the trust domains: derive two independent sub-keys from the master via HKDF — k_admin = HKDF(ADMIN_KEY_HMAC_SECRET, info='admin-api-key') and k_vk = HKDF(..., info='virtual-key') — so a side-channel/leak scoped to one domain cannot forge the other, and the admin and data planes no longer share one raw MAC key. (2) Give the HMAC secret the same versioned keyring CREDENTIAL_ENCRYPTION_KEY already has: store keyVersion alongside each keyHash, accept a comma-separated key map, and verify against the version stamped on the row — so a compromised version can be rotated out by re-hashing lazily on next use instead of a fleet-wide lockstep re-issue. (3) Defense-in-depth: stop relying on the keyed digest as the only barrier for externally-sourced/low-entropy keys — keep raw keys at crypto/rand width (already true for minted keys) and add an alarm on bulk keyHash reads.

## SEC-W2-02 — INTERNAL_SERVICE_TOKEN is a single flat shared secret across four services that simultaneously gates the Hub config-WRITE API (/api/hub/*: shadow/desired-state push) AND grants impersonate-ANY-thing authority on the internal Things API (service-token branch of DeviceOrServiceAuth where requireThingMatch no-ops) — one leak is fleet-wide config-injection + node-impersonation, with no rotation and no per-service scoping — **HIGH** (3/3)

> **STATUS: FIXED — FIX-5/C (C1 authority split + C2 impersonation gate).** (2026-06-09) Both conflated authorities are now separated.
>
> **C1 — config-WRITE authority split.** A dedicated **`HUB_CONFIG_TOKEN`** ([MUST MATCH] **CP ↔ Hub only**) gates `/api/hub` and `/api/v1/admin/alerts`; `INTERNAL_SERVICE_TOKEN` is scoped to `/api/internal/things` + the WS registration path. Control Plane verified the **sole** caller of both groups (single `hub.Client`; ai-gateway / compliance-proxy / agent make zero `/api/hub` calls). Both Hub and CP **fail closed at boot** without the token (no fallback to the old token — an empty `ServiceAuth("")` would accept an empty bearer). A data-plane service's service-token leak can no longer write fleet config. Regression: `auth_authority_split_test.go` + `HUB_CONFIG_TOKEN` required-boot tests on both config loaders.
>
> **C2 — act-as-any-thing impersonation gate.** The caller-verification gate revised the fix shape: register/heartbeat/shadow/deregister/break-glass have legitimate **service-token** callers (CP/ai-gw/compliance-proxy self-register via the thingclient HTTP fallback; compliance-proxy sends break-glass) — each honest service only ever operates on its **own** backend-service Thing (the thingclient always sends its own ThingID). So `requireThingMatch` became `requireMutationAuthority` (`internal_things.go`): device-token callers stay bound to their own id; **service-token callers may operate only on a `thingtype.IsBackendService` Thing — never an agent** (resolved via the body type for register, a `GetThing` type lookup otherwise; fail-closed on an unresolvable id). This closes the dangerous surface — a leaked fleet service token can no longer forge an arbitrary agent's shadow, flip its kill-switch via break-glass, deregister it, or upload forged exemptions. The object-authz gate sits immediately before the mutation (after cheap shape validation, so a malformed body needs no DB lookup). Regression: `internal_things_authority_test.go` (`TestRequireMutationAuthority` all branches + per-handler `ServiceTokenImpersonatingAgent` wiring) + the rewritten shadow agent-target test; the existing device-mismatch guard preserved. All 5 build; full Hub + shared sweeps green.
>
> **Residual (a) — C3 cross-backend-service impersonation — RISK-ACCEPTED (won't-fix), decided 2026-06-09.** A service token can still act as a *different backend-service* Thing — a flat token doesn't identify the calling edge. Full containment would be per-node identity (the explored options were **C3 per-edge tokens** or **per-node mTLS — "Layer B"**). **The maintainer reviewed both against the implementation cost and accepted this residual instead of closing it.** Rationale: (1) the residual is already **bounded by the shipped [[SEC-C3-02]]** — break-glass keys are per-Thing overrides only (no key is fleet-scoped), so a service token break-glassing a *different backend-service* Thing sets at most a per-node override, NOT the fleet kill-switch (IAM-gated operator endpoint); the residual is "act as one other backend-service node," strictly lower severity than the closed act-as-any-agent surface. (2) The attack requires **first compromising a backend-service process** on an operator-controlled server host — at which point that node's own identity is already breached and can act directly; per-node credentials would only narrow lateral impersonation, not prevent the breach. (3) The mesh is the same shape as the proven **agent device-token model** (each node presents its own credential to the Hub); for backend services on operator-controlled hosts the maintainer judged a single shared service token an acceptable trust assumption pre-GA. Layer B (mTLS) was **explored and cancelled** — the design surfaced that the Hub serves agents (device tokens, no client cert) and services on **one shared listener**, so `RequireAndVerifyClientCert` could not be applied without splitting the listener topology and moving the whole Hub + every agent onto TLS; the cost was judged disproportionate to narrowing an already-bounded, post-compromise residual. The B-1 service-cert scaffolding (`agentca.SignServiceCSR` / `ServiceIdentityFromCert`) was **reverted** as unused speculative surface; the mTLS path remains re-openable from the design doc if the trust model tightens (e.g. multi-tenant hosting). See `docs/developers/security/rotation-custody-substrate-design.md` "Layer B" for the full cancellation record.
>
> **Residual (b/c, documented):** (b) token versioning/rotatability without a lockstep flip (Layer A4 token-keyring — still open as a follow-up, independent of the per-node-identity question). (c) the `/api/internal/things/audit` HTTP fallback keeps its documented service-token self-report path (forged-attribution is [[SEC-C5-01]]'s domain). Env/prod-deploy lockstep in the C1 commit; no schema change for C2.

- **Persona:** A3 rogue/compromised service node (any of the 4 services holds the same token in its env), A5 low-priv insider who reads any service's environment/EnvironmentFile, A6 host/log/config-management read
- **Invariant:** A high-authority inter-service credential must be scoped to the minimum operation a given peer needs (least privilege per edge), must not double as a fleet-wide impersonation bypass, and must be rotatable; a single shared value gating both config-write authority and act-as-any-node authority is an over-broad single point of failure.

**Preconditions.** INTERNAL_SERVICE_TOKEN provisioned once and distributed identically to CP, AI-Gateway, Compliance-Proxy, and Hub (the [MUST MATCH] shared-secret pattern). Attacker obtains the value from ANY one of the four service environments (the weakest of the four custody postures sets the blast radius). No per-service or per-route scoping exists; the same string is the only credential on every internal edge.

**Attack / impact.** 1) The one value gates two qualitatively different authorities at the Hub. First, ServiceAuth(cfg.ServiceToken) protects the entire /api/hub group (packages/nexus-hub/internal/handler/routes.go:133) — the CP→Hub config-write surface (shadow/desired-state updates, fleet config push). A token holder can therefore WRITE desired state that the Hub fans out to nodes. 2) Second, the SAME token is the service-token arm of DeviceOrServiceAuth on /api/internal/things/* (routes.go:187). When a caller presents the service token (no mTLS device identity), ThingFromContext is nil, so requireThingMatch(c, operatedID) no-ops and returns without enforcing thing.ID == operatedID (packages/nexus-hub/internal/fleet/handler/hubapi/internal_things.go:45-73 — confirmed in chain SEC-C3-02). The holder can thus act as ANY thingId on every internal-things handler: shadow reports, break-glass reports (flip fleet-wide kill-switch / exemptions per SEC-C3-02), agent-audit uploads with forged attribution (per SEC-C5-01), cert renewals. 3) The validation is just subtle.ConstantTimeCompare against the one configured string (packages/nexus-hub/internal/handler/middleware.go:23, mirrored in ai-gateway runtimeapi/auth.go and compliance-proxy runtime/auth/auth.go). There is NO keyring/version: it is a single env string on all four sides (nexus-hub/internal/config/config.go:234,437,555 require it non-empty but offer no rotation). 4) Blast radius of one leak: fleet-wide config injection (via /api/hub write + via Force=true-capable internal paths), impersonation of every node type, and the ability to drive every node's desired state — recoverable only by simultaneously rotating the token in all four services in lockstep (any drift = total inter-service 403 outage, the documented most-common failure mode).

- **Affected:** packages/nexus-hub/internal/handler/middleware.go:14-29 (ServiceAuth single-token compare); packages/nexus-hub/internal/handler/routes.go:133 (/api/hub config-write gated by the token) + :187 (DeviceOrServiceAuth service-token arm); packages/nexus-hub/internal/fleet/handler/hubapi/internal_things.go:45-73 (requireThingMatch no-ops for service-token callers → act-as-any-thing); shared identical value loaded at packages/ai-gateway/internal/runtimeapi/auth.go:23-47, packages/compliance-proxy/internal/runtime/auth/auth.go:34-82, packages/control-plane/internal/handler/helpers.go:44-45, packages/nexus-hub/internal/config/config.go:234,437,555. No keyring anywhere (single os.Getenv string on all four).
- **Re-check probe.** Confirm one value gates both config-write and act-as-any-thing: rg -n 'ServiceAuth\(|DeviceOrServiceAuth\(' packages/nexus-hub/internal/handler/routes.go — both must take cfg.ServiceToken. Confirm the impersonation bypass survives: rg -n 'requireThingMatch' packages/nexus-hub/internal/fleet/handler/hubapi/internal_things.go and verify it early-returns true/no-op when ThingFromContext(c)==nil (service-token path). Confirm no rotation: rg -n 'INTERNAL_SERVICE_TOKEN|InternalServiceToken|KEY_MAP|version|rotate' across the four config.go files returns a single-value requirement with no keyring. Closed when (a) the service token is split per-edge (CP→Hub, ai-gw↔Hub, proxy↔Hub each hold a distinct credential) so one service's leak is contained to that edge, AND (b) service-token callers can no longer act as an arbitrary thingId on fleet-scoped mutations (requireThingMatch enforced or service-token denied on break-glass/shadow-write), AND (c) the token is versioned/rotatable without lockstep.
- **Remediation.** (1) Per-edge credentials, not one shared string: issue a distinct internal token (or mTLS client cert) for each service→Hub edge so the blast radius of a single compromised service is that one edge, not the whole mesh. (2) Separate the two Hub authorities: the /api/hub config-write authority and the internal-things act-as-thing authority must not be the same credential — config-write should require a dedicated CP-only credential, and the service-token arm of DeviceOrServiceAuth must NOT silently bypass requireThingMatch for fleet-scoped mutations (kill-switch/exemptions/shadow-write); require a real thing identity or a separately-scoped operator credential there (ties to SEC-C3-02/SEC-C5-01 remediation). (3) Make the token versioned/rotatable (key map keyed by version, accept old+new during a rotation window) so a known leak does not force a fleet-wide simultaneous flip with its associated 403 outage risk.

## SEC-W2-03 — CREDENTIAL_ENCRYPTION_KEY is a single master AES key (flat 32-byte, no KDF/salt, nil AAD) whose leak decrypts TWO distinct crown-jewel secret classes — every stored provider API key AND every Hub alert-channel secret — offline with zero work factor; the rotation keyring does NOT contain a leaked OLD key because historical ciphertext remains decryptable under it — **HIGH** (2/3)

> **STATUS: FIXED** (2026-06-09, FIX-5/A) — the master is no longer used directly as an AEAD key. Both the CP provider `Vault` and the ai-gw `Decryptor` HKDF-derive a **provider-credential** class sub-key (`keyderive.DeriveKey32(master, ClassProviderCredential)`), and the Hub alert-channel cipher (`secret.go`) derives a **distinct** `ClassAlertChannelSecret` sub-key from the same master — so one env value seals two classes but neither class's AEAD key IS the master, and a usage/leak scoped to one class no longer yields the other. The "no KDF" gap is closed (HKDF-SHA256 via shared `keyderive`, golden-vector-pinned so a derivation drift fails the build), and the "nil AAD" gap is closed by the row-identity AAD binding of [[SEC-C1-02]] (folded into this fix). The `CREDENTIAL_KEY_MAP` keyring is retained for version rotation. Residual, now documented not silent: a CONFIRMED master-key leak still requires rotating the underlying provider keys + alert secrets themselves (key-version rotation alone does not revoke access to already-captured ciphertext) — `credentials-architecture.md` §2.1 + `prod-deploy` §5.6 state this and the one-time re-entry. Scheme change has no legacy fallback (dev-phase); existing rows re-entered on the FIX-5/A deploy. Regression: `keyderive_test` (class separation + golden vector) + the C1-02 swap test; all 5 build, full CP/ai-gw/hub sweeps green. Smoke: ai-gateway smoke on the FIX-5/A prod deploy. (Note: SEC-W2-01 HMAC-secret split and SEC-W2-02 internal-token split are the remaining FIX-5/B + FIX-5/C sub-phases.)

- **Persona:** A6 host/log/backup/DB-read (exfiltrates the env value and the sealed columns), realized by A5 insider who reads the process environment or A4 if a weak/known key is in use
- **Invariant:** A master encryption key's compromise must be containable: distinct secret classes should not all collapse to one key, the KDF/AAD should bind ciphertext so a swap or cross-class reuse fails, and key rotation must actually retire the blast radius of a leaked key (forward secrecy of access), not merely change which key seals NEW writes.

**Preconditions.** One CREDENTIAL_ENCRYPTION_KEY (or one CREDENTIAL_KEY_MAP version) provisioned. Attacker obtains the key value (env read, EnvironmentFile, secrets-manager misconfig, the weak/known-key path of SEC-M2-02) plus a copy of the sealed columns (DB backup, replica, log). Sealed data spans both Credential rows (provider upstream keys) and Hub alert channel secrets.

**Attack / impact.** 1) The identical key seals two unrelated secret classes. Provider credentials are sealed by the CP Vault (packages/control-plane/internal/platform/crypto/aes_gcm.go:124 aead.Seal(nil, iv, plaintext, nil)) and decrypted by the ai-gateway (packages/ai-gateway/internal/credentials/decrypt/decrypt.go:85 gcm.Open(nil, iv, sealed, nil)). Hub alert-channel secrets are sealed/opened under the SAME env key (packages/nexus-hub/internal/alerts/engine/secret.go:58-74 ChannelSecretCipherFromEnv reads CREDENTIAL_ENCRYPTION_KEY). One leaked value therefore unlocks every provider upstream API key in the fleet AND every alert-channel webhook/SMTP/PagerDuty secret — two distinct vaults, one key. 2) No work factor and no KDF: the env hex is the AES key directly (aes_gcm.go:84-93; decrypt.go:29-47) — no per-row salt, no HKDF, no Argon2 — so an exfiltrated ciphertext+iv+tag triple is decryptable instantly offline; there is no brute-force cost even for the weak-key path. 3) nil AAD on both seal and open means ciphertext is not bound to its row identity (confirmed cross-credential swap in SEC-C1-02), widening misuse beyond decryption to confused-deputy reuse. 4) The rotation keyring does NOT contain a leaked key: MultiVault/MultiDecryptor (aes_gcm.go:187-274; decrypt.go:93-132) decrypt by the encryption_key_id stamped on each row, and the rotation worker re-seals rows onto CurrentKeyID (key_rotation.go:32,165-171). But every historical row keeps its old key_id until the worker re-seals it, and even after re-sealing, an attacker who already exfiltrated the OLD ciphertext+the old key value can still decrypt that captured copy — rotation changes which key seals NEW writes, it does not revoke an attacker's ability to read data captured under the old key. Blast-radius containment for a confirmed key leak is therefore effectively nil for any data the attacker already copied; the only true remediation is rotating the underlying provider keys themselves, which the system does not automate.

- **Affected:** packages/control-plane/internal/platform/crypto/aes_gcm.go:67-93 (flat key load, no KDF) + :124 (Seal nil AAD); packages/ai-gateway/internal/credentials/decrypt/decrypt.go:28-47 (flat key) + :85 (Open nil AAD); packages/nexus-hub/internal/alerts/engine/secret.go:58-74 (same env key seals the SECOND secret class); keyring that does not contain a leak: packages/control-plane/internal/platform/crypto/aes_gcm.go:187-274 + packages/ai-gateway/internal/credentials/decrypt/decrypt.go:93-132 + packages/control-plane/internal/ai/providers/handler/key_rotation.go:32,126-171.
- **Re-check probe.** Confirm two secret classes share one key: rg -n 'CREDENTIAL_ENCRYPTION_KEY' packages/control-plane/internal/platform/crypto/aes_gcm.go packages/ai-gateway/internal/credentials/decrypt/decrypt.go packages/nexus-hub/internal/alerts/engine/secret.go — all three must resolve the same env var. Confirm no KDF: rg -n 'hkdf|Argon|scrypt|pbkdf|salt|Derive' packages/control-plane/internal/platform/crypto/ packages/ai-gateway/internal/credentials/decrypt/ returns ZERO (key used raw). Confirm nil AAD (SEC-C1-02 overlap, here as a containment factor): rg -nE 'Seal\(|Open\(' aes_gcm.go decrypt.go | grep ', *nil\)'. Closed when (a) provider creds and alert secrets use derived, class-separated sub-keys (HKDF with distinct info), (b) AAD binds row identity, and (c) the operator runbook + tooling treat a master-key leak as mandating provider-key rotation, not just key_id rotation.
- **Remediation.** (1) Separate the two secret classes cryptographically: derive class-scoped sub-keys via HKDF(CREDENTIAL_ENCRYPTION_KEY, info='provider-credential') and HKDF(..., info='alert-channel-secret') so a usage scoped to one class does not directly yield the master that unlocks the other, and document them as distinct blast radii. (2) Add a real KDF + per-row AAD (folds in SEC-C1-02): seal each row with AAD = credentialId|providerId|keyId so swap/reuse fails and the ciphertext is identity-bound. (3) Treat the master key as a KEK, not a DEK: wrap a per-row/per-class DEK so master rotation re-wraps DEKs cheaply, and — most importantly — document and tool the fact that a CONFIRMED master-key leak requires rotating the underlying provider API keys and alert-channel secrets themselves (the only action that revokes an attacker's access to already-captured ciphertext); key_id rotation alone is not containment. (4) Pursue the deferred KMS/HSM custody so the master never sits as plaintext hex in an environment that A6 can read.

## SEC-W4-01 — AMI appliance build pulls and compiles all runtime infra (Valkey + valkey-search vendored submodules, NATS server, Node.js) over unverified curl/git — no pinned checksum, no GPG/signature check — so a release-asset or upstream-tag compromise (or AMI-build-time on-path MITM) silently lands attacker code in the image that intercepts all customer AI traffic — **MEDIUM** (3/3)

> **STATUS: FIXED** (2026-06-09) — every externally-fetched infra artifact is now integrity-pinned and verified fail-closed before it is compiled/installed. New `nexus-ami/scripts/lib-verify.sh` exports two helpers (sourced by the installers; non-zero return aborts the `set -euo pipefail` build): `verify_sha256 <hex> <file>` and `verify_git_commit <dir> <sha>`. (1) **Valkey core** — `install-valkey.sh` pins `VALKEY_SHA256=8d65e12c…` and `verify_sha256`s the tarball before `tar`/compile. (3) **NATS** — `install-nats.sh` pins per-arch `NATS_SHA256` (amd64 `979c6e63…` / arm64 `f12b65d7…`) from the NATS-published `SHA256SUMS`. (4) **Node** — `install-node-prisma.sh` pins per-arch `NODE_SHA256` (x64 `b4e33658…` / arm64 `dbe339e5…`) from the nodejs.org-published `SHASUMS256.txt`.
