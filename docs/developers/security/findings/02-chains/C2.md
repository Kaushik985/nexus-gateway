# C2 — Enrollment & identity

> Phase 2 chain audit. Confirmed findings only (≥2/3 adversarial Opus verifiers). Base = post-arch-audit + SEC-M2-01 fix. Each finding attacks an end-to-end trust-boundary crossing.


## SEC-C2-01 — SSO enroll by physical_id takes over an unassigned token-enrolled node — **CRITICAL** (3/3)
> **STATUS: FIXED** (2026-06-09) — the `assignment == nil` branch no longer reuses an existing thing by fingerprint alone. A world-readable DeviceFingerprint is an identifier, not an authenticator: an unowned/token-enrolled match now mints a NEW thing_id (no token overwrite, no rebind/elevation), so an attacker who knows the victim's fingerprint cannot take over its thing_id. Same-SSO-user re-enroll (assignment.UserID==userID) still dedups; another user's device still 409s (F-0329). Regression: `TestEnroll_JWT_WithFingerprint_Unowned_MintsNewThingID`.

- **Persona:** A2 valid-VK (any provisioned SSO user who can obtain an enrollment JWT), enabled by A3/A6 reading the victim host's world-readable hardware identifiers
- **Invariant:** A device token issued for one enrolled node must never be re-issued to a different principal, and a level-1 node must not be silently rebound to a new owner (and elevated to trust level 2/3) merely by presenting a matching, non-secret hardware fingerprint.
- **Arch cross-ref:** F-0329

**Preconditions.** (1) Victim node enrolled via X-Enrollment-Token (legacy path) — it therefore has physical_id populated but NO DeviceAssignment (trust_level 1). This is the normal state for fleet/kiosk/service-managed agents that are not SSO self-enrolled. (2) Attacker holds any valid enrollment JWT (their own SSO employee identity — enterprise-login mode). (3) Attacker knows the victim's DeviceFingerprint. The fingerprint is NOT a secret: ComputeDeviceFingerprint (packages/shared/core/metrics/platform/fingerprint.go:57) is a deterministic SHA-256 over hardwareUUID (/etc/machine-id, world-readable on Linux), hardwareSerial, primaryMAC, and cpuModel — any local process or low-priv user on the victim host can recompute it, and it is also exposed in Hub-side admin/node reads as physical_id.

**Attack steps.** Stage 1 (victim boot→enroll): victim agent enrolls with X-Enrollment-Token; enroll.go:291 always sends DeviceFingerprint=FP; doEnroll → RegisterThing stores PhysicalID=FP (enrollment_handler.go:486); no DeviceAssignment is created on the token path (enrollWithToken, line 282 comment), so the row sits at trust_level 1 with physical_id=FP. Stage 2 (attacker recon): attacker computes FP from the victim host's world-readable hardware IDs (or reads it from a node admin view). Stage 3 (takeover enroll): attacker POSTs /api/internal/things/enroll with `Authorization: Bearer <attacker-own-enrollment-JWT>` and body `{"thingType":"agent","deviceFingerprint":"FP"}`. enrollWithJWT verifies the JWT (it is the attacker's legitimate token — passes), then FindAgentByPhysicalID(FP) returns the victim's thingID `existing` (enrollment_handler.go:338); GetActiveDeviceAssignment(existing) returns nil because the victim never had an assignment; the `case assignment == nil` branch (line 349) reuses thingID=existing. Stage 4 (token theft + DoS): doEnroll mints a FRESH device token for the victim's thingID and returns its plaintext to the attacker, and StoreDeviceTokenHash overwrites the victim's stored hash (enrollment_handler.go:494) — the victim's existing token is now invalid (DoS: victim's WS auth via ValidateDeviceToken fails on the next reconnect). Stage 5 (elevation): UpsertDeviceAssignment binds the victim's thingID to the ATTACKER's userID, and ComputeAndStoreTrustLevel(existing) now returns 2/3 (enrollment_handler.go:442) — the level-1 node is silently rebound and elevated. Stage 6 (impersonation): attacker connects WS Path 2 with the stolen token + `id=<victim thingID>`; authenticate() returns thing.Type/ID from the store (server.go:248-252), so the attacker is now the victim node — receives the victim's config pushes (desired shadow), can send shadow_report/break-glass frames as that node, and inherits its fleet identity.

- **Affected:** /Users/zhangtiebin/workspaces/workspace-nexus/itech-nexus-gateway/packages/nexus-hub/internal/identity/handler/enroll/enrollment_handler.go:349 (the `case assignment == nil || assignment.UserID == userID:` reuse branch); enabling context: lines 337-366 (physical_id reuse), 494 (device-token hash overwrite on reused thingID), 442 (trust-level elevation); fingerprint non-secret: /Users/zhangtiebin/workspaces/workspace-nexus/itech-nexus-gateway/packages/shared/core/metrics/platform/fingerprint.go:57-67; victim sends FP unconditionally: /Users/zhangtiebin/workspaces/workspace-nexus/itech-nexus-gateway/packages/agent/internal/identity/enrollment/enroll.go:291
- **Re-check probe.** Re-open packages/nexus-hub/internal/identity/handler/enroll/enrollment_handler.go and assert the physical_id reuse switch (~line 344-357) STILL admits reuse when GetActiveDeviceAssignment returns nil: grep -n 'assignment == nil' enrollment_handler.go must return a hit inside enrollWithJWT, AND there must be NO additional guard between FindAgentByPhysicalID (line ~338) and doEnroll (line ~368) that (a) requires the existing thing to have been SSO-enrolled, or (b) requires a host-proof challenge beyond the plaintext fingerprint, or (c) refuses reuse of a thing whose enrolled_by/auth path is the X-Enrollment-Token path. Confirm enrollWithToken does NOT create a DeviceAssignment (so token-enrolled things are always assignment==nil): grep -n 'UpsertDeviceAssignment' enrollment_handler.go returns hits only inside enrollWithJWT. Confirm doEnroll overwrites the device-token hash on the reused id: grep -n 'StoreDeviceTokenHash' enrollment_handler.go. If all three hold, the takeover is live.
- **Remediation.** The F-0329 ownership gate must treat an UNASSIGNED existing thing as 'not claimable by fingerprint alone', not as 'free to claim'. Concretely: in the `assignment == nil` branch, refuse silent reuse of a thing that was enrolled via a different credential path / has no prior binding to THIS user — require proof of possession of the existing device's current credential (e.g. the caller must present the existing valid device token, or a host-bound attestation signature over a server nonce) before rebinding. A bare, world-readable fingerprint is an identifier, not an authenticator, and must never by itself transfer a thingID + mint a fresh device token. Minimum fix: only reuse-by-physical_id when the request also authenticates as the existing device (device-token-authenticated re-enroll), OR when assignment != nil && assignment.UserID == userID; otherwise mint a NEW thingID (never overwrite the victim's token hash) and surface a 409 if the operator intends a deliberate transfer (admin-mediated, audited).

## SEC-C2-02 — Attestation key trusted indefinitely — verify path ignores cert 90-day expiry and has no revocation; decoupled from device-token rotation/revoke — **MEDIUM** (3/3)

> **STATUS: FIXED** (2026-06-09) — attestation cert expiry is now enforced end-to-end. The Hub `GET /api/internal/things/:id/attestation-pubkey` response carries `certExpiresAt` (new store `GetAttestationPubKeyWithExpiry` reading `sysinfo.attestation.certExpiresAt`); the CP loader parses it into `tlsbump.AttestationKey{Key, CertExpiresAt}`, which threads through the key cache to `AttestationVerifier.Verify` — it now returns outcome `expired` (→ MITM fallback) when `now > CertExpiresAt`, so a 90-day cert actually bounds trust. A zero expiry (legacy stamp) is treated as non-expiring (fail-open). Regression: `TestVerifier_CertExpired_RejectsOtherwiseValid` / `_CertNotYetExpired_Valid` / `_CertExpiryZero_FailOpenValid` (verifier), `TestAttestationKeyCache_CertExpiresAtRoundTrips` (cache), `TestFetchAttestationPubKey_HappyPath` expiry assertion (CP loader), `TestGetAttestationPubKeyWithExpiry` (store) + `TestGetAttestationPubKey_Happy_IncludesCertExpiresAt` (endpoint). Doc: `nexus-hub-enrollment-architecture.md`. NOTE: this is also the **expiry half of SEC-M4-01 (HIGH)**; M4-01 stays open pending its revocation lever (immediate clear-on-decommission + cache invalidation).

- **Persona:** A3 rogue-node / attacker holding a stolen agent Ed25519 attestation private key
- **Invariant:** A signing key used to authorize a compliance-bypass passthrough must be bounded by its certificate validity and immediately revocable; a revoked or stale-cert agent must lose its passthrough privilege.

**Preconditions.** attestationEnabled=true fleet-wide. Attacker has exfiltrated an agent's Ed25519 attestation private key (kept in the OS keystore but extractable on a compromised host), OR an agent that was decommissioned/rotated still has its pubkey row. Cert was issued with 90-day validity (agentca/ca.go:28 certValidityDays=90).

**Attack steps.** 1) Hub signs the attestation CSR with NotAfter = now+90d and StoreAttestationPubKey records publicKey + certSerial + certExpiresAt into thing_agent.sysinfo->attestation (enrollment_handler.go:530, thing_registry.go:1230). 2) At verify time the loader endpoint GetAttestationPubKey returns ONLY sysinfo->attestation->>'publicKey' (thing_registry.go:1262-1283) — it never reads certExpiresAt or certSerial. 3) The CP key cache stamps its OWN 60s TTL as expiresAt (attestation_key_cache.go:115,142) — unrelated to the cert's notAfter. 4) AttestationVerifier.Verify (attestation.go:132-213) checks the ±5min ts replay window and the Ed25519 sig but NEVER the cert expiry or any revocation list. 5) Consequence: a stolen attestation key produces valid passthrough headers forever — past the 90-day cert expiry, and even after the agent's device token is rotated/revoked (rotation overwrites only the device-token hash at thing_registry.go:1285+; nothing clears sysinfo->attestation). The two trust roots are fully decoupled, so the documented device-token revocation control (F-0202) gives a false sense that disabling an agent stops it.

- **Affected:** packages/nexus-hub/internal/fleet/store/thing_registry.go:1262-1283 (GetAttestationPubKey returns pubkey only; stored certExpiresAt/certSerial unused at verify); packages/nexus-hub/internal/fleet/handler/hubapi/internal_things.go:68-90 (loader endpoint emits publicKey with no expiry/revocation gate); packages/compliance-proxy/internal/proxy/server/attestation.go:132-213 (no notAfter / CRL check); packages/shared/transport/tlsbump/attestation_key_cache.go:115,142 (expiresAt = 60s cache TTL, not cert validity).
- **Re-check probe.** grep -nE "expir|notAfter|NotAfter|revok|Revok|certExpiresAt|certSerial" packages/compliance-proxy/internal/proxy/server/attestation.go packages/shared/transport/tlsbump/attestation_key_cache.go  -> zero cert-validity/revocation references in the verify+cache path. grep -n "certExpiresAt\|cert_expires\|attestation->>'expiresAt'" packages/nexus-hub/internal/fleet/store/thing_registry.go  -> StoreAttestationPubKey writes it, GetAttestationPubKey (line 1262) never SELECTs it.
- **Remediation.** Read certExpiresAt back in GetAttestationPubKey and return 404/ErrUnknownAgent (or a typed expired error) once now > certExpiresAt; add a revoked flag to sysinfo->attestation that device-token revoke/rotation clears, and have the loader honor it. CP verifier should treat an absent/expired/revoked key as outcome=unknown_agent (fail-open to MITM, which is the SAFE direction here — it just re-enables compliance). Tie attestation-key lifetime to the device-token lifecycle so a single revoke kills both trust roots.

## SEC-C2-03 — SSO enrollment JWT (a device-enrollment grant) lets a low-priv user mint a fake SERVICE-type Thing — thingType is caller-controlled and unauthorized in the JWT path (F-0200 fixed only the token path) — **LOW** (2/3)

> **STATUS: FIXED** (2026-06-09) — applied the F-0200 type-authority invariant to the JWT path. `enrollWithJWT` (`enrollment_handler.go`) no longer trusts `req.ThingType`: a caller-supplied non-agent type is rejected `403 ENROLL_TYPE_FORBIDDEN` and the type is hard-pinned to `agent` (the SSO/browser enrollment JWT is a *device*-enrollment grant gated by `admin:device-enrollment.enroll`, so it may only mint agent-class Things). A service-type Thing (`ai-gateway`/`compliance-proxy`/`control-plane`) can no longer be minted via a user-obtained enrollment JWT — and thus cannot inherit that tier's desired state (providers/credentials/virtual_keys/routing_rules/…) or receive its config-invalidation pushes; service Things enroll only via the operator-issued enrollment-token path whose token row pins the type. Rejection happens BEFORE any thing is minted or desired state written (no `doEnroll`/`RegisterThing`). Regression: `TestEnrollWithJWT_ServiceTypeRejected_SEC_C2_03` (table over all three service types — each → 403 `ENROLL_TYPE_FORBIDDEN`, no thing id, and the pgxmock asserts no enrollment query ran); the existing `TestEnrollWithJWT_ThingTypeDefaultsToAgent` still passes (omitted type still defaults to agent). All 5 services build; nexus-hub full sweep green. Smoke N/A: enrollment authZ, no traffic_event/codec path. Doc: `nexus-hub-enrollment-architecture.md` (per-path thingType authority). Re-check probe now satisfied: `enrollWithJWT` contains a server-side non-agent rejection.

- **Persona:** A5 low-priv-admin / A2 valid-VK-equivalent — any authenticated nexus_user who holds the IAM permission admin:device-enrollment.enroll (the grant a normal user needs to enroll their OWN desktop agent via browser SSO).
- **Invariant:** The authorized TYPE of an enrollment credential must be authoritative server-side; a credential issued to enroll a DEVICE (agent) must not be usable to enroll a Thing of a privileged service type (ai-gateway / compliance-proxy / control-plane).
- **Arch cross-ref:** F-0200 (residual: F-0200 made the enrollment credential's authorized type authoritative for the X-Enrollment-Token path — enrollWithToken pins token.ThingType. The same invariant was never applied to the Bearer enrollment-JWT / SSO path (enrollWithJWT), which still trusts caller-supplied req.ThingType. The JWT is gated by the device verb admin:device-enrollment.enroll yet can mint service-type Things.)

**Preconditions.** device.auth.mode is enterprise-login or local-login (not mtls-only) — the documented default for SSO agent enrollment. Attacker is a normal user with admin:device-enrollment.enroll (the exact permission a self-service agent install requires) and can complete the public POST /api/agent/sso-enroll PKCE+IAM flow to obtain a 5-min enrollment JWT.

**Attack steps.** 1) Attacker runs the normal browser-SSO enrollment to obtain a valid enrollment JWT from CP POST /api/agent/sso-enroll (sso.go:160). The JWT only carries sub=userID, purpose=enrollment, aud=nexus-hub-enrollment; it does NOT carry any authorized thingType.
