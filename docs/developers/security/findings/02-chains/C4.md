# C4 — Cache key derivation & isolation

> Phase 2 chain audit. Confirmed findings only (≥2/3 adversarial Opus verifiers). Base = post-arch-audit + SEC-M2-01 fix. Each finding attacks an end-to-end trust-boundary crossing.


## SEC-C4-01 — Semantic-cache prewarm lets a low-priv admin plant attacker-chosen responses under any victim VK's scope (cross-VK cache poisoning) — **MEDIUM** (3/3)

> **STATUS: FIXED** (2026-06-09) — remediation option (a): `PrewarmCache` (`semantic_prewarm.go`) now **ignores the caller-supplied `vkScope`** and forces every prewarmed entry to a reserved, non-VK `corpus` scope (a non-empty caller scope is dropped and logged). The live L2 read resolves a lookup's scope via `resolveL2VKScope` to `vk:<id>` (or `""` under `vary_by=none`) — never `corpus` — so a prewarmed entry can never be returned under a real VK's scope. This satisfies the invariant (no entry retrievable under VK-X contains non-VK-X content): a low-priv admin can no longer tag attacker-chosen content with a victim VK's scope. Prewarm is now a non-targetable shared corpus; a future opt-in that lets a VK consult the corpus is a product feature, not a security regression (so prewarmed entries are currently inert by design). The `vkScope` API field is retained for old-client decode compatibility but server-overwritten. Regression: `TestPrewarm_ForcesCorpusScope_IgnoresCallerVKScope` (a caller's `v1:vk:victim` scope is forwarded to the AI-GW as `corpus`). Smoke N/A: admin prewarm seed scope only — the live cache read/write keying the ai-gateway smoke validates is untouched. Doc: `cp-ai-providers-virtualkeys-architecture.md`. (Deferred-as-noted hardening: also gating prewarm behind a higher verb than `admin:semantic-cache.update` is optional defense-in-depth once targeting is removed; not added to avoid IAM-catalog churn.)

- **Persona:** A5 low-priv-admin (any principal holding the single IAM action admin:semantic-cache.update)
- **Invariant:** A semantic-cache entry retrievable under VK-X's scope may only contain a response actually produced by traffic authenticated as VK-X; no actor may write cache content tagged with another VK's scope.

**Preconditions.** (1) Semantic L2 cache enabled fleet-wide (semantic_cache_config.enabled=true with an embedding provider/model configured). (2) Attacker holds admin:semantic-cache.update (the only IAM gate on POST /api/admin/semantic-cache/prewarm — semanticcache.go:123). (3) Attacker knows or can guess a victim VK id (e.g. v1:vk:nvk_xxx) OR the fleet is on varyBy="none" (then no id needed — empty scope matches every lookup). The internal AI-GW endpoint itself is behind INTERNAL_SERVICE_TOKEN (internalauth.go), so the reachable path is the CP admin route, not direct.

**Attack steps.** Stage 1 (plant): POST /api/admin/semantic-cache/prewarm with body {"entries":[{"prompt":"<text semantically near the victim's expected prompt>","response":"<attacker-chosen answer, e.g. a malicious instruction or false data>","vkScope":"<VICTIM_VK_ID>","model":"gpt-4o","ttlSeconds":604800}]}. CP PrewarmCache (semantic_prewarm.go:90-222) validates only prompt/response non-empty and ttl range — it never validates that vkScope belongs to or is authorized for the caller — then forwards entries verbatim (semantic_prewarm.go:172-175). Stage 2 (cross boundary): AI-GW SemanticPrewarmHandler embeds entry.Prompt and writes a Valkey HASH via Writer.Write with VKScope=entry.VKScope (semantic_prewarm_endpoint.go:247) and ResponseBody = the attacker's Response (semantic_prewarm_endpoint.go:252). With the post-arch-audit folded entry-key (client.go:253-269 hashing input+vk_scope+kind+provider/model) this is a clean, dedicated key — no collision needed; the attacker simply supplies the victim's scope. Stage 3 (victim consumes): victim VK sends a real request whose prompt is within cosine threshold of the planted prompt. proxy_l2.tryL2Lookup computes vkScope=resolveL2VKScope(rec,varyBy) = the victim VK id (proxy_l2.go:84-85, or "" under varyBy=none proxy_l2.go:82-83). Reader.Read → Client.Lookup issues FT.SEARCH whose ONLY isolation filter is @vk_scope:{<victim>} (lookup.go:82). The planted entry carries exactly that tag, so it is returned and served as a cache HIT (proxy_l2.go GatewayCacheKindSemantic). The poison-list check (lookup.go:512) keys on entryKey+vkScope and is not triggered for an attacker-planted entry. Result: the victim receives an attacker-fabricated LLM response across the VK isolation boundary.

- **Affected:** packages/control-plane/internal/ai/cache/handler/semantic_prewarm.go:59,90-135,172-175 (vkScope accepted + forwarded with no authorization/existence check); packages/control-plane/internal/ai/cache/handler/semanticcache.go:123 (route gated only on admin:semantic-cache.update); packages/ai-gateway/internal/ingress/debug/semantic_prewarm_endpoint.go:247,252 (caller-controlled VKScope + Response written to Valkey); packages/ai-gateway/internal/cache/semantic/lookup.go:82 (@vk_scope is the sole read-side isolation tag).
- **Re-check probe.** grep -n 'VKScope' packages/control-plane/internal/ai/cache/handler/semantic_prewarm.go && grep -nE 'Valid|authorize|belongs|ownVK|known.*vk|caller.*vk' packages/control-plane/internal/ai/cache/handler/semantic_prewarm.go ; echo 'finding holds while line 59 forwards vkScope verbatim AND the second grep returns no per-caller VK authorization of the vkScope field'
- **Remediation.** Do not let the prewarm caller choose an arbitrary vk_scope. Options (pick one): (a) drop the vkScope field from the prewarm API entirely and always write entries with an explicit fleet/FAQ scope (e.g. vk_scope="corpus") that the read path only consults when an admin opts a VK into corpus-backed answers — keeps planted answers out of real VK lanes; or (b) validate in PrewarmCache that every entry.vkScope names a VirtualKey that exists AND that the authenticated admin is authorized to write cache content for it (resolve via the VK store + IAM scope check) before forwarding; reject with 400/403 otherwise. Additionally treat semantic-cache prewarm as a higher-privilege action than admin:semantic-cache.update (which currently also gates benign config reads/writes) given it can inject content into arbitrary VK lanes.
