# Service call framework

## §1 — Scope

The service-call framework is the bidirectional coordination layer between the Nexus Hub and the four backend services + the agent. Every service that calls itself a "Thing" — `control-plane`, `ai-gateway`, `compliance-proxy`, `agent`, and the Hub itself in self-registration mode — uses the same client library to register, receive config pushes, report applied state, and stream telemetry.

In scope:

- Client implementation: `packages/shared/transport/thingclient/` (`client.go`, `http.go`, `shadow.go`, `outcomes.go`, `opsmetrics.go`, `mq.go`).
- Hub-side endpoints: `/ws` (WebSocket) and `/api/internal/things/*` (HTTP fallback + agent-only endpoints), mounted in `packages/nexus-hub/internal/handler/routes.go` and served by `packages/nexus-hub/internal/ws/server.go` + `packages/nexus-hub/internal/fleet/handler/hubapi/`.
- Authentication: `packages/nexus-hub/internal/identity/handler/enroll/device_auth.go` (`DeviceOrServiceAuth` middleware accepts either an internal service token or an agent device token).
- Identity types: `packages/shared/schemas/thingtype/thingtype.go` (the 5-value `ThingType` enum) + `tools/db-migrate/schema.prisma` `Thing` model.

Out of scope (covered by sibling docs):

- The semantics of *what* gets pushed inside `config_changed` (Cat A / Cat B keys, configKey registry) — [[thing-config-sync-architecture]].
- The Thing/Shadow data model and reported-outcomes ledger — [[thing-model]].
- How `registry.nexusHubUrl` reaches `Config.HubURL` at boot — [[service-bootstrap-config-architecture]].
- NATS JetStream — a third, independent channel for high-volume event streaming alongside this framework — [[mq-architecture]].
- Agent enrollment — how an agent obtains its first device token before it can use this framework — [[agent-identity-enrollment-architecture]].

## §2 — Thing identity dimensions

### 2.1 — Common dimensions (all five ThingTypes)

Every Thing has the same identity fields:

| Field | Where it lives |
|---|---|
| `ThingType` | enum in `packages/shared/schemas/thingtype/thingtype.go` — `Agent`, `AIGateway`, `ComplianceProxy`, `ControlPlane`, `NexusHub` |
| `ThingID` | DB primary key `thing.id`; carried on every WS frame and HTTP request as the `id` query param or `X-Thing-Id` header |
| `PhysicalID` | DB `thing.physical_id` (nullable); the stable natural key |
| `ThingName` | display label; the Hub falls back to `ThingID` when empty |
| `Token` | bearer credential — kind depends on Thing type (see 2.5) |

The combination `(ThingType, ThingID)` is unique. The combination `(ThingType, PhysicalID)` is also unique **but only for agents** — see 2.3 point (4).

### 2.2 — Service shape (`control-plane`, `ai-gateway`, `compliance-proxy`, `nexus-hub`)

For services, the `ThingID` is **predictable at boot**:

- Method A — yaml `id` field. Operators set `id: "cp-east-1"` in `<svc>.config.yaml` to pin the ID across restarts.
- Method B — auto-derived from `hostname + type + port` when the yaml `id` is empty. Works well for container deployments where each pod's hostname is already unique.

The `PhysicalID` field is optional for services; when empty it stays `NULL` in the DB and does not participate in the partial unique constraint described in 2.3 point (4).

The service `Token` is the `INTERNAL_SERVICE_TOKEN` env var — a **deployment-wide** secret shared by every service instance and the Hub (cross-service `[MUST MATCH]` per the env-only secrets binding). Adding more instances does not require minting more tokens.

A service can boot, register, and start exchanging config the moment it has its yaml + the env token. There is no pre-registration step.

### 2.3 — Agent shape — five departures from the service shape

The agent's identity model differs in five concrete ways. All five are visible in code.

**(1) Enrollment is required before registration.** Agent startup checks `enrollMgr.IsEnrolled()` in `packages/agent/cmd/agent/cmd_run.go`. An unenrolled agent blocks in `runPendingEnrollment` until an operator approves it. Enrollment runs through a separate, unauthenticated-by-design endpoint family (`packages/nexus-hub/internal/identity/handler/bootstrap/agent_bootstrap.go` documents this — "pre-enrollment agents have no client credential"). Services skip this step entirely; they go straight from yaml + env token to registration.

**(2) `ThingID` is generated by the Hub at enroll time and persisted locally by the agent.** The agent does not pick its own ID. `packages/nexus-hub/internal/identity/handler/enroll/enrollment_handler.go` `resolveThingID` either echoes a client-supplied ID or generates a random one when the client did not propose one. The agent caches the resulting ID in its credential directory and reads it back via `enrollMgr.ThingID()` on every subsequent boot. The resulting `ThingID` is opaque (no hostname, no human-meaningful structure), the opposite of the service convention.

**(3) `PhysicalID` is mandatory and is the hardware fingerprint.** For agents, the `PhysicalID` is a 32-hex SHA-256 of `IOPlatformUUID + serial + MAC + cpu brand`, computed by the `shared/opsmetrics` package and stamped into the thingclient `Config.PhysicalID` field. The doc-comment in `packages/shared/transport/thingclient/client.go` describes this contract on the `PhysicalID` field. The purpose is **reinstall idempotency**: an agent reinstalled on the same hardware reuses the same `ThingID`. `enrollment_handler.go` performs the lookup — when `(type='agent', physical_id=X)` already exists it returns the existing `ThingID` rather than minting a new one.

**(4) A partial DB unique constraint enforces (3).** Defined in `tools/db-migrate/migrations/20260521000000_thing_physical_id_column/migration.sql` (Prisma's `@@unique` cannot express the `WHERE` clause, so the constraint is declared as raw SQL): `UNIQUE (type, physical_id) WHERE type='agent' AND physical_id IS NOT NULL`. The reasoning is documented in the `schema.prisma` `Thing` model docstring: services use yaml-configured IDs in `thing.id` rather than `physical_id`, so a constraint that applied to all rows would force them either to NULL the column or collide.

**(5) Authentication is via device token, not service token.** The agent sends the device token issued at enroll time as the Bearer header, accompanied by an `X-Thing-Id` request header (or a `?id=` query parameter fallback). Hub-side, `enroll/device_auth.go` hashes the bearer via `agentca.HashDeviceToken` and looks it up in the registry store — not a constant-time compare, because there are N device tokens (one per agent) rather than one cluster-wide secret. The same `DeviceOrServiceAuth` middleware accepts either path on `/api/internal/things/*`.

### 2.4 — Multi-instance semantics

The Thing model supports many instances of every type — services included. Distinguishing instances is a matter of how the `ThingID` is differentiated:

- **Service multi-instance** — operators run multiple control-planes, AI Gateways, compliance proxies, or Hubs for HA, capacity, or regional placement. ID differentiation uses method A (yaml `id`, e.g. `cp-east-1` / `cp-east-2`) for explicit control, or method B (hostname-derived) for container fleets. All instances share the same `INTERNAL_SERVICE_TOKEN`.
- **Agent multi-instance** — every enrolled endpoint device is one Thing row with its own random `ThingID` and its own device token. The N-agents case is the default for any non-trivial deployment.

Both paths produce one row per instance in `thing`; the `(ThingType, ThingID)` pair is what distinguishes them on the wire and in the DB. For agents the `(ThingType, PhysicalID)` pair adds a second uniqueness guard for reinstall idempotency.

### 2.5 — Authentication summary

| Caller | Token kind | Where it's sent | Hub-side validator |
|---|---|---|---|
| Service | `INTERNAL_SERVICE_TOKEN` (env, deployment-wide) | `Authorization: Bearer <tok>` header + WS subprotocol on dial | constant-time compare in `enroll/device_auth.go` |
| Agent | device token (issued at enrollment, one per agent) | `Authorization: Bearer <tok>` header + `X-Thing-Id` header (or `?id=` query) | SHA-256 hash + registry-store lookup in `enroll/device_auth.go` |

A bad token returns HTTP 401 from the HTTP path. The WS path closes the connection with an abnormal-closure code if the upgrade-time check fails.

## §3 — Two transport channels (WS primary + HTTP fallback)

### 3.1 — WebSocket (primary)

The Thing dials the Hub at `/ws`. The dial URL carries the Thing's identity as query parameters — `id`, `type`, `name`, `version`, `address`, `metricsUrl`, `managementUrl`, `role`, `runtimeApiUrl`, `physicalId` — built by `buildWSDialURL` in `packages/shared/transport/thingclient/client.go`. The bearer token rides in the `Authorization` header **and** in the WS subprotocols list as `["nexus.bearer", <token>]`, giving the Hub two equivalent ways to extract it during upgrade.

The Hub serves the upgrade via `packages/nexus-hub/internal/ws/server.go` `Server.HandleUpgrade`. Once connected, the client runs two pumps:

- `readPump` — receives Hub messages and dispatches them in `handleHubMessage`.
- `writePump` — drains two priority lanes onto the socket. The lanes are:
  - `outChControl` (capacity 32, **must-deliver**) — `shadow_report`, `shadow_report_break_glass`, `static_info`.
  - `outChMetrics` (capacity 64, **drop-tolerable**) — `metrics_sample`, `diag_event`.

Each outbound frame is encoded as JSON with a 5 s write deadline (`writeFrame`).

### 3.2 — HTTP fallback

Every operation the WS channel performs has an HTTP equivalent under `/api/internal/things/`:

| Method + Path | Purpose |
|---|---|
| `POST /register` | initial registration; returns the desired-state snapshot + desired version |
| `POST /heartbeat` | periodic liveness poll; returns `DesiredVer` + optional desired map for delta detection |
| `POST /shadow` | upload reported state + reported-outcomes ledger |
| `GET /config?type=&id=` | pull the full desired snapshot on version mismatch |
| `GET /config/:key` | pull one configKey by name |
| `POST /deregister` | cooperative shutdown notice |

Agent-only endpoints registered alongside (same auth, same route group):

| Method + Path | Purpose |
|---|---|
| `POST /audit` | agent audit log upload |
| `POST /exemption` | agent exemption-grant upload |
| `GET /update-check` | software update poll |
| `POST /renew-cert` | device certificate renewal |
| `GET /:id/attestation-pubkey` | attestation public key fetch |

Plus one service-only endpoint:

| Method + Path | Purpose |
|---|---|
| `POST /agent-audit` | normalized agent audit ingested by compliance-proxy / ai-gateway via MQ |

All of the above are mounted under the same `deviceAuth := enroll.DeviceOrServiceAuth(cfg.Store, cfg.ServiceToken)` middleware in `packages/nexus-hub/internal/handler/routes.go`, so either token kind works on either kind of endpoint (subject to the handler's own ThingType check where applicable).

### 3.3 — When does the channel flip?

The client tracks consecutive WS dial failures. Once the count reaches `Config.WSFailureThreshold` (default 3), the client switches to `ModeHTTPFallback` and starts polling `/api/internal/things/heartbeat` instead of holding a WS connection.

While in HTTP fallback, an independent retry counter keeps trying to recover the WS channel on its own exponential backoff schedule. When a WS dial finally succeeds, the client transitions back to `ModeWSConnected` and resumes the normal pump flow.

This is why services boot successfully even when the Hub is briefly unreachable: the first few dial failures demote the connection mode but do not crash the process.

## §4 — Authentication wire details

For services, the bearer is taken straight from the `INTERNAL_SERVICE_TOKEN` env var (loaded at boot per [[service-bootstrap-config-architecture]]). The Hub's `cfg.ServiceToken` field holds the same value; `enroll/device_auth.go` compares them constant-time when the header matches the `Bearer ` prefix and no `X-Thing-Id` is present.

For agents, the bearer is the device token persisted in the agent's credential directory after enrollment. The agent always sets `X-Thing-Id: <thingID>` alongside the bearer. The Hub's middleware sees both, hashes the bearer with `agentca.HashDeviceToken`, and queries the registry store via `ValidateDeviceToken`. A miss returns 401 with `"invalid or revoked device token"`.

The same middleware sits in front of every `/api/internal/things/*` route, so the channel boundary is uniform.

The `/api/hub/*` route group uses a stricter middleware (`ServiceAuth`, not `DeviceOrServiceAuth`) so agents cannot reach it. Inter-Hub calls and admin-API → Hub calls flow through `/api/hub/*` and therefore require the service token.

## §5 — Direction of calls

### 5.1 — Hub → Thing (push, WS only)

The Hub initiates a single message type, `config_changed`, which the client decodes in `handleHubMessage`:

- Full-snapshot variant: `{ Desired: map[key]ConfigState, DesiredVer: int64 }` — sent on initial connect and on Hub-side cache rebuilds.
- Per-key delta variant: `{ ConfigKey, State, DesiredVer, Force? }` — sent for individual configKey updates.

On receipt, the client invokes `OnConfigChanged(desired)` (declared in `shadow.go`). The Thing applies whatever it needs to apply, then sends a `shadow_report` (5.2) so the Hub can close the drift.

There is no HTTP equivalent for push because services typically sit behind a firewall and the Hub does not hold their addresses. When the WS is down, the Thing pulls via `GET /config[/:key]` instead — pull replaces push.

### 5.2 — Thing → Hub (upload, both channels)

**WS upload** (when connected). The two priority lanes carry:

| Lane | Message types | Semantics |
|---|---|---|
| `outChControl` (must-deliver) | `shadow_report`, `shadow_report_break_glass`, `static_info` | Backed up by the implicit ack mechanism (5.3). A drop here means drift the Hub cannot close. |
| `outChMetrics` (drop-tolerable) | `metrics_sample`, `diag_event` | Five-second enqueue timeout; if the channel is full the message is dropped and the `outbox_blocked` metric ticks. |

`PushStaticInfo`, `PushMetricsSample`, and `PushDiagEvent` (declared in `opsmetrics.go`) are the public entry points; `sendShadowReport` (in `shadow.go`) is the entry point for state reporting.

**HTTP upload** (always available; the only upload path while in fallback). The five core endpoints listed in 3.2 plus the agent-only and service-only endpoints. Each call is synchronous: the Thing waits for the HTTP response and uses its body (where applicable — e.g. the heartbeat response carries `DesiredVer`).

The reported-outcomes ledger (an apply-success / apply-failure record per configKey) is attached to every `shadow_report` automatically by `c.outcomes.Snapshot()` (`outcomes.go` → `shadow.go`). The Hub renders it on the Nodes admin page so operators can see `apply failed` with an error message rather than just `out of sync`.

### 5.3 — Implicit ack via version advance

There is no explicit ack message in this framework. A successful round-trip looks like:

1. Hub pushes `config_changed` for `(key, ver=N)`.
2. Thing's `OnConfigChanged` callback applies the change.
3. Thing sends `shadow_report` with `ReportedVer >= N`.
4. Hub sees `reported_ver >= desired_ver` and considers the drift closed.

If step 2 fails, step 3 still fires but with `ReportedVer` at its prior value plus an `ApplyError` entry in the outcomes ledger. The Hub keeps the drift open and surfaces the error to operators.

## §6 — Patterns

- **Synchronous request/response** — every HTTP endpoint in 3.2.
- **Fire-and-forget with bounded buffering** — metrics/diag-events on the WS metrics lane and MQ publishes (the `outChMetrics` queue is drop-tolerable; the MQ ring buffer drops oldest on overflow).
- **Push-then-implicit-ack** — `config_changed` → `OnConfigChanged` → `shadow_report` (5.3).
- **Heartbeat** — a periodic tick driven by `Config.HeartbeatInterval` (default 15 s). The interval is settable at runtime via `SetHeartbeatInterval`; a broadcast channel wakes any sleeping ticker so the new cadence takes effect immediately rather than waiting for the next firing.
- **Reconnect backoff** — exponential 1 s, 2 s, 4 s, 8 s, 16 s, 30 s with 0–25 % jitter, computed in `calculateBackoffFor`. The first failure is logged at DEBUG (boot races are expected); failures 2 onward log at WARN.

## §7 — Failure modes + safety

| Trigger | Behaviour |
|---|---|
| Hub unreachable at boot | Service stays up. WS dial fails → exponential backoff → after `WSFailureThreshold` consecutive failures, switches to `ModeHTTPFallback`. Applied config may be stale until Hub recovers; the Thing serves whatever it had on disk (Cat A only — Cat B has no last-known-good outside Hub). |
| WS write times out (5 s deadline) | `writeFrame` returns false, `writePump` exits. The outbound queue backs up; new control messages eventually drop with `outbox_blocked`. The `readPump` is unaffected — pushes from Hub still arrive. |
| WS dial fails after the connection was established | Mode transitions to `ModeWSConnecting`; failure counter increments; backoff applies; threshold check applies. The `OnConfigChanged` callback is not invoked while disconnected. |
| `OnConfigChanged` returns an error | `shadow_report` still fires with the **previous** `ReportedVer` and an `ApplyError` entry in the outcomes ledger. The Hub sees `desired_ver > reported_ver` and keeps the drift open; operators see the error on the Nodes page. |
| MQ buffer overflows | The ring buffer (default capacity 10 000) drops the oldest event, increments the dropped-count metric, and logs WARN. MQ unavailability does **not** block the Thing — events are queued locally and drained on reconnect. |

The framework is **boot-degraded, not boot-fatal**. Every service can start with `WSFailureThreshold` consecutive failures, run for an arbitrarily long time in `ModeHTTPFallback`, and recover transparently once Hub becomes reachable.
