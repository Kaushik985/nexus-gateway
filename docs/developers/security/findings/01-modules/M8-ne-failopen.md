# M8 — macOS NE transparent proxy fail-open (safety-critical)

> Phase 1 module audit (wave 2). Confirmed findings only (≥2/3 adversarial Opus verifiers). Base = post-arch-audit + SEC-M2-01 fix.


## SEC-M8-01 — QUIC-fallback kill-list is never validated against the system DNS/DHCP/Push allowlist; a low-priv admin can close UDP for critical Apple networking daemons fleet-wide — **MEDIUM** (2/3)

> **STATUS: FIXED (Go gates 2026-06-09; NE Swift symmetry implemented 2026-06-10, pending macOS build + on-host run-test).** The kill-list is now validated against a canonical protected-system-bundle set on BOTH Go entry points that can reach the NE-read file, covering both personas. The authoritative set lives in one shared package `packages/shared/policy/systembundles` (a SUPERSET of the NE's incomplete 13-entry list — it adds `identityservicesd`/`rapportd`/`networkserviceproxy`/the `com.apple.kerberos` namespace the finding named). `Covers(entry)` flags an entry iff it is **equal to / an over-broad ancestor of / a child of** any protected bundle (`E==P || P startsWith E+"." || E startsWith P+"."`), so a bare `com.apple` (ancestor of every `com.apple.*` daemon) and a named daemon both reject, while a legitimate QUIC app like `com.apple.Safari` / `com.google.Chrome` passes. **(1) CP write path (A5 low-priv admin):** `agent_settings.go` rejects the whole request with a clear "would disable UDP for a protected system service" error before the value enters the shadow. **(2) Agent daemon write path (A3 node that pushes the shadow directly, bypassing CP):** `WriteQUICFallbackBundlesFile` (`platformshim/quic_fallback_darwin.go`) strips protected entries and logs loudly before writing the file the NE reads — the last Go-side gate before the safety-critical NE. Regression: `systembundles_test.go` (matcher table — equal/ancestor/descendant + legit-allow, 100% cov), CP `RejectsForceQUICSystemBundle` + `RejectsForceQUICOverBroadPrefix`, agent `WriteQUICFallbackBundlesFile_StripsProtected` (hermetic via temp-dir path redirect). **NE Swift symmetry (implemented 2026-06-10 — see next paragraph):** the NE's own `systemNetworkServiceBundles` fast-decline was exact-match-only while the kill match is prefix-capable; the NE protection is now symmetric (exact-OR-prefix) over a set mirroring the shared package. That NE change is defense-in-depth behind the two Go gates shipped first; the reachable attack (admin or shadow-write closing system UDP) was already closed at the source. "Derive the set from the OS" was considered and rejected: there is no stable macOS API enumerating system networking daemons, and a static mirror of the audited shared set is reviewable while an OS-derived set could silently shrink.
>
> **NE-side spec (IMPLEMENTED 2026-06-10 in `14a98c6a2` — pending macOS build + on-host run-test, maintainer-owned via `Skill('build-agent')`):** in `TransparentProxyProvider.swift:301`, replace the exact `systemNetworkServiceBundles.contains(bundleId)` fast-decline with the same exact-OR-prefix test the kill uses (`QUICFallbackBundles.swift:51-61`), sourced from a set mirroring `packages/shared/policy/systembundles.ProtectedBundles`. Re-check probe: `rg -n 'systemNetworkServiceBundles' TransparentProxyProvider.swift` must show prefix-capable matching, not `.contains`. **What was implemented:** `systemNetworkServiceBundles` expanded from the 13-entry mixed-case set to the full 17-entry lowercase mirror of `ProtectedBundles` (adds `identityservicesd`, `rapportd`, `networkserviceproxy`, the `com.apple.kerberos` namespace); new `isProtectedSystemBundle(_:)` helper matches exact-OR-helper-prefix (`lowered == P || lowered.hasPrefix(P + ".")`), case-insensitively (signing identifiers carry mixed case, e.g. `com.apple.mDNSResponder`); the UDP fast-decline call site now uses the helper and still runs BEFORE `shouldForceFallback`, so a poisoned kill-list naming a protected daemon (or a bare `com.apple` ancestor) declines the flow — macOS routes it natively, UDP never closed. The set remains fail-SAFE only (can cause under-enforcement, never over-enforcement against admin policy), so hardcoding stays consistent with the no-enforcement-list rule. Verified: `xcrun -sdk macosx swiftc -typecheck` clean across all six extension sources (pre-existing deprecation warnings only); re-check probe passes (prefix arm present, exact-only `.contains(bundleId)` call site gone).

- **Persona:** A5 (low-priv insider admin holding device-defaults:update) — secondarily A3 (compromised node able to write the agent_settings shadow)
- **Invariant:** No process may be added to a UDP kill-list until it has been validated to NOT be a system DNS/DHCP/Push/Continuity networking daemon (CLAUDE.md NE rule 5).

**Preconditions.** Attacker holds the `deviceDefaults:update` IAM permission (PUT /settings/device-defaults — NOT super-admin-only), or can otherwise push the `agent_settings` shadow to enrolled macOS nodes. Target Macs run the NE transparent proxy with the daemon writing /var/run/nexus-agent/quic-bundles.json.

**Attack steps.** 1. The kill-list is the admin-controlled `forceQUICFallbackBundles` array. CP validates it ONLY for FORMAT — printable ASCII, no whitespace, <=200 chars, <=64 entries, dedupe — at packages/control-plane/internal/settings/handler/settings/agent_settings.go:248-289. There is NO check that an entry is a system networking daemon and NO check that an entry is a broad prefix. 2. The value flows agent_settings shadow -> daemon applier configappliers.go:175-178, which calls platformshim.WriteQUICFallbackBundlesFile(as.ForceQUICFallbackBundles, ...) writing the array VERBATIM (quic_fallback_darwin.go:29-57 — no filtering). 3. The NE reads it; QUICFallbackBundles.shouldForceFallback (QUICFallbackBundles.swift:51-61) returns true on EXACT match OR helper/child PREFIX match: `bundleId.hasPrefix(parent + ".")`. 4. On true, handleNewFlow closes the UDP flow: udpFlow.closeReadWithError(...) + closeWriteWithError(nil) (TransparentProxyProvider.swift:305-310). 5. The ONLY guard is the NE's `systemNetworkServiceBundles` fast-decline at TransparentProxyProvider.swift:301, which is (a) EXACT-match only (`.contains(bundleId)`) and (b) a fixed 13-entry hardcoded set. Because the kill check is prefix-capable but the protection is exact-only, an admin who submits ["com.apple"] makes shouldForceFallback return true for EVERY com.apple.* UDP flow whose exact bundle id is not one of the 13 protected strings. Critical Apple networking daemons that are NOT in the 13-entry set — e.g. com.apple.identityservicesd (iMessage/FaceTime signalling), com.apple.rapportd (Continuity/Handoff/AirDrop), com.apple.networkserviceproxy, com.apple.Kerberos.* — have their UDP closed. Even without the prefix trick, naming any non-listed Apple UDP daemon directly (e.g. ["com.apple.identityservicesd"]) passes CP validation and kills it. Yield: targeted UDP outage of Apple system services across every enrolled Mac, a fleet-wide availability hit reachable from a routine settings permission.

- **Affected:** packages/control-plane/internal/settings/handler/settings/agent_settings.go:248-289 (format-only validation, no system-bundle reject); packages/agent/cmd/agent/platformshim/quic_fallback_darwin.go:29-57 (verbatim write, no validation); packages/agent/cmd/agent/configappliers.go:175-178 (verbatim apply); packages/agent/platform/darwin/NexusAgent/NexusAgentExtension/QUICFallbackBundles.swift:51-61 (prefix-capable kill match); packages/agent/platform/darwin/NexusAgent/NexusAgentExtension/TransparentProxyProvider.swift:301 (exact-only, incomplete protection set) and :305-310 (UDP close)
- **Re-check probe.** rg -n 'ForceQUICFallbackBundles' packages/control-plane/internal/settings/handler/settings/agent_settings.go | rg -i 'com.apple|system|denyBundle|reservedBundle' — expect ZERO matches today (no system-bundle reject). Confirm asymmetry: `rg -n 'hasPrefix\(parent' packages/agent/platform/darwin/NexusAgent/NexusAgentExtension/QUICFallbackBundles.swift` (prefix kill present) vs `rg -n 'systemNetworkServiceBundles.contains' packages/agent/platform/darwin/NexusAgent/NexusAgentExtension/TransparentProxyProvider.swift` (exact-only protection). Regression test: a Go table test asserting the CP handler / WriteQUICFallbackBundlesFile REJECTS or strips any entry that is a com.apple.* bundle or a bare com.apple prefix.
- **Remediation.** Add a system-service denylist gate that runs BEFORE the kill-list is honored, on BOTH the CP write path (reject the whole request with a clear error) and the daemon write path (defense-in-depth strip). Reject (a) any entry equal to or prefix-of a known Apple/system networking bundle namespace, and (b) over-broad prefixes such as bare com.apple / com.apple. that would prefix-match system daemons. Make the NE protection consistent with the kill semantics: apply the same exact-OR-prefix matching for the systemNetworkServiceBundles decline as for the kill, and consider deriving the protected set from the OS rather than a fixed 13-string list. Per CLAUDE.md rule 5, no process reaches a UDP kill-list without passing the DNS/DHCP/Push/Continuity allowlist check first.

## SEC-M8-02 — Post-peek relay connection establishment has no fail-open watchdog — black-holed upstream/bridge SYN hangs a claimed flow until the ~75s OS TCP timeout (residual beyond F-0315d's flow.open-only watchdog) — **LOW** (3/3)

> **STATUS: IMPLEMENTED (code) — pending macOS build + on-host run-test (maintainer-waived 2026-06-09).** The fix is written in `TransparentProxyProvider.swift`; the maintainer explicitly waived the build-here/test-here gate and owns building (via `Skill('build-agent')`) + run-testing on a macOS host. **What was implemented:** a single bounded establishment watchdog (`relayEstablishTimeout = 5s`, the upper end of the 2–5s spec range so a slow-but-working WAN handshake is not reset spuriously; reuses the proven `TimeoutGuard` + `queue.asyncAfter` idiom already at the 500ms peek site — no new untested KVO mechanism on this safety-critical surface) armed immediately after `createTCPConnection` in BOTH `relayInspectViaBridgePostPeek` and `relayPassthroughPostPeek`. **Bridge path** (loopback inspect): on timeout → `cancel()` + fall back to `relayPassthroughPostPeek` — consistent with the existing `writeErr` branch, and the passthrough it cascades to arms its OWN watchdog, so the window stays bounded end-to-end. **Passthrough path** (terminal, no further fallback): on timeout → `cancel()` + new `failOpenResetFlow` helper (the `closeReadWithError(ETIMEDOUT)` + `closeWriteWithError(nil)` + `completeFlow` reset shape from commit `61b597d93`) → app gets immediate ECONNRESET, macOS retries natively. **Disarm signal** is the first evidence the connection is usable: the peek/header write completion when a peek exists; otherwise the first app→remote write success (`relayFlowToRemote`) OR the first remote→app read completion (`relayRemoteToFlow`) — both relay loops are threaded with the guard so a **server-speaks-first** protocol (SSH/SMTP/FTP/IMAP) through no-peek passthrough is not falsely reset (a genuinely established server silent >5s gets a fail-open reset+native-retry — acceptable, never worse than the ~75s freeze). `TimeoutGuard` one-shot semantics make the watchdog and the relay paths race-safe; `completeFlow` is idempotent so a late completion after a watchdog fire is harmless. The watchdog IS the restored viability guard (`createTCPConnection` returns a non-optional `NWTCPConnection`, so there is no literal nil to guard). Subsumes F-0315(d). **Re-check probe (now PASSES):** `rg -n 'createTCPConnection' …/TransparentProxyProvider.swift` → the two real call sites (bridge + passthrough) each arm `TimeoutGuard`+`asyncAfter` within the function; `swiftc -parse` clean; braces/parens balanced. **Remaining (maintainer-owned):** build via `Skill('build-agent')` + on-host run-test with a black-holed SYN (route a claimed flow at a routable-but-dead IP; assert reset within ~5s, not ~75s) before this lands/ships.

- **Persona:** A4 on-path-or-rogue-provider (also A2 valid-VK-holder steering a flow at an attacker-chosen host)
- **Invariant:** Once handleNewFlow returns true and the OS has handed the flow to the extension, every subsequent step on that flow's relay path must reach a bounded fail-open fallback (reset/native-retry) within a small deadline — no async relay step may leave a claimed flow waiting on a never-completing callback.
- **Arch cross-ref:** F-0315

**Preconditions.** Agent NE proxy active; a monitored app opens a TCP flow that handleNewFlow claims (returns true). The flow reaches applyDecisionAfterPeek with decision passthrough or inspect. The chosen upstream (passthrough: attacker-controlled host:port from the flow's remoteEndpoint/SNI; inspect: the 127.0.0.1:9443 bridge) accepts the SYN's routing but never completes the handshake — i.e. it black-holes the SYN (no RST, no SYN-ACK). Trivially arrangeable by A4 (a rogue/on-path provider whose firewall drops inbound SYNs) or A2 (a VK holder pointing a request at a dead-but-routable IP).

**Attack steps.** 1) handleNewFlow claims the TCP flow (returns true at TransparentProxyProvider.swift:395). 2) peekSNIThenDecide opens the flow and runs the 500ms peek; dispatchDecision gets a passthrough/inspect decision (the decision path IS bounded — requestDecision 2s + disconnect drain + 500ms peek all fail-open). 3) applyDecisionAfterPeek (:514) calls relayPassthroughPostPeek (:597) or relayInspectViaBridgePostPeek (:557). Each calls createTCPConnection(to: remoteHost/bridgeEndpoint) (:567,:599) then remoteConn.write(peeked)/bridgeConn.write(combined) (:574,:603). 4) For an NWTCPConnection whose SYN is black-holed, the connection never reaches a usable state and the write completion handler never fires; relayFlowToRemote is started ONLY inside that completion (:589-590 inspect, :609 passthrough), so the app->upstream direction never begins. There is NO asyncAfter watchdog around relay establishment (unlike the 500ms peek timer at :444 and the 2s decision timer in IPCProtocol.swift:357) and NO guard-let nil check on createTCPConnection. The claimed flow hangs until the OS-default TCP connect timeout (~75s) finally surfaces a write error, then the relay tears down. 5) Net effect: the user's app sees that one connection frozen for up to ~75s instead of an immediate reset+native-retry. Blast radius is a SINGLE flow — flowLock is only held around short dictionary ops (never across the relay), handleNewFlow already returned synchronously, and DNS/UDP/other flows are unaffected, so this is NOT a host brick. The #82 fix (commit 61b597d93) added the reset-on-failure shape but only for flow.open failures and a now-DELETED pre-peek relay family (openAndRelay/openAndRelayViaBridge, removed in a33d599b1); the surviving post-peek relay functions never regained an upstream-establishment watchdog.

- **Affected:** packages/agent/platform/darwin/NexusAgent/NexusAgentExtension/TransparentProxyProvider.swift:557-615 (relayInspectViaBridgePostPeek + relayPassthroughPostPeek — createTCPConnection at :567/:599, write-then-relay at :574/:603 with no establishment-timeout watchdog and no nil/viability guard); contrast bounded fallbacks at :444 (500ms peek) and IPCProtocol.swift:357 (2s decision). Arch finding F-0315(d) flagged only the unguarded flow.open and proposed a watchdog 'before flow.open', which does not cover the upstream/bridge connect step that runs AFTER open.
- **Re-check probe.** rg -n 'createTCPConnection' packages/agent/platform/darwin/NexusAgent/NexusAgentExtension/TransparentProxyProvider.swift then assert each relay function calling createTCPConnection also arms a bounded fallback: within relayPassthroughPostPeek/relayInspectViaBridgePostPeek, rg -n 'asyncAfter|TimeoutGuard|observeStateChanges|\.viable|maximumDuration' must be NON-empty. Negative re-detect: rg -nP 'createTCPConnection[\s\S]{0,400}\.write\(' with no preceding/wrapping asyncAfter watchdog in the same function indicates the gap persists.
- **Remediation.** Arm a single bounded watchdog (reuse TimeoutGuard + queue.asyncAfter, e.g. 2-5s) around the relay-establishment step in BOTH relayPassthroughPostPeek and relayInspectViaBridgePostPeek: if the first write completion (or an observeStateChanges/viability transition to ready) has not fired by the deadline, cancel the NWTCPConnection and apply the existing fail-open reset shape (flow.closeReadWithError + closeWriteWithError + completeFlow) so the app gets an immediate ECONNRESET and macOS routes the retry natively — the contract commit 61b597d93 established for flow.open. Also restore the createTCPConnection nil-guard lost when the pre-peek relay family was deleted. This subsumes F-0315(d) by placing the watchdog around the whole claim-to-first-relay window rather than only around flow.open.
