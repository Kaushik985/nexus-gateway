# C6 — Quota / rate-limit

> Phase 2 chain audit. Confirmed findings only (≥2/3 adversarial Opus verifiers). Base = post-arch-audit + SEC-M2-01 fix. Each finding attacks an end-to-end trust-boundary crossing.


## SEC-C6-01 — Quota policy-cache boot-load failure fails open silently and persistently — **HIGH** (2/3)
> **STATUS: FIXED** (2026-06-09) — added a `loaded` flag (distinguishes boot-load-failure from a legitimately empty policy set); `Engine.Check` emits `quota_check_failopen_total{reason="policy_cache_unloaded"}` while the cache is unloaded (observable window); `InitQuota` starts a background retry-with-backoff loop that reloads until the first success, so the unenforced window self-heals in seconds instead of persisting until an admin edit. Regression: `TestPolicyCache_LoadedFlag`.

- **Persona:** A2 valid-VK (beneficiary); A1 ext-unauth can widen the window by stressing the DB during an AI-gateway (re)start
- **Invariant:** A cost-cap'd VK must never dispatch upstream spend above its configured limit; if the quota policy set cannot be loaded the gateway must fail-closed or at minimum emit an observable/self-healing fail-open signal — not silently enforce nothing forever.

**Preconditions.** AI-gateway boots (or is restarted/redeployed) while the Postgres pool is reachable enough to pass store.New's Ping (pgx.go:59) but the subsequent QuotaPolicy/QuotaOverride/Organization SELECTs in PolicyCache.Load fail or time out — transient lock, replica lag, statement timeout under load, or a DB hiccup in the millisecond window between Ping and the quota queries. Common during a rolling deploy when the DB is also under migration/load. Requires at least one VK whose only cost ceiling comes from a QuotaPolicy/QuotaOverride row (the normal case for cents-based caps).

**Attack steps.** Stage 1 (config->cache load): At boot InitQuota calls policyCache.Load(ctx); on error it only logs ('failed to load quota policies') and RETURNS THE ENGINE ANYWAY with an empty policiesByScope/overridesByKey map (packages/ai-gateway/cmd/ai-gateway/wiring/quota.go:23-25, 33-34). Boot does not abort. Stage 2 (gateway counter/enforce): every subsequent request runs Engine.Check; for each chain level FindPolicy returns nil on the empty map (policy_cache.go:343-346) and GetOverride returns nil, so the loop hits 'continue // No limit at this level.' (enforcement.go:245-246) for EVERY level. Check resolves to Action='allow', Allowed=true unconditionally. Stage 3 (no signal): observeDecision records action='allow' (metrics.go:58-63) — indistinguishable from a legitimately-under-limit request. The fail-open counter quota_check_failopen_total only fires on a usage-cache (Redis) GET error (enforcement.go:264-273), NOT on an empty policy cache, so monitoring sees nothing. Stage 4 (no self-heal): PolicyCache.Load is invoked ONLY at boot and on a Hub config-change broadcast for quota_policies/quota_overrides/organizations (configdispatch.go:305-314). There is no periodic reload ticker (grep for ticker/NewTicker in policy/quota returns nothing) and no retry of the failed boot Load. The empty cache therefore persists — every VK in the deployment dispatches upstream with NO cost ceiling — until an admin happens to edit a quota policy/override/org, which re-triggers Load. A2 (any valid VK) exploits the window simply by sending traffic; an attacker who can apply DB pressure during a deploy (A1 flooding to trigger a restart loop, or timing requests around a known maintenance window) can make the bad-load window recur. Net effect: an unbounded financial DoS (upstream provider spend with the cost ceiling fully removed) that is invisible to ops and does not self-recover.

- **Affected:** packages/ai-gateway/cmd/ai-gateway/wiring/quota.go:23-25 (error swallowed, engine returned with empty cache); packages/ai-gateway/internal/policy/quota/enforcement.go:244-247 (nil policy => continue => allow); packages/ai-gateway/internal/policy/quota/policy_cache.go:339-359 (FindPolicy nil on empty map); packages/ai-gateway/internal/policy/quota/metrics.go:24-55 (no policy-load-failure / empty-cache counter). Distinct from the documented bounded fail-opens: F-0156 (Redis usage read, which IS metered via quota_check_failopen_total and self-heals via Backfill) and F-0175 (soft-reservation overshoot, bounded by in-flight concurrency). This path is unmetered AND non-self-healing.
- **Re-check probe.** PROBE-1 (residual present): `grep -n 'policyCache.Load(ctx); err != nil' packages/ai-gateway/cmd/ai-gateway/wiring/quota.go` returns the line AND the next non-blank line is `logger.Error(...)` with no `return nil, nil` / `panic` / fatal — i.e. boot continues on load failure. PROBE-2 (no metric): `grep -rn 'policy.*load.*fail\|empty.*cache\|observePolicyLoad' packages/ai-gateway/internal/policy/quota/metrics.go` returns no policy-load counter. PROBE-3 (no self-heal): `grep -rn 'NewTicker\|time.Tick\|periodic' packages/ai-gateway/internal/policy/quota packages/ai-gateway/cmd/ai-gateway/wiring/quota.go` returns nothing. If PROBE-1 matches AND PROBE-2/PROBE-3 are empty, the finding stands.
- **Remediation.** Make the quota policy cache load fail-closed-or-loud, not silently-empty: (1) In InitQuota, on Load error, increment a new quota_policy_load_failed_total counter and either (a) fail-closed: mark the engine 'degraded' so Check treats an unloaded/empty cache as reject (or down-grade) for any VK that is supposed to be capped, or (b) retry Load with backoff on a background goroutine until it succeeds, refusing to serve enforced-paths until the first successful load. (2) Add an explicit 'cache never successfully loaded' flag distinct from 'loaded, legitimately zero policies' so Check can distinguish 'no policy configured' (allow) from 'policy unknown due to load failure' (deny/degrade) — today both collapse to allow. (3) Add a periodic reload safety-net (e.g. 60s ticker) so a transient boot-time DB failure self-heals without waiting for an admin config edit. (4) Mirror the F-0156 observability pattern: any time Check runs against a never-loaded cache, emit quota_check_failopen_total{reason="policy_cache_unloaded"} so the unenforced window is alertable.

## SEC-C6-02 — VK-scoped quota policy/override created in the UI is silently never enforced  
> **STATUS: FIXED** (2026-06-09) — gateway `PolicyCache.Load` now canonicalizes the VK scope token (`vk`→`virtual_key`) for both policies and overrides via `canonicalQuotaScope`, so a VK quota set in the UI (stored as `vk`) is found by the engine's `virtual_key` lookup. Single-boundary fix (no DB migration, no UI/CP-write change). Regression: `TestPolicyCache_Load_NormalizesVKScope`.

### (original) — CP stores targetType/scope="vk" while the gateway engine queries "virtual_key" (config→engine vocabulary split) — **HIGH** (3/3)

- **Persona:** A5 (low-priv / ordinary admin with quota.create) — but the victim is the deployment: any VK-level cost cap or reject override an admin sets through the UI is a no-op at the gateway. A2 (valid-VK) is the silent beneficiary: its spend is unbounded by the VK cap the admin believes is active.
- **Invariant:** A VK-scoped quota policy or override that the Control Plane accepts and the UI shows as active MUST be honored by the AI Gateway enforcement engine for that VK.

**Preconditions.** Operator uses the CP UI / admin API (POST /quota-policies or /quota-overrides) to create a VK-scoped row — the only path the UI exposes for per-VK quotas. (The DB seed rows use scope/targetType='virtual_key' and DO work, which masks the bug in any seeded demo; only UI-created rows are dead. Gateway loads QuotaPolicy/QuotaOverride directly via quota.NewPolicyCache(db.Pool).)

**Attack steps.** 1. Admin observes VK `vk-abc` (persona A2) burning spend. 2. Admin creates a tightening control via the UI: POST /quota-overrides {targetType:"vk", targetId:"vk-abc", costLimitUsd: 5, enforcementMode:"reject"} — or a VK-scoped QuotaPolicy with scope:"vk". CP validates: validOverrideTargetTypes/validQuotaPolicyScopes accept ONLY "vk" (overrides.go:29, policies.go:62); there is no "virtual_key" alias. Row is stored verbatim with targetType/scope="vk". UI lists it as active. 3. Gateway PolicyCache.Load reads the column RAW with no normalization: override map key = o.TargetType+":"+o.TargetID = "vk:vk-abc" (policy_cache.go:201); policy bucket = newPolicies[p.Scope] = newPolicies["vk"] (policy_cache.go:164). 4. On each request BuildCheckChain emits the VK level as TargetType:"virtual_key" (chain.go:36) and Engine.Check calls GetOverride("virtual_key", vkID) (enforcement.go:102/211) and FindPolicy("virtual_key", …) (enforcement.go:123, 244). The "virtual_key" lookups never hit the "vk"-keyed entries. 5. The VK level resolves no override and no policy → `continue` (enforcement.go:246) → no cap, no reject. A2 keeps spending; only a looser user/org-level cap (if any) bounds it. The reject/cap the admin set is a silent no-op; no error, no log, UI says active. The codebase already knows the two vocabularies coexist: analytics.go:397-404 comments "UI/override rows use 'vk'; seed/gateway rows use 'virtual_key'" and maps them for the READ/analytics path — but the ENFORCEMENT path was never bridged.

- **Affected:** WRITE (stores "vk"): packages/control-plane/internal/ai/quota/handler/overrides.go:27-32,79,126 ; packages/control-plane/internal/ai/quota/handler/policies.go:60-65,180,248. READ-RAW (no map): packages/ai-gateway/internal/policy/quota/policy_cache.go:164 (newPolicies[p.Scope]) and :201 (key=o.TargetType+":"+o.TargetID). ENFORCE (queries "virtual_key"): packages/ai-gateway/internal/policy/quota/chain.go:36 ; packages/ai-gateway/internal/policy/quota/enforcement.go:102,123,211,244. Awareness of the split: packages/control-plane/internal/ai/quota/handler/analytics.go:73,397-404.
- **Re-check probe.** Fails (mismatch present) when BOTH hold: (a) `grep -c '"virtual_key"' packages/control-plane/internal/ai/quota/handler/overrides.go packages/control-plane/internal/ai/quota/handler/policies.go` == 0 AND those files contain `"vk": true`; (b) `grep -c '"vk"' packages/ai-gateway/internal/policy/quota/{enforcement.go,chain.go}` == 0 while those files contain `"virtual_key"`. Passes (fixed) only once one side normalizes — e.g. CP write maps "vk"→"virtual_key" before INSERT, or PolicyCache.Load canonicalizes scope/targetType on read. Equivalent runtime probe: insert QuotaOverride(targetType='vk', costLimitUsd=0.01, enforcementMode='reject') for a live VK, send a priced request on that VK, assert HTTP 429 QUOTA_EXCEEDED — currently returns 200.
- **Remediation.** Pick ONE canonical token and normalize at the boundary (dev-phase: no compat layer needed, just unify). Cleanest: make the CP write path store the same token the engine queries — in CreateQuotaOverride/CreateQuotaPolicy (and Update) map body.TargetType/Scope "vk"→"virtual_key" before persisting, and add "virtual_key" to validOverrideTargetTypes/validQuotaPolicyScopes; migrate existing "vk" rows to "virtual_key" with a one-shot UPDATE. Alternatively normalize on read in PolicyCache.Load (fold "vk"→"virtual_key" when building newPolicies key and override key). Either way, add a gateway-side enforcement test that creates a VK override via the store layer and asserts Engine.Check rejects — the existing tests seed "virtual_key" directly and so never caught the UI's "vk".
