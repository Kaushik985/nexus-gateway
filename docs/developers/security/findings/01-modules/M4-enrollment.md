# M4 — Node enrollment + attestation signer

> Phase 1 module audit (wave 2). Confirmed findings only (≥2/3 adversarial Opus verifiers). Base = post-arch-audit + SEC-M2-01 fix.


## SEC-M4-01 — Attestation pubkey grants permanent, unrevocable compliance-pipeline bypass — verifier enforces no cert expiry and no revocation lever exists — **HIGH** (3/3)

> **STATUS: FIXED** (2026-06-09). **(1) Expiry** — the attestation cert `NotAfter` rides the `GET /attestation-pubkey` response (`certExpiresAt`) → CP loader → `tlsbump.AttestationKey.CertExpiresAt` → key cache → `AttestationVerifier.Verify`, which rejects an expired cert (outcome `expired` → MITM fallback), bounding trust to ≤90 days (also closes sibling [SEC-C2-02](../02-chains/C2.md)). **(2) Revocation** — `GetAttestationPubKeyWithExpiry` now JOINs `thing` and excludes `status = 'revoked'`, so `UnenrollDevice` (which sets `status='revoked'`) stops the key being served → `404` → `unknown_agent` → MITM, bounded by the CP key cache's ≤60 s TTL. The serving query is the single trust-decision chokepoint (reversible; catches every revocation path, not just the admin route), so no separate clear-sysinfo write or new endpoint is needed — `UnenrollDevice` IS the revocation lever. The dead, non-revocation-aware `Store.GetAttestationPubKey` was deleted so no caller can bypass the guard. An immediate Hub→CP `AttestationKeyCache.Invalidate(agentID)` push to close the ≤60 s window is noted as an available optimization (not required for the invariant). Regression: `TestGetAttestationPubKeyWithExpiry/revoked_device_not_served` (store), `TestGetAttestationPubKey_RevokedDevice_Returns404` (endpoint), plus the expiry suite under C2-02. Doc: `nexus-hub-enrollment-architecture.md`.

- **Persona:** A3 rogue/compromised node (also A6 host/log-read who exfiltrates the key)
- **Invariant:** A credential that disables compliance inspection of AI traffic must be revocable and time-bounded; once a device is decommissioned or its attestation key is known-compromised, its attested traffic must stop being trusted.

**Preconditions.** Per-cluster feature flag attestationEnabled=true (config.go:147). Target node enrolled with an attestation CSR at least once, so sysinfo.attestation.publicKey is stamped. Attacker either controls the enrolled node (A3) or has read the agent's plaintext attestation-key.pem (A6, see companion finding).

**Attack steps.** 1) Enrollment signs the agent's Ed25519 CSR with a 90-day cert (agentca/ca.go:147 certValidityDays=90) and stamps the raw pubkey + certExpiresAt into thing_agent.sysinfo via StoreAttestationPubKey (thing_registry.go:1230). 2) CP's loader fetchAttestationPubKey hits GET /api/internal/things/:id/attestation-pubkey, which returns ONLY {agentId, publicKey} (internal_things.go:86-89) — certExpiresAt is never sent to CP. 3) On every attested CONNECT, AttestationVerifier.Verify (server/attestation.go:132) fetches the pubkey, checks ts-window + Ed25519 sig + nonce-LRU, and returns Valid — it performs ZERO cert parsing, expiry, or revocation check (grep for NotAfter/expir/revoc/x509 across the verify path returns empty). 4) A Valid outcome routes the request to attestationPassthrough (forward_handler.go:129-137 -> forward_handler_helpers.go:372), which forwards upstream with NO compliance pipeline, NO audit emission, and NO payload capture. 5) Because (a) the verifier ignores the 90-day cert expiry, (b) AttestationKeyCache.Invalidate (attestation_key_cache.go:151) is never called for attestation in production — the only Invalidate call sites are unrelated CacheManager categories in configdispatch.go:241-262 — and (c) no code path ever clears sysinfo.attestation on device unenroll/decommission, the pubkey is trusted FOREVER. A compromised node (or anyone holding the exfiltrated key) keeps minting valid attestation headers indefinitely, sending uninspected AI traffic past the compliance proxy long after the cert 'expires' and with no admin ability to revoke a single agent. The agent's own audit row is supposed to be the system-of-record (forward_handler.go:121), but a compromised/rogue node controls that row, so the bypass is also unaudited at CP.

- **Affected:** packages/compliance-proxy/internal/proxy/server/attestation.go:132-214 (Verify — no expiry/revocation); packages/nexus-hub/internal/fleet/handler/hubapi/internal_things.go:86-89 (returns pubkey only, drops certExpiresAt); packages/shared/transport/tlsbump/attestation_key_cache.go:151 (Invalidate dead for attestation); packages/shared/transport/tlsbump/forward_handler.go:129-137 + forward_handler_helpers.go:372-400 (Valid -> full bypass); packages/nexus-hub/internal/identity/agentca/ca.go:147 (90-day cert nobody enforces)
- **Re-check probe.** Structural assertion: `rg -n 'NotAfter|certExpiresAt|Expires|revoc|Revoke|\.Invalidate\(' packages/compliance-proxy/internal/proxy/server/attestation.go packages/shared/transport/tlsbump/attestation_key_cache.go packages/shared/transport/tlsbump/attestation.go` MUST return at least one expiry-enforcement and one production attestation-Invalidate call site once fixed; today it returns none in the verify path. Also `rg -n "sysinfo.*'\{attestation\}'|attestation.*jsonb_set|#- *'attestation'" packages/nexus-hub` should show a clear-on-decommission path (today only the set/get exist). Behavioral test: stamp a pubkey with certExpiresAt in the past, build an in-window signed header, call Verify — it MUST return outcome=expired/unknown_agent, not valid.
- **Remediation.** (1) Include certExpiresAt (and a revoked flag) in the GET /attestation-pubkey response and have AttestationVerifier.Verify reject when now > certExpiresAt, mapping to outcome=expired -> MITM fallback. (2) Wire a real revocation signal: on device unenroll/decommission/device-token-revoke, clear sysinfo.attestation (sysinfo - 'attestation') AND push an Invalidate(agentID) to CP's AttestationKeyCache (the method already exists but is dead). (3) Add an admin 'revoke attestation' lever for a single agent that survives the cache TTL. (4) Shorten the positive cache TTL acceptance of a stale-but-revoked key by treating revocation as a hard negative entry.

## SEC-M4-02 — Attestation private key stored as plaintext 0600 PEM on agent disk (not the platform keystore the CA comment claims) — chains to unrevocable compliance bypass on key theft — **MEDIUM** (3/3)

> **STATUS: IMPLEMENTED (code) — pending macOS/Windows build + on-host run-test (maintainer-waived 2026-06-09).** The attestation private key is migrated off the plaintext on-disk PEM into the existing platform keystore abstraction (`packages/agent/internal/identity/keystore`, already production-proven for the SQLCipher DB key): macOS **Keychain** (Security.framework via go-keychain), Windows **DPAPI**, Linux **0600 file** (best-effort). The maintainer explicitly waived the build-here/test-here gate and owns building (via `Skill('build-agent')`) + on-host run-testing the Keychain/DPAPI behaviour. **What was implemented:** (1) `keystore.AttestationKeyName` label + an in-process `keystore.NewMemoryStore()` (hermetic test backend). (2) The attestation `Signer` now takes a `keystore.Store` + label instead of a key path; `loadKey` does `store.Get` (cached after first load, off the request-path lock) instead of `os.ReadFile`; keystore-absent → `ErrAttestationNotEnrolled` (fail-open). (3) The enrollment `Manager` `Set`s the Ed25519 key into the keystore in `persistHubEnrollment` (removed from the on-disk file batch) and `Delete`s it in `Unenroll`; `NewManager` defaults to the real platform store, `WithKeyStore` injects the memory store in tests. (4) `cmd_run.go` wires `keystore.NewPlatformStore()` into `NewSigner`. (5) The `agentca/ca.go` comment now states the keystore reality (was "plaintext PEM on disk"). **The "chains to UNREVOCABLE/permanent bypass" half stays eliminated** by [SEC-M4-01](#sec-m4-01--attestation-pubkey-grants-permanent-unrevocable-compliance-pipeline-bypass--verifier-enforces-no-cert-expiry-and-no-revocation-lever-exists--high-33). On macOS/Windows a host/backup filesystem read no longer yields the signing key; on Linux the keystore is still a 0600 file, so the Linux residual stays BOUNDED by SEC-M4-01 (90-day expiry + ≤60s revoke), not eliminated — acceptable, and documented. **Regression:** `keystore_test.go` (memory store Get/Set/Delete + copy semantics + `GetOrCreateDBKey` stable-key), `signer_test.go` rewritten keystore-backed (all paths: happy/disabled/missing/malformed/wrong-alg/garbage/entropy/cache/invalidate + keystore-error rate-limit), `enroll_test.go` (`persist` writes the cert to disk but the KEY only to the keystore and NEVER `attestation-key.pem` on disk; `Unenroll` deletes from the keystore; a full **enroll→keystore→signer round-trip** test signs with the keystore-loaded key and verifies the Ed25519 signature against the public half). All 5 services build (on this darwin host the Keychain path compiles); agent full `go test -count=1 ./...` sweep green. **Re-check probe (now PASSES):** `rg attestation-key.pem packages/agent packages/nexus-hub` → zero production refs; signer `loadKey` uses `store.Get`, not `os.ReadFile`; the `ca.go` comment matches reality. **Remaining (maintainer-owned):** build via `Skill('build-agent')` + on-host run-test that enroll Sets and the signer Gets the key through the real Keychain (macOS) / DPAPI (Windows) before this lands/ships. This OS-keystore custody is the desktop-agent arm of the "TWO node classes" model in `rotation-custody-substrate-design.md`.

- **Persona:** A6 host/log/backup read (also A3 same-host compromise)
- **Invariant:** The private key whose possession alone disables compliance inspection of AI traffic must be held in hardware/OS-protected storage, so reading the agent's filesystem (or a backup snapshot) does not hand an attacker a forgery oracle.

**Preconditions.** attestationEnabled=true on the cluster. Attacker can read the agent's cert directory (root/same-uid on the host, a stolen disk image, or an unencrypted backup). No memory-only or Keychain/Enclave protection is in place.

**Attack steps.** 1) Enrollment writes the agent's Ed25519 attestation private key as a PKCS8 plaintext PEM via writeFileAtomic(.../attestation-key.pem, attestKeyPEM, 0600) (packages/agent/internal/identity/enrollment/enroll.go:392 + :399). The signer loads it with plain os.ReadFile (attestation/signer.go:170) — there is no Keychain / Secret Service / CNG / TPM binding, directly contradicting the Hub CA comment at agentca/ca.go:123-125 which asserts 'the agent keeps the private key in its platform keystore (Apple Keychain / Linux Secret Service / Windows CNG)'. 2) An attacker reading attestation-key.pem now holds the full Ed25519 signing key. 3) They reconstruct the canonical pre-image (tlsbump/attestation.go:61 SignatureInput: version/ts/nonce/sha256("")/agent_id), sign with the stolen key, and emit X-Nexus-Attestation on their own CONNECTs to the compliance proxy. 4) CP's verifier accepts (the key matches the stamped pubkey) and routes to attestationPassthrough — uninspected, unaudited-at-CP AI traffic. 5) Because of the companion finding (no expiry/revocation enforcement at the verifier and no decommission-time key clear), the stolen key is a PERMANENT compliance-bypass oracle: rotating the device token or re-imaging the box does not revoke it, and the 90-day cert lifetime is never enforced. The body is sha256("") in v1 so the signature is not even body-bound — one stolen key forges attestation for arbitrary request content.

- **Affected:** packages/agent/internal/identity/enrollment/enroll.go:392,399 (plaintext 0600 write); packages/agent/internal/identity/attestation/signer.go:57,170 (plain os.ReadFile, on-disk PEM); packages/nexus-hub/internal/identity/agentca/ca.go:123-125 (false 'platform keystore' comment); chains into packages/compliance-proxy/internal/proxy/server/attestation.go:132 (verifier trusts it forever)
- **Re-check probe.** `rg -n 'attestation-key.pem|loadKey|os.ReadFile' packages/agent/internal/identity` — a hardened build should show the attestation key loaded from keystore_darwin/linux/windows (Keychain/Secret Service/CNG), not a bare os.ReadFile of a PEM. Also `rg -n 'platform keystore|Apple Keychain' packages/nexus-hub/internal/identity/agentca/ca.go` — the comment MUST be corrected to match reality (plaintext PEM) or the implementation moved into the keystore so the comment becomes true.
- **Remediation.** Either (a) move the attestation private key into the same platform keystore the device mTLS key uses (the keystore_{darwin,linux,windows}.go seams already exist) so it never sits as a readable plaintext file, or (b) if v1 must keep an on-disk key, fix the ca.go comment to stop claiming keystore protection AND treat the threat model accordingly — pair it with the companion finding's revocation/expiry enforcement so a stolen key has a bounded, revocable lifetime rather than being a permanent forgery oracle.

## SEC-M4-03 — Enrollment-JWT replay guard (JTI cache) is in-memory only — a captured 5-min JWT is replayable across a Hub restart — **LOW** (3/3)

> **STATUS: FIXED** (2026-06-09) — the JTI replay guard now has a Redis L2 behind the in-memory L1. `jtiCache.MarkSeen` performs a Redis `SETNX nexus:enroll:jti:<jti>` with `TTL = exp − now` via the shared `spillupload.Dedup` (the same primitive the spill-upload flow uses), wired with zero new bootstrap plumbing through the existing `cfg.SpillDedup` → `EnrollmentAPI.JTIDedup` → `newJTICache`. A captured JWT can no longer be replayed across a Hub restart (fresh L1, but Redis still holds the JTI → SETNX miss → `JWT_REPLAYED`) or on another Hub (multi-Hub HA). A Redis error degrades to the L1 single-Hub guard so a transient outage never blocks enrollment; nil dedup keeps the legacy L1-only behaviour. Regression: `TestJTICache_RedisGuardSurvivesRestart`, `_RedisRejectsSecondHub`, `_RedisError_DegradesToL1`, `_InMemoryOnly_ReplayableAcrossRestart` (contrast). Doc: `nexus-hub-enrollment-architecture.md`. **This completes FIX-3** except the deferred SEC-M4-02 keystore migration (Mac/Windows env).

- **Persona:** A4 on-path / A6 host-or-log-read (anyone who captures a still-valid enrollment JWT) combined with a Hub restart
- **Invariant:** A single-use enrollment JWT (purpose=enrollment, JTI-guarded) must not be redeemable a second time within its validity window, even across Hub process restarts.

**Preconditions.** Attacker captures an enrollment JWT before it is redeemed or within its 5-minute TTL (enrollJWTTTL, sso.go:39) — e.g. from a proxy log, crash dump, or by being on-path during the unauthenticated /sso enroll exchange. The Hub process restarts (deploy, crash, OOM, scale event) within that 5-minute window. Only meaningful at single-Hub scope; the code comment already flags multi-Hub HA as needing a shared store.

**Attack steps.** 1. verifyEnrollmentJWT's replay guard is jtiSeen.MarkSeen (enrollment_handler.go:197), backed by jtiCache, an in-process `map[string]time.Time` (jti_cache.go:17-24). 2. On Hub restart Init() builds a fresh empty cache (enrollment_handler.go:85-88) — the comment at lines 75-79 acknowledges 'Hub restarts clear the cache'. 3. If the legitimate agent has NOT yet redeemed the JWT (or the attacker captured it pre-redemption), the JTI is not yet in any persistent store; after restart the attacker replays the captured JWT and MarkSeen returns true (first-seen), so the JWT is accepted a second time, minting a node identity bound to the victim's SSO user. The signature/aud/iss/exp checks still pass because the JWT is genuine and unexpired.

- **Affected:** packages/nexus-hub/internal/identity/handler/enroll/jti_cache.go:17-51 (in-memory map, lost on restart); guard invoked at packages/nexus-hub/internal/identity/handler/enroll/enrollment_handler.go:192-199; TTL source packages/control-plane/internal/identity/sso/handler/sso.go:39,263-266
- **Re-check probe.** Static: `rg -n "jtiSeen|jtiCache|MarkSeen" packages/nexus-hub/internal/identity/handler/enroll/` — if the JTI store is an in-process map (not Redis SETNX-with-TTL or a DB unique row), the cross-restart window exists. Confirm there is no `redis`/`SETNX`/DB-backed JTI table in the enroll package. Dynamic: redeem-or-capture a JWT, restart the Hub, replay the same JWT; expect 401 JWT_REPLAYED (current in-memory cache would accept it if the JTI was never persisted).
- **Remediation.** Back the JTI replay guard with the existing Redis instance (SETNX nexus:enroll:jti:<jti> with TTL = JWT exp - now) or a DB unique-constraint table, so the single-use property survives Hub restarts and (as a bonus) extends to multi-Hub HA — which the in-code comment already lists as the intended follow-up. Keep the in-memory cache as an L1 fast-path in front of it.
