# Nexus Gateway

Enterprise AI Traffic Gateway for compliance, routing, caching, and analytics.

## Mandatory rules

**Every rule below is binding. Waivers require explicit user approval in chat.**

- **English only (repository).** Project artifacts (`CLAUDE.md`, `.cursor/rules`, `docs/**`, source comments, user-visible UI copy, commit messages, READMEs, config/doc strings) must be English. Chat language may follow the user; anything committed stays English. Legal quotes etc. need explicit waiver.
- **Workflow discipline (binding).** Every change goes through this cycle. Each numbered sub-rule is binding in its own right.
  1. **Plan first.** Before edits / research / implementation, produce a written plan — approach, scope, risks, file touch list. Use Cursor Plan Mode or Claude Code's plan output. Do not jump straight to code.
  2. **Todo list captures every user message and stays live.** Every task request from the user becomes a todo immediately, before other work — Cursor `TodoWrite` / Claude Code `TaskCreate`. Each item specific, actionable, verifiable; update statuses as you go; never leave a falsely `in_progress` item. New requests in multi-message sessions do NOT interrupt the current task — capture them as todos, finish in-flight work to a natural commit point, then pick up the queue in order. Reordering or skipping only on explicit user instruction.
  3. **Complex tasks: Plan + Todo non-waivable.** A task is "complex" if any apply: (a) >2 files; (b) cross-cutting (multi-service, Thing-shadow contract, IAM, MQ schemas, audit events, anything in `packages/shared/`); (c) introduces/modifies an SDD epic/story; (d) touches API contract, data model, or migration; (e) touches a high-blast-radius surface (`packages/agent/platform/darwin/NexusAgent/NexusAgentExtension/**`, admin endpoint registration, IAM policies, token-field stamping, kill-switch, emergency passthrough, credential encryption). Skipping Plan or Todo on a complex task requires the user to explicitly state the reason; echo the reason in the response before proceeding.
  4. **Goal-anchored execution.** Before any non-trivial work, write the user's request as a one-line "Goal:" at the top of the plan or todo list and echo it in chat. When mid-stream constraints arrive, do NOT silently slot them in — re-state the *updated* goal in chat, capture the delta as new todos, then continue.
  5. **Self-audit before "done" — 4 questions × 2 rounds.** **Q1**: every todo completed (not deferred, not silently dropped)? **Q2**: no `TODO` / `FIXME` / `XXX` / `unimplemented` / `not implemented` / `stub` / `mock` strings in production code (test mocks are fine — grep your diff)? **Q3**: every changed code path is exercised by a real test OR explicitly acknowledged as untested with a reason? **Q4**: no "we'll fix this later" claims unless the user said so? Round 1 surfaces issues; fix each or capture as explicit follow-up todo. Round 2 verifies the fixes. Iterate until two consecutive rounds are clean.
  6. **Commit reminder, no auto-commit.** When work is done (implementation + verify + self-audit), ask the user whether to commit. Never `git commit` unsolicited. Suggest a concise message.
- **Adversarial product review + less-is-more (binding).** Two halves of the same product discipline. Apply both to any proposed feature, config knob, UI surface, menu item, behavior change, or "wouldn't it be nice if…" idea — including user proposals.

  **Half 1 — Steel-man, then attack.** Before implementing, surface in chat: (a) Is the user-facing value clear and large? (b) How often will real users hit this path? (c) Could a sensible default, an existing surface, or a convention solve it without a new knob? (d) Is the placement correct (right table / page / layer / menu) or is it being attached to the nearest convenient form? (e) Does the cost (UI surface, config burden, cognitive load, future maintenance) exceed the value? Counter-arguments go in chat BEFORE implementation; if they're strong and unanswered, push back with "I'd challenge: …" rather than building. If the user persists after seeing them, that's signal — proceed.

  **Half 2 — Less is more / delete instead of add.** Defaults must work out of the box — admins should not face a 5-tab detail page or a JSON editor for every protocol quirk. Spring-style: sensible default for every knob; configuration only when divergence is genuinely needed. When in doubt about adding a UI surface, config field, menu item, or code path: delete instead. Prefer (i) adapter/runtime auto-fills over admin-facing config (canonical pattern: Anthropic `max_tokens` default in `packages/ai-gateway/internal/providers/specs/anthropic/codec/codec.go:101-108` — adapter fills the protocol-required field from model capability when caller omits it); (ii) global settings over per-route / per-rule overrides unless real divergence is documented; (iii) one location per concept over scattering across menus; (iv) extending an existing surface over adding a new tab / page / route. New surfaces require an explicit "what user journey, how often, why nothing existing covers it" in the plan; without that, the default is no.
- **Worktree per session (binding).** Each parallel Claude Code session runs in its own `git worktree` so working tree, index, and `.git/index.lock` are private to that session. Conventions:
  1. **Spawn**: `git worktree add ./worktrees/<topic> [-b feature/<new-branch>] [<base>]`. `./worktrees/` is gitignored.
  2. **Clean up after merge**: `git worktree remove ./worktrees/<topic>`; if the branch is fully merged, also `git branch -d feature/<topic>`.
  3. **One branch per worktree** — git enforces this. For a second session on the same logical work, create a sibling branch.
  4. **Shared module coordination** — worktrees isolate the working tree, not the module graph. Two worktrees editing the same `packages/shared/<subpkg>/` file at once will conflict at merge. Coordinate at the subpackage level; pick disjoint shared sub-packages per worktree.
  5. **Commit hygiene**: `git status --short` before commit; `git log -1 --stat` after — confirms you committed what you intended.
  6. **`.git/index.lock` in your worktree** = your own prior git invocation crashed mid-flight. Investigate freely.

  Memory anchor: [[project_parallel_worktree_sessions]].
- **Handoff at context-full.** When the session is heavy with accumulated state — auto-compact threshold near, a multi-phase program closing out, >~50 tool-call turns, or after a major push — proactively offer to write a handoff document (typically `docs/handoffs/program.md` or `<program-area>/HANDOFF.md`) capturing: (a) program goal + current phase, (b) load-bearing architecture / API / data facts the next session needs, (c) work completed this session, (d) next steps + memory anchors + binding rules to pre-load. Then suggest starting a fresh session. Auto-compact summarizes but loses fidelity; a deliberate on-disk handoff file the next session reads is more reliable than relying on compact alone. Anchor the program in maintainer memory too (project-type memory entry).
- **Sub-agent dispatch discipline.** Use the Agent tool for (a) mechanical multi-file work that doesn't need main-context reasoning (test stub renames, doc cross-rewrites), (b) parallel-safe independent research lookups (Explore agent for code search), or (c) bounded scope where the agent's report-back is enough. Do NOT delegate (1) understanding a new problem, (2) decisions that change scope/contract, (3) anything that runs `git commit`, `git push`, force-pushes, or destructive ops on shared history. Every dispatched prompt MUST: (i) state goal + non-goals in the first 3 lines, (ii) state which worktree the agent is operating in if non-default, (iii) demand a 2-round self-audit before report-back, (iv) ask for a per-file list of touched paths + verification command outputs in the response. Reason: sub-agents have zero session context — terse prompts produce shallow work.
- **Real implementation only (production code).** No TODO/FIXME/XXX, no `not implemented` throws, no empty/no-op handlers (unless the spec defines them), no fake returns, no stub modules, no "demo" logic. Mocks/spies/test doubles allowed only in test code (Vitest etc.) or user-approved harnesses. If scope too large, narrow the plan with the user — do not merge placeholder production code.
- **Development-phase policy: no backward compatibility, no defer.** Pre-GA, no installed users; treat all refactoring as greenfield. (1) Delete obsolete code/tables/endpoints/configs outright when replacing them — no parallel "legacy" paths. (2) No phased compatibility rollouts — specs/plans must not contain "Phase 1 keeps old behavior, Phase 2 migrates". (3) No `@deprecated` markers kept alive, no "cleanup later" TODOs, no "will remove after X ships". (4) No data-migration code for dev-only records — fresh schema + seed is acceptable. (5) No feature flags for rollback — rollback is `git revert`. "Phase" in plans means **order of work**, not **compatibility layer**.
- **macOS Agent builds MUST go through `Skill('build-agent')`.** Never improvise `codesign` / `pkgbuild` / `productbuild` / `xcrun notarytool` / `wails build` / `swift build` calls directly — the skill is the single source of truth for signing, provisioning, notarization, launch-constraints, and the install/uninstall sequence.
- **API / menu / route changes require IAM impact review (binding).** Whenever a PR adds/moves/renames/removes an admin API endpoint, sidebar nav item, or admin route path, run the 5-step audit in `.cursor/rules/iam-impact-review.mdc` (or invoke `Skill('iam-impact-review')`) in the same PR. Drift between UI `allowedActions` and handler `iamMW(...)` produces silent 403s. Record the IAM decision ("kept on `admin:settings.read`" / "carved out as `prompt-cache`") in the plan or commit message.
- **macOS NE proxy must fail-open, never fail-closed (safety-critical).** `NETransparentProxyProvider` in `packages/agent/platform/darwin/NexusAgent/NexusAgentExtension/TransparentProxyProvider.swift` is in the host's outbound packet path — any hang/panic/silent claim-without-relay takes down the entire Mac's network (DNS, DHCP, mDNS, NTP, Apple Push, VPN). Recovery is manual `launchctl unload` + delete plist. Five rules: (1) `handleNewFlow` decides synchronously, returns `true` ONLY for flows we can fully relay; UDP without a relay = `return false` for unknown bundles. (2) Every async daemon callback has a fail-open timeout — `requestDecision` defaults to passthrough after 2s; `peekSNIThenRelay` falls through to plain relay after 500ms; never let a flow hang waiting on an absent daemon. (3) No hardcoded enforcement lists in NE Swift — bundle-ID allowlists live in the Hub-pushed `agent_settings` shadow blob; the daemon writes `/var/run/nexus-agent/quic-bundles.json`; NE reads file-only with empty-as-fail-safe. A hardcoded fallback would silently override admin policy. (4) `isLikelyXyz = true` patterns are banned — write the real condition or return false; don't flip a TODO to true and ship. (5) Validate against the system DNS/DHCP/Push list before adding any process to a kill-list — `mdnsresponder`, `configd`, `dhcpcd`, `apsd`, `nsurlsessiond`, `kdc`, `ntpd` MUST NOT have their UDP closed.
- **Adapter format-translation follows `provider-adapter-architecture.md` §3a (binding).** Any change under `packages/ai-gateway/internal/providers/specs/<name>/` (codec, stream session, error normalizer, hub_ingress) must conform to §3a Rules 1-8 before shipping: canonical = OpenAI shape; non-OpenAI adapters own their full canonical↔wire translation; per-model wire quirks stay in the adapter that talks to that wire (never in `spec_adapter.go`); extension fields ride inside `nexus.ext.<provider>.<key>` via `canonicalext`; cross-format callers canonicalize via `canonicalbridge.IngressChatToCanonical` before the codec; streaming + non-streaming parity; every prefix-list rule needs a comment citing the observed 400. Run `/adapter-conformance-check` before completion.
- **Unit test coverage ≥95% per Go package.** Every Go package in `packages/**` must hit at least 95% statement coverage under `go test -cover -count=1 ./...`, OR be listed in `scripts/.coverage-allowlist` with one of the allowed categories — (A) `cmd/*` entry point, (B) test helper, (C) DB-bound, (D) OS-bound, (E) network-infra-bound (real S3/NATS/Redis Sentinel), (F) integration-only behind build tags — and a one-line rationale. Adding to the allowlist requires explicit user approval; the long-term goal is an empty allowlist. Enforced by `scripts/check-go-coverage.sh` (pre-commit on staged Go packages via `--staged`; full sweep via `npm run check:coverage` and CI). Tests must assert **observable business behavior and named failure modes** — coverage padding (calling a function only to bump the percentage, asserting only `err == nil` without checking outputs) defeats the rule. Run `scripts/check-go-coverage.sh --strict-allowlist` to surface entries that can be removed. See `.cursor/rules/unit-test-coverage-95.mdc` for the full policy.
- **Test/skill env files live under `tests/.env.<target>` (binding).** Tests and prod-* skills (`prod-login` / `prod-deploy` / `prod-debug`) read config from `tests/.env.<target>` where target ∈ {`local`, `dev`, `prod`}. Loader contract + target selection + fail-closed safety guards + adding-new-var workflow are documented in `docs/developers/workflow/local-dev-debugging.md` "Test / skill env files" — that section carries the same binding force as this file.
- **Secrets are env-only — never yaml (binding).** No secret field (auth tokens, HMAC keys, credential-encryption keys/passphrases/salts, internal-service tokens, DB passwords) may appear in any yaml committed to the repo. Every secret has a corresponding env variable documented in `.env.example` (repo root); services read via `os.Getenv` / `bootenv`. YAML carries only service-shape + non-secret tunings (ports, timeouts, feature flags, log levels, allowlists). Adding a new yaml `secret`/`password`/`token`/`key` field requires explicit user approval. Cross-service shared secrets are tagged `[MUST MATCH]` in `.env.example`; drift between consumers is the most common source of inter-service 403s. Operational detail (local `.env` via `bootenv`, prod via systemd `EnvironmentFile=` / K8s Secret) is in `docs/developers/workflow/local-dev-debugging.md` "Environment variables".
- **Code / doc lockstep (binding).** When a PR touches code in an area covered by **any** doc — architecture (`docs/developers/architecture/`), product feature (`docs/users/features/`), OpenAPI spec (`docs/users/api/openapi/`), operator runbook (`docs/operators/ops/runbooks/`), or SDD epic/story (`docs/developers/specs/`) — every matching doc is **the single source of truth** and MUST update in the same commit. A single code change frequently touches multiple doc trees: a new admin endpoint → OpenAPI spec + architecture doc + feature doc (if UI surface) + runbook (if ops procedure). A cost-stamping change → `cost-estimation-architecture.md` + `prod-deploy-data-changes.md` (if historical recompute) + feature doc for Traffic drawer (if visible breakdown changes). The lockstep map (`scripts/doc-lockstep.config.mjs`) is the authoritative "code glob → doc files" registry. Enforced by `scripts/check-doc-lockstep.mjs` (PR-diff scan; fails if any code file in a mapped glob changed but none of the mapped docs are in the diff). Drift means the docs are wrong — fix the docs, not the lint. Skipping requires explicit user approval; record the waiver reason in the commit message or PR description. See `.cursor/rules/code-doc-lockstep.mdc` for the per-doc trigger table.
- **Configuration changes go through `configuration-architecture.md` (binding).** Any PR that adds / removes / renames a yaml field, env variable, `thing_config_template` configKey, `system_metadata` key, or publisher/receiver wiring must conform to the 4-layer model + R1-R5 invariants in `docs/developers/architecture/cross-cutting/foundation/configuration-architecture.md`. New keys: update the §7 per-key catalog AND `packages/shared/schemas/configkey/` (constants + `ValidByThingType` + `TypedRegistry`) in the same PR. Rename PRs: run `scripts/check-rename.sh OLD NEW` as the sweep gate (covers all 14 layers from §6.5); the negative grep `git grep -E '\bOLD\b'` must return zero before shipping.
- **L5 scenario landing rule (binding).** New `tests/scenarios/*_test.go` cannot land with only `go vet` / `go test -c` evidence. PR must include a live `cd tests/scenarios && NEXUS_TEST_TARGET=local GOWORK=off go test -run ^Test<NNN>$ -count=1 -v` output showing `--- PASS` or `--- SKIP` (skip must cite a specific architectural / env precondition, not a "we'll fix later"). Reason: 2026-05-21 session shipped 21 new scenarios with clean `go vet`; live run uncovered 13 shape errors (wrong column names like `virtual_key_id`/`endpoint_type`, fabricated metric `nexus_ai_gateway_requests_total`, wrong table case `iam_group` vs `"IamGroup"`, missing required body fields). Compile-clean ≠ test-effective. See `docs/developers/specs/e86-decision-log.md` D11.
- **AI Gateway / traffic_event changes require ai-gateway smoke run before "done".** Any edit that touches `packages/ai-gateway/**`, the `TrafficEvent` table or related schemas in `tools/db-migrate/schema.prisma` (`TrafficEvent`, `TrafficEventNormalized`, `traffic_event_*` migrations), the canonical/cost/cache/normalize/codec code in `packages/shared/transport/normalize/{core,codecs,extract}/`, `packages/shared/canonical*`, `packages/shared/traffic/**`, or any provider adapter under `packages/ai-gateway/internal/providers/specs/<name>/` MUST run an ai-gateway smoke before reporting the work done. Full-surface = `tests/scripts/smoke-gateway.py --all-ingress` (29-model × 4-ingress); partial = `--models <a,b,c>` or scope-limited (`--no-cache` / single-ingress) when the blast radius is provably narrow (e.g., a single adapter's stream parser) — call out the scoping decision in the completion message. Skipping the smoke requires explicit user approval. What the smoke uniquely catches: traffic_event correctness (cost, tokens, cache classification, normalized text), ingress codec parity, cross-ingress asymmetry (A passes, B silently drops a field — binding [[feedback_cache_mandatory_all_ingress]]), cache-hit threshold enforcement, and per-ingress nonce isolation. A passing run is the strongest "no regression" signal short of prod.

## Pre-edit reading (3-doc rule)

Before editing code, read **all three** so architecture, user surface, and code style stay aligned. Skipping any of the three requires explicit waiver.

1. **Architecture doc(s)** — find your edit area in `docs/developers/architecture/README.md` and open the listed doc(s). Canonical "what to read when editing X" map.
2. **Feature doc(s)** — for user-visible surface changes, open the matching doc in `docs/users/features/cp-ui/`, `docs/users/features/agent-ui/`, or `docs/users/features/flows/`. Index at `docs/README.md` §6.
3. **Conventions** — `docs/developers/workflow/conventions.md` for soft style guidance (naming, idioms, commit style, PR review checklist) complementing the hard bindings above.

The trigger table + feature catalog are kept outside CLAUDE.md on purpose — they're living catalogs that grow row by row. Every new `docs/developers/architecture/**/*-architecture.md` PR appends/updates the trigger file in the same PR; CI enforces lockstep via `npm run check:arch-doc-triggers`. Editing code in an area with no row signals either an undocumented architecture (raise it) or a new subsystem (add doc + row).

## Mandatory Development Workflow

```
Plan + Todo  →  Architecture  →  Requirements  →  SDD  →  OpenAPI  →  Code  →  Unit Tests  →  Verify  →  Ask-about-commit
```

No step skipped. No code without a plan + a live todo list.

- **Step 0 — Plan + Todo (every change, including one-liners).** Enter Plan Mode / produce plan output. Use `/brainstorm` (superpowers) on non-trivial problems. Create the todo list in Cursor `TodoWrite` or Claude Code `TaskCreate`. Get user confirmation on the plan (unless already approved in-thread). Never skip Plan or Todo silently — see Mandatory rules.
- **Step 1 — Architecture (`docs/developers/architecture/overview.md`).** Read first. Update if components / boundaries / data flow / deployment / external integrations change. If no impact, record a one-line "no architecture impact" note in the plan — do not skip silently.
- **Step 2 — Requirements (`docs/developers/specs/e{epic}-{name}.md`).** Capture from `docs/users/product/overview.md` / `docs/users/product/features.md` + chat. Each doc has: Functional Requirements, Non-Functional Requirements, User Roles & Personas, Constraints & Assumptions, Glossary, MoSCoW priority.
- **Step 3 — SDD (`docs/developers/specs/e{epic}-s{story}-{name}.md`).** Break Requirements into Epics → Stories → Tasks. Each Story has user story statement + tasks + acceptance criteria.
- **Step 4 — OpenAPI 3.1 (`docs/users/api/openapi/e{epic}-s{story}-{name}.yaml`).** For every Story with an API endpoint: paths + request/response schemas + error responses + examples. CP-UI API calls must match.
- **Step 5 — Code.** Go route handlers conform to OpenAPI. Prisma models align with spec schemas. CP-UI service layer matches spec. No placeholder production code (see Mandatory rules → Real implementation only). Update Todo as tasks complete.
- **Step 6 — Unit tests.** Go: `go test -race -count=1` (table-driven where appropriate). CP-UI: Vitest. Tests deterministic + aligned with SDD acceptance criteria. Do not mark complete until tests pass.
- **Step 7 — Verify (mandatory gate).** Run workspace `npm test` / package-level scripts; confirm green. Verify: plan was approved; all todos completed; architecture impact addressed or "no impact" recorded; requirements + SDD + OpenAPI exist; code matches OpenAPI; acceptance criteria satisfied; tests passing; verification commands run; commit reminder asked; no new TODO/stub in prod code; non-trivial work used deep reasoning + asked for needed user decisions; new content is English-only.

**Agent rules.** Cursor loads `.cursor/rules/sdd-workflow.mdc` (`alwaysApply: true`) — binding alongside this file. After every completed change, prompt the user about `git commit`; never commit without explicit instruction. Use `/brainstorm` for cross-cutting design decisions (auth, IAM, encryption, data model) where a wrong choice is expensive.

## Current state

Key facts:

- **Hub-centric model** — All 4 services (Control Plane, AI Gateway, Compliance Proxy, Agent) register with Nexus Hub as Things via `shared/transport/thingclient` (WebSocket primary, HTTP fallback).
- **Config flow** — Admin UI → CP admin API → Hub HTTP API → Hub updates shadow → WebSocket push → `thingclient.OnConfigChanged` callback. No Redis pub/sub for config invalidation.
- **Kill switch** — Managed via Hub shadow (Category A inline config), not local pub/sub.
- **Redis** — Pure cache only: sessions, IAM cache, rate limiting, response cache, desired state cache, quota counters. No pub/sub.
- **Control Plane UI** — Includes Infrastructure section: Nodes, Config Sync, Jobs, Kill Switch pages.
- **In-flight epics & queued work** — `docs/developers/roadmap.md` is the single living tracker of major in-flight features, queued epics, and reserved epic-number ranges. Read it first when asked "what's the status of E\<n\>?" or "what big features haven't been done yet?" — it links into requirements / SDD / OpenAPI / memory for each entry.

## Project structure

`docs/developers/architecture/overview.md` is the canonical system topology. Quick map:

- `packages/{nexus-hub,control-plane,ai-gateway,compliance-proxy,agent}` — five Go services

## Conventions

Code-level style + micro-decisions live in `docs/developers/workflow/conventions.md` (CI-enforced sections marked binding there). Read it as part of the Pre-edit reading 3-doc rule above. The CI-enforced bindings most often touched:

- **TypeScript / Control Plane UI** — `i18n mandatory` (`t()` for all user-visible strings; locale parity enforced); `design-token strict` (no hex / raw numeric in `*.module.css` or `style={{}}`; CSS variables only); `useApi` queryKey domain-prefixed (`['admin' | 'my' | 'user' | 'proxy', '<resource>', ...stateVars]`).
- **Go** — `go.work` workspace + full-module-path imports; `replace` directives sibling-only contract (binding — break it and `GOWORK=off` builds silently pull stale GitHub snapshots); `packages/shared/` dependency vetted set (binding — core tier + driver-scoped per-subpackage); no `sqlc`; `shared/` API additive-only once shipped in a released Agent binary.
- **General** — IoT terminology boundary (internal "Thing/Shadow/desired/reported/drift" vs user-facing "node/config sync/target config/applied config/out of sync"; CI: `npm run check:terminology`).

For full naming / errors / concurrency / logging / metrics / testing conventions, plus commit style + PR review checklist + tooling table, see `conventions.md` directly.
