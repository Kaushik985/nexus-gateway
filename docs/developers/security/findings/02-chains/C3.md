# C3 — Config push

> Phase 2 chain audit. Confirmed findings only (≥2/3 adversarial Opus verifiers). Base = post-arch-audit + SEC-M2-01 fix. Each finding attacks an end-to-end trust-boundary crossing.


## SEC-C3-01 — Inter-Hub `nexus.hub.signal` MQ path injects forced fleet-wide config — **CRITICAL** (2/3)
> **STATUS: FIXED** (2026-06-09, FIX-2) — `nexus.hub.signal` frames are now HMAC-signed with a Hub-to-Hub secret (`HUB_SIGNAL_HMAC_SECRET`, or a per-process random when unset). The subscriber (`SubscribeHubSignals`) drops any frame without a valid signature, so a data-plane producer / on-path actor that can reach NATS but lacks the Hub secret can no longer inject fleet-wide config (incl. a permissive killswitch) onto nodes. Robust signed-envelope (MAC over exact payload bytes — no State round-trip fragility). Single-Hub never verifies its own skipped signals, so a random secret suffices there; multi-Hub shares the env value. Regression: `TestSubscribeHubSignals_RejectsForgedSignal` + `_AcceptsValidlySignedSignal`.
> **Residual fixed (2026-06-10, `ab1dd6403`)** — the targeted-resync MQ fallback publish (`drift.go`) bypassed `SignHubSignal` (plain `json.Marshal`), so peer Hubs that now require HMAC provenance dropped legitimate resync frames in multi-Hub deployments; the resync publish now goes through `SignHubSignal`, with regressions covering the signed round-trip and the unsigned-drop path — closing the MORPHED residual the G6 verify round surfaced.

- **Persona:** A3 rogue-node (any service holding NATS_URL, e.g. a compromised compliance-proxy) / A4 on-path on the NATS link
- **Invariant:** Every config that reaches a node's live enforcement state (killswitch, exemptions) must pass the same provenance + F-0139 writable-key allowlist + schema validation that the authenticated admin and break-glass paths enforce — regardless of which Hub-internal transport carries it.
- **Arch cross-ref:** F-0139 (residual): the server-side {killswitch,exemptions} allowlist + schema gate is enforced only on the break-glass report-IN path (manager.ValidateBreakGlassReport, re-checked in handleBreakGlassReport). The inter-Hub config_changed signal-OUT path (ws/signal.go) reaches the SAME live enforcement keys on nodes with Force=true and has NO equivalent allowlist/schema/provenance check — a concrete residual on a different stage of the same chain.

**Preconditions.** Attacker can publish to NATS subject `nexus.hub.signal` (port 4222). In docker-compose.yml the NATS server runs with `--jetstream` only — NO `--auth`, no accounts, no nkey/token; `mq.NewProducer` calls `nats.Connect(cfg.URL)` with zero credentials (packages/shared/transport/mq/producer.go:34). Any co-located/reachable service (compliance-proxy holds a producer via wiring/mq.go) or on-path actor can publish. Multi-Hub deployment makes the subject load-bearing (single-Hub still subscribes; the subscriber only skips its OWN hubID).

**Attack steps.** 1) Attacker connects to NATS (no creds required) and publishes one JSON message to `nexus.hub.signal`: `{"action":"config_changed","sourceHub":"evil","thingType":"compliance-proxy","configKey":"killswitch","state":{"engaged":false},"version":9999999,"force":true}`. 2) `SubscribeHubSignals` (packages/nexus-hub/internal/ws/signal.go:34-72) unmarshals it, sees `sourceHub != hubID` so does NOT skip, matches `action==config_changed`, and — with NO allowlist check, NO `ValidateBreakGlassReport`, NO `validateBreakGlassState` schema decode — builds a `ConfigChangedMessage{Force:true, State:{engaged:false}, DesiredVer:9999999}` and `pool.Broadcast("compliance-proxy", data)` to every live node of that type (signal.go:61-65). 3) On each node, `handleHubMessage` case `config_changed` (packages/shared/transport/thingclient/client.go:982) sees `msg.Force==true`, so the stale-version guard `!msg.Force && msg.DesiredVer <= reportedVer` is SKIPPED (client.go:994); it accepts any `ConfigKey`, merges the delta, and calls `applyConfigForce(merged, ver)` (client.go:1021-1022). 4) The node's OnConfigChanged applies the injected `killswitch{engaged:false}` to live enforcement, DISENGAGING a kill-switch an admin had engaged (fail-open), fleet-wide. Same vector with `configKey:"exemptions"` injects an arbitrary compliance bypass; with a `thingId` set, it targets one specific node. The huge `version` also pins the node so a later legitimate lower-version push is ignored.

- **Affected:** /Users/zhangtiebin/workspaces/workspace-nexus/itech-nexus-gateway/packages/nexus-hub/internal/ws/signal.go:34-72 (no auth/allowlist/schema gate on inbound signal before broadcasting Force=true config to nodes); enabler /Users/zhangtiebin/workspaces/workspace-nexus/itech-nexus-gateway/packages/shared/transport/mq/producer.go:34 + docker-compose.yml:68-76 (NATS no-auth, no subject ACL); node-side Force bypass /Users/zhangtiebin/workspaces/workspace-nexus/itech-nexus-gateway/packages/shared/transport/thingclient/client.go:994-1022
- **Re-check probe.** grep -nE 'Validate|writable|allowlist|Killswitch|schema' packages/nexus-hub/internal/ws/signal.go  # MUST be non-empty after fix (currently 0 matches). AND: grep -nE 'nats.Connect\(cfg.URL\)' packages/shared/transport/mq/{producer,consumer}.go && grep -nE 'auth|accounts|nkey|--user|--pass|authorization' docker-compose.yml  # MUST gain NATS auth. Regression: assert a hub.signal carrying configKey not in {killswitch,exemptions} or state failing interception.Killswitch DisallowUnknownFields is dropped, not broadcast.
- **Remediation.** (1) In SubscribeHubSignals, before broadcasting, reject signals whose SourceHub is not in a configured known-Hub allowlist, and for Category-A inline keys run the SAME gate the break-glass path uses — reuse `manager.ValidateBreakGlassReport`/`validateBreakGlassState` (writable-key allowlist + DisallowUnknownFields schema decode) on (ConfigKey, State); drop on violation. (2) Do NOT propagate attacker-controllable `Force:true` from an unauthenticated MQ message into a node version-gate bypass; only authenticated admin re-sync should set Force, and the signal should carry an authenticity proof. (3) Enable NATS authentication + per-subject permissions so only Hubs may publish `nexus.hub.signal` (data-plane producers get pub rights only to `nexus.event.*`); add `nats.UserCredentials`/nkey to mq.NewProducer/Consumer and an accounts file to docker-compose. This is the same F-0139 invariant the break-glass report path already enforces — apply it symmetrically on the signal-OUT path.

## SEC-C3-02 — Single compromised node can flip the FLEET-WIDE kill-switch via a self-authored break-glass report — **HIGH** (3/3) 
> **STATUS: FIXED** (2026-06-09, FIX-2) — node-initiated break-glass can no longer write the fleet-wide `thing_config_template`: killswitch (and every key) now adopts as a PER-THING override (self-protection only); fleet-wide killswitch is operator-only via `POST /api/admin/compliance/killswitch`. Removed `breakGlassFleetScopedKeys` + the dead `adoptBreakGlassFleet` fleet-write path; updated kill-switch-architecture.md §131. Regression: `TestHandleBreakGlassReport_Killswitch_WritesPerThingOverride`. **Note: this narrows the documented "node break-glass is fleet-wide" semantics — see commit + doc.**


- **Persona:** A3 rogue/compromised node (compliance-proxy or agent) holding only its own device/service WS token; A5 low-priv anyone holding the shared service token reaches it for any thingId/type
- **Invariant:** Flipping a fleet-wide safety control (kill-switch engage/disengage, which turns TLS-bumping + the entire compliance pipeline ON/OFF for every node of a type) requires an operator credential (COMPLIANCE_PROXY_API_TOKEN); a node's own identity must NEVER be sufficient to author a desired-state change that takes effect on OTHER nodes.
- **Arch cross-ref:** F-0139

**Preconditions.** Attacker controls one compliance-proxy (or agent) process, or holds the shared internal service token. That node already has an authenticated Hub WebSocket (or can POST the HTTP fallback) — i.e., it is a normally-enrolled node whose process is compromised. No admin/operator credential and no COMPLIANCE_PROXY_API_TOKEN is needed.

**Attack steps.** STAGE 1 (data-plane→Hub frame): Over its already-authenticated WS connection the node sends a hand-crafted frame, bypassing the legitimate break-glass client entirely: {"type":"shadow_report_break_glass","reported":{"killswitch":{"engaged":true}},"reportedVer":999999,"actorTokenId":"deadbeef","keyVersions":{"killswitch":999999}}. actorTokenId is arbitrary free text (8 chars). reportedVer/keyVersions are attacker-chosen and set high to beat the stale-downgrade guard. STAGE 2 (Hub dispatch, server.go:280-301): handleMessage accepts shadow_report_break_glass from ANY authenticated thing, stamps Reason="break_glass" using the WS-authenticated thingID, and calls HandleShadowReport. STAGE 3 (authority gate, break_glass.go ValidateBreakGlassReport L79-100): killswitch is in breakGlassWritableKeys and {engaged:true} decodes cleanly into interception.Killswitch, so the gate PASSES. No check verifies actorTokenId against any operator credential (probe: grep for operator/actor verification returns 0). STAGE 4 (fleet adoption, break_glass.go adoptBreakGlassFleet L191-238): killswitch is in breakGlassFleetScopedKeys, so the state is UPSERTed into the FLEET-WIDE thing_config_template[thing.Type][killswitch] at reportedVer=999999 (version guard only blocks reportedVer<=cur.Version, never a forged-high version). STAGE 5 (convergence): every compliance-proxy node of that type pulls/drift-detects the new template and toggles to killswitch.engaged=true → drops into transparent passthrough: no TLS bump, no compliance hooks, no payload capture, fleet-wide (kill-switch-architecture.md §1,§5). The same primitive with engaged:false lets a node COUNTERMAND an operator-engaged kill-switch. Via the shared service token (A5), the HTTP route POST /api/internal/things/shadow/break-glass with any req.ID also reaches this — requireThingMatch (internal_things.go:46) is a no-op when ThingFromContext is nil, which is exactly the service-token case.

- **Affected:** packages/nexus-hub/internal/ws/server.go:280-301 (accepts break-glass frame from any node, no operator gate); packages/nexus-hub/internal/fleet/manager/break_glass.go:79-100 (ValidateBreakGlassReport gates key+schema only, never verifies actorTokenId / operator authority); packages/nexus-hub/internal/fleet/manager/break_glass.go:191-238 (adoptBreakGlassFleet writes fleet-wide template; version guard L198 only blocks stale, not forged-high); packages/nexus-hub/internal/fleet/handler/hubapi/internal_things.go:46 (requireThingMatch no-ops for service-token callers); packages/nexus-hub/internal/fleet/manager/shadow.go:30-32 (ActorTokenID documented as elevated-token prefix but only ever stamped into audit, never validated)
- **Re-check probe.** grep -rn 'VerifyOperator\|operatorToken\|ValidateActor\|verifyActorToken\|requireOperator' packages/nexus-hub/internal/fleet packages/nexus-hub/internal/ws --include=*.go | grep -v _test | wc -l  → expect 0 (no operator-authorization verification on the Hub break-glass adoption path). Confirm: ValidateBreakGlassReport in break_glass.go references ActorTokenID zero times (sed -n '79,100p' break_glass.go | grep -c ActorTokenID → 0). Confirm fleet scope: grep -n 'breakGlassFleetScopedKeys' break_glass.go shows configkey.Killswitch:true, so a single node's killswitch break-glass writes thing_config_template[type] (fleet-wide).
- **Remediation.** Treat fleet-scoped break-glass as a privileged operator action, not a node self-report. Options, in order of strength: (1) Require the Hub to independently verify operator authority on any FLEET-scoped break-glass key — e.g. the data-plane break-glass PUT must forward a Hub-verifiable operator proof (signed nonce / HMAC over the operator's COMPLIANCE_PROXY_API_TOKEN that the Hub validates), and ValidateBreakGlassReport rejects a fleet-scoped key whose proof does not verify. The current actorTokenId (unverified prefix) is insufficient. (2) Demote killswitch from fleet-scoped to per-thing for the node-initiated break-glass path: a node may only break-glass-engage the kill-switch ON ITSELF (per-thing override, like exemptions), never fleet-wide; genuine fleet-wide engage stays exclusively on the operator admin endpoint POST /api/admin/compliance/killswitch (admin:kill-switch.toggle IAM verb). This removes the cross-node blast radius entirely while preserving the single-node Hub-outage fallback. (3) At minimum, alarm + rate-limit: any break-glass killswitch adoption whose actorTokenId cannot be matched to a known operator token id should raise a SIEM alert and be rate-limited per source node, since the engaged-passthrough audit row is the only current signal.
